## Executive Summary

All 31 v1.3 requirements (5 DEBT + 26 DEPLOY) are structurally accounted for across Phases 23-30, with all 8 phase VERIFICATION.md files present and 6/6 integration flows largely wired. **However, the milestone CANNOT advance to GA tagging in its current state because the cross-phase integration check surfaced two real wiring bugs that per-phase verifications could not detect:**

1. **`verify-deploy.sh` step [5/7] will fail on every clean smoke run** — its success regex `'rotated|success|created'` does not match the create-admin CLI's actual output `'Updated password for ${email}'`. The smoke test wrapper aggregates this as MAX_RC=1, producing a false-negative GA verdict on every fresh VM run.
2. **First-deploy seed ordering leaves zero default StreamProfile** — `bootstrap.sh` runs `sms-migrate` (which contains the seed) before `create-admin`, so the seed runs against an empty org list and no-ops. The operator will hit "no default profile" on the first camera registration. Both per-phase verifiers (Phase 26 + 29) caught the static contracts but not the runtime ordering.

These are NOT static-vs-dynamic deferrals (those are properly tracked under Phase 30 HUMAN-UAT). They are bugs in shipped code that need fixing before any operator can run a clean smoke. **Recommend a small gap-closure phase (or v1.3.1 patch) before `/gsd-complete-milestone`.**

Additionally, **7 of 8 phases are missing formal Nyquist `VALIDATION.md`** (only Phase 23 has one). This is a process gap, not a code gap, but a clean GA milestone should record validation strategy for downstream regression-test work.

## Critical Wiring Gaps

### Gap #1 — DEPLOY-25 / smoke-test regex blocks every GA run

**Severity:** blocker
**Location:** `deploy/scripts/verify-deploy.sh:241` + `apps/api/src/cli/sms.ts:171`

verify-deploy.sh step [5/7] (`create-admin --force` re-run) greps for `'rotated|success|created'` in the CLI output. But the CLI prints exactly:

```
[create-admin] Updated password for ${email}.
```

when `existing && force` (the path that fires after step [1/7] already created the admin).

`'Updated password'` matches NONE of `rotated`, `success`, `created`. The earlier "Created super-admin" message is in the `else` branch (only fires if user does NOT exist). Step [5/7] therefore records FAIL → smoke-test.sh aggregates `MAX_RC=1` → exit code 1 → **HARD GA BLOCK on every clean smoke run**.

**Fix (one-line, either side):**
- `verify-deploy.sh:241` — broaden regex to `'rotated|success|created|Updated password'`
- OR `apps/api/src/cli/sms.ts:171` — change emitted string to include `'rotated'`

**Affected requirements:** DEPLOY-17 (create-admin), DEPLOY-25 (smoke test acceptance gate).

### Gap #2 — DEPLOY-16 seed ordering leaves no default StreamProfile

**Severity:** soft blocker (operator hits "no default profile" error on first camera registration)
**Location:** `deploy/scripts/bootstrap.sh:107` (sms-migrate) before `bootstrap.sh:141` (create-admin)

The seed runs in `sms-migrate` (`docker-compose.yml:149-159` chains `prisma migrate deploy && init-buckets.js && seed-stream-profile.js`). On a fresh VM the seed runs first → encounters zero orgs → no-ops correctly per `seed-stream-profile.ts:32-37`. Then `create-admin` creates the system org. But bootstrap.sh does NOT re-trigger `sms-migrate` to re-run the seed against the now-existing org.

**Result:** A clean deploy ends with zero StreamProfile rows. The operator sees a misleading "no default profile" error when registering their first camera, defeating the "first camera in <10 minutes" claim.

**Fix:**
- `bootstrap.sh` — add `${DC} run --rm sms-migrate` after the `bin/sms create-admin` step
- OR fold a one-time `seed-stream-profile.js` invocation into `create-admin` itself (cleaner — admin creation already implies system org exists)
