```bash
#!/usr/bin/env bash
# .github/scripts/smoke-api.sh — Phase 28 (DEPLOY-03 pre-push gate).
# Asserts Phase 25 D-19 invariants #3 (non-root UID), #4 (ffmpeg present),
# #5 (tini installed) on a built api image BEFORE it ships to GHCR.
# Called by .github/workflows/build-images.yml after `docker buildx build --load`.
#
# Usage: bash .github/scripts/smoke-api.sh <image-ref>
# Exit 0 = all assertions pass (image safe to push).
# Exit non-zero = at least one assertion failed (build job fails, no push).

set -euo pipefail

if [ "$#" -ne 1 ]; then
  echo "Usage: $0 <image-ref>" >&2
  exit 64
fi

IMAGE="$1"

echo "[smoke-api] 1/3 non-root UID check (Phase 25 D-19 #3)"
ACTUAL_UID="$(docker run --rm "$IMAGE" id -u)"
if [ "$ACTUAL_UID" != "1001" ]; then
  echo "[smoke-api] FAIL: expected uid 1001, got '$ACTUAL_UID'" >&2
  exit 1
fi

echo "[smoke-api] 2/3 ffmpeg present (Phase 25 D-19 #4)"
if ! docker run --rm --entrypoint /usr/bin/ffmpeg "$IMAGE" -version | grep -qE 'ffmpeg version (5|6|7)\.'; then
  echo "[smoke-api] FAIL: ffmpeg version 5/6/7 not detected" >&2
  exit 1
fi

echo "[smoke-api] 3/3 tini installed (Phase 25 D-19 #5)"
if ! docker run --rm --entrypoint /usr/bin/tini "$IMAGE" --version | grep -qE '^tini version '; then
  echo "[smoke-api] FAIL: /usr/bin/tini --version did not report a version line" >&2
  exit 1
fi

echo "[smoke-api] PASS: all assertions met for $IMAGE"
```

After writing, run `chmod +x .github/scripts/smoke-api.sh`.

Decision references: per D-11 (CONTEXT.md L165-176), per D-13 (script lives under `.github/scripts/`, not `scripts/`).

Why ffmpeg version regex `5|6|7`: Bookworm-slim ships ffmpeg 5.1.x today, but a base-image bump to a future Bookworm point release or trixie could promote to 6.x or 7.x. Pinning to "5 only" creates a CI break when the runtime base auto-updates. The Phase 25 image budget (≤450MB) is the actual constraint on FFmpeg version — the smoke script just confirms a recognized version is installed.

Why `--entrypoint /usr/bin/ffmpeg` instead of relying on PATH: the Phase 25 Dockerfile sets `ENTRYPOINT ["/usr/bin/tini", "--", "node", "dist/main"]`, so plain `docker run --rm $IMAGE ffmpeg -version` would try to run ffmpeg as a node arg. Overriding the entrypoint isolates the assertion.

Why `id -u` works without entrypoint override: `id` is run as the default `app` user (USER directive), and `id -u` is a tiny coreutils binary — fast and side-effect free. The Phase 25 entrypoint runs `node dist/main` which would block forever, so we override via the default `id` shell-replacement (Docker treats `id -u` as the command, NOT as a node arg, because `--rm` + bare command bypasses the configured entrypoint when the user provides a CMD; on this image, ENTRYPOINT is `["/usr/bin/tini", "--", "node", "dist/main"]` which means `id -u` becomes a node arg). For robustness, also use `--entrypoint /usr/bin/id "$IMAGE" -u` if the simple form has issues — confirm with a local test before committing.

```bash
#!/usr/bin/env bash
# .github/scripts/smoke-web.sh — Phase 28 (DEPLOY-03 pre-push gate).
# Asserts Phase 25 D-19 invariants #1 (non-root UID), #7 (boot), #8
# (/api/health serves) on a built web image BEFORE it ships to GHCR.
# Boots a real container (web has no external deps) so a regression in
# Phase 25 D-18 outputFileTracingRoot would manifest as boot failure here.
# Called by .github/workflows/build-images.yml after `docker buildx build --load`.
#
# Usage: bash .github/scripts/smoke-web.sh <image-ref>
# Exit 0 = boot + health probe pass (image safe to push).
# Exit non-zero = at least one assertion failed.

set -euo pipefail

if [ "$#" -ne 1 ]; then
  echo "Usage: $0 <image-ref>" >&2
  exit 64
fi

IMAGE="$1"

echo "[smoke-web] 1/2 non-root UID check (Phase 25 D-19 #1)"
ACTUAL_UID="$(docker run --rm --entrypoint /usr/bin/id "$IMAGE" -u)"
if [ "$ACTUAL_UID" != "1001" ]; then
  echo "[smoke-web] FAIL: expected uid 1001, got '$ACTUAL_UID'" >&2
  exit 1
fi

echo "[smoke-web] 2/2 boot + /api/health probe (Phase 25 D-19 #7, #8)"
CID="$(docker run -d -p 3000:3000 "$IMAGE")"
cleanup() { docker rm -f "$CID" >/dev/null 2>&1 || true; }
trap cleanup EXIT

for i in $(seq 1 30); do
  if curl -fsS http://localhost:3000/api/health 2>/dev/null | grep -q '"ok":true'; then
    echo "[smoke-web] PASS: /api/health returned ok=true after ${i}s"
    exit 0
  fi
  sleep 1
done

echo "[smoke-web] FAIL: /api/health did not return ok=true within 30s" >&2
echo "[smoke-web] container logs (last 50 lines):" >&2
docker logs --tail 50 "$CID" >&2 || true
exit 1
```

After writing, run `chmod +x .github/scripts/smoke-web.sh`.

Decision references: per D-12 (CONTEXT.md L179-197), per D-13 (script location).

Why `-p 3000:3000`: GitHub Actions ubuntu-latest runner allows binding to high ports without conflict; port 3000 is conventional and matches Dockerfile's `EXPOSE 3000`/`ENV PORT=3000`. Inside CI no other workload contends.

Why 30s timeout: Phase 25's web image is `node apps/web/server.js` cold start. On ubuntu-latest runner with shared CPU, expect ~3-8s boot + ~2s static asset trace. 30s gives 4× safety margin without dragging job time.

Why `grep -q '"ok":true'`: Phase 25 Plan 02 health route returns `{ok: true, ...}`. Looser than full JSON parse (no jq dependency in CI runner image — though jq IS available, prefer grep for portability + clarity).

Why `trap cleanup EXIT`: ensures the test container is removed even on assertion failure or signal — keeps the CI runner clean for the matrix's other job (api).
