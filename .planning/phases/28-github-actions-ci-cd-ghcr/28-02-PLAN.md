```yaml
# .github/workflows/release.yml — Phase 28 (DEPLOY-03 SC #4, DEPLOY-04).
# Creates a GitHub Release on every v*.*.* tag push. Runs in parallel with
# build-images.yml — release.yml does NOT push images (no `packages:write`),
# does NOT depend on build-images.yml job success (operators see the Release
# entry immediately; image refs resolve once build-images.yml finishes the
# push step). Auto-flags pre-release tags (alpha/beta/rc/test) per D-17.
#
# Phase 28 D-15, D-16, D-17. Image build/publish workflow lives in
# .github/workflows/build-images.yml (separate file).

name: Release

on:
  push:
    tags:
      - 'v*.*.*'

permissions:
  contents: write   # required by softprops/action-gh-release@v2 to create the release

jobs:
  release:
    runs-on: ubuntu-latest
    timeout-minutes: 5
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Detect prerelease (alpha/beta/rc/test suffixes per D-17)
        id: prerelease_check
        run: |
          if [[ "${{ github.ref_name }}" =~ -(alpha|beta|rc|test) ]]; then
            echo "is_prerelease=true" >> "$GITHUB_OUTPUT"
          else
            echo "is_prerelease=false" >> "$GITHUB_OUTPUT"
          fi

      - name: Create GitHub Release
        uses: softprops/action-gh-release@v2
        with:
          generate_release_notes: true
          prerelease: ${{ steps.prerelease_check.outputs.is_prerelease }}
          body: |
            ## Container Images

            - `ghcr.io/${{ github.repository_owner }}/sms-api:${{ github.ref_name }}`
            - `ghcr.io/${{ github.repository_owner }}/sms-web:${{ github.ref_name }}`

            Provenance verified via build attestation:

            ```sh
            gh attestation verify oci://ghcr.io/${{ github.repository_owner }}/sms-api:${{ github.ref_name }} --owner ${{ github.repository_owner }}
            gh attestation verify oci://ghcr.io/${{ github.repository_owner }}/sms-web:${{ github.ref_name }} --owner ${{ github.repository_owner }}
            ```

            ## Upgrade an existing deployment

            ```sh
            cd /opt/sms-platform/deploy   # or wherever you cloned `deploy/`
            export IMAGE_TAG=${{ github.ref_name }}
            docker compose pull
            docker compose up -d
            ```

            See [deploy/README.md](https://github.com/${{ github.repository }}/blob/main/deploy/README.md) for the full operator guide (lands in Phase 29).
```

Decision references:
- D-15 (CONTEXT.md L213-218): `on: push: tags: ['v*.*.*']` — tag-only trigger
- D-16 (CONTEXT.md L221-263): `permissions: contents: write`, prerelease detection step, softprops/action-gh-release@v2 with the body template above
- D-17 (CONTEXT.md L266): regex `(alpha|beta|rc|test)` covers Phase 28 SC #1 test tag pattern `v1.3.0-test`

Why `permissions: contents: write` ONLY (no `packages: write`): release.yml does not push container images. Per D-21 + threat model T-28-07 (excessive scope), keep this workflow's permissions minimal. Image push lives in Plan 03 build-images.yml under a separate, larger permissions block.

Why `timeout-minutes: 5`: release.yml does one checkout + one API call to `softprops/action-gh-release@v2`. Default GitHub timeout is 360 minutes — explicit 5 makes a stuck run fail loudly instead of silently consuming runner minutes.

Why `generate_release_notes: true` AND custom `body:`: softprops/action-gh-release@v2 prepends auto-generated commit-history notes BEFORE the custom body when both are set (per its README). Operators see the changelog first, then the deploy snippet.

Why duplicate the gh attestation verify lines (api + web): a single command would force operators to mentally substitute. Two literal lines = copy-paste safe.

Why `cd /opt/sms-platform/deploy`: Phase 29 deploy/README.md will document this as the canonical install path. Mentioning it here primes operators before Phase 29 doc lands. Comment in the snippet (`# or wherever you cloned`) covers operators with custom paths.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Git tag push → workflow trigger | Tag name `${{ github.ref_name }}` is operator-controlled; flows into Release body and output expressions |
| Workflow → GitHub Releases API | Authenticated via auto-provisioned `GITHUB_TOKEN` (read-write to release; cannot push packages) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-28-05 | Tampering | Workflow injection via tag name in shell `run:` step | mitigate | The `if [[ "${{ github.ref_name }}" =~ ... ]]` line uses `${{ }}` interpolation INSIDE a bash double-quoted string. GitHub recommends using `env:` to pass values to shell, but tags are constrained by Git's ref name validation (no shell metacharacters except `-`, `.`, `/`, `_`). The regex compares against fixed alternatives — even a maliciously-crafted tag like `v1.0.0$(rm -rf /)` would (a) be rejected by Git ref validation, and (b) the regex match would fail because `$(...)` doesn't match `-(alpha|beta|rc|test)`. ACCEPTED for this minor sliver — full mitigation path is to switch to `env: REF_NAME: ${{ github.ref_name }}` + `if [[ "$REF_NAME" =~ ... ]]`, which we adopt as the recommended hardening if v1.4 audit flags it |
| T-28-06 | Elevation of Privilege | Excessive workflow scope grants release.yml access to packages | mitigate | `permissions: contents: write` ONLY — no `packages:` scope. release.yml literally cannot push to GHCR even if compromised |
| T-28-07 | Spoofing | Non-owner pushes a tag to create a fake release | mitigate | GitHub default branch protection: only users with push access can create tags; for protected tags (admin setup), tag creation is gated. v1.3 ships with default settings (anyone with write access can tag) — accepted operationally since this is a single-developer/small-org repo per project context |
| T-28-08 | Information Disclosure | Release body leaks internal infrastructure details | accept | Body contains: image refs (already public on GHCR per D-19), generic compose path (`/opt/sms-platform/deploy`), gh CLI command. No secrets, no internal hostnames. Information is operator-facing by design |
</threat_model>
