**Patch 1 — Add caddy service block.** Insert AFTER the closing of the `web:` service block (after line ~250 — the line containing `logging: *default-logging` at the end of the web service) and BEFORE the `# ===` separator that introduces the `networks:` section.
```yaml
  # ==========================================================================
  # caddy — reverse proxy + auto-TLS via Let's Encrypt (Phase 27, DEPLOY-06)
  # Joins BOTH networks (D-17 amends Phase 26 D-06): edge for :80/:443 host
  # exposure + internal so it can resolve minio:9000 for /avatars/* and

  caddy:
    image: caddy:2.11
    restart: unless-stopped
    init: true
    ports:
      - "80:80"
      - "443:443"
    environment:
      DOMAIN: ${DOMAIN}
      ACME_EMAIL: ${ACME_EMAIL}
      ACME_CA: ${ACME_CA:-}
    volumes:
      - caddy_data:/data
      - caddy_config:/config
      - ./Caddyfile:/etc/caddy/Caddyfile:ro
    healthcheck:
      test: ["CMD", "wget", "--spider", "-q", "http://localhost:80"]
      interval: 30s
      timeout: 5s
      retries: 3
      start_period: 30s
    depends_on:
      api:
        condition: service_healthy
      web:
        condition: service_healthy
    networks:
      - edge
      - internal
    logging: *default-logging
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Host kernel → Caddy container | Caddy joins host network namespace ONLY via the published :80/:443 port mappings; container-side egress flows on the `edge` Docker bridge. |
| Caddy → MinIO/api/web | Internal Docker bridge network DNS; only services attached to the same bridge can resolve each other. Caddy on `internal` (D-17) is the only way to reach `minio:9000`. |
| Operator host → caddy_data + caddy_config volumes | Docker named volumes; default mode 0755 root-owned. Phase 30 host-hardening (out of Phase 27 scope) covers volume permissions. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-27-CERT-PRIV | Information Disclosure | caddy_data named volume (cert + ACME private key) | mitigate | Volume is named (not bind-mounted to host path) → no accidental world-readable cert files. Phase 29 `backup.sh` (DEPLOY-20) will include it under controlled access. Operator host-level hardening is Phase 30 territory. |
| T-27-CADDYFILE-RW | Tampering | Caddyfile inside running container | mitigate | Volume mount line `./Caddyfile:/etc/caddy/Caddyfile:ro` — `:ro` flag prevents container runtime from writing back. Verified via `grep -c '/etc/caddy/Caddyfile:ro' deploy/docker-compose.yml == 1`. |
| T-27-ADMIN-API | Elevation of Privilege | Caddy admin endpoint :2019 | mitigate | NO port mapping for 2019; combined with `admin off` in plan 27-01's Caddyfile, the endpoint is disabled at config layer AND not host-reachable at network layer. |
| T-27-HTTP3-SURFACE | Denial of Service / firewall surface | UDP :443 (QUIC) | accept | Per D-12, HTTP/3 explicitly disabled; ports block lists ONLY `"80:80"` + `"443:443"` (no `/udp`). Reversible — 1 line change to enable QUIC + protocols update. |
| T-27-BOOT-502 | Availability | Caddy upstream proxying during cold boot | mitigate | `depends_on api/web service_healthy` ensures upstreams pass image-side healthchecks before Caddy starts proxying. `start_period: 30s` on Caddy itself protects against unhealthy-marker during ACME issuance. |
</threat_model>
