**1. `apps/web/src/app/admin/developer/docs/api-workflow/page.tsx` -- API Workflow Guide:**
Sections:
- **Overview**: What the API does (manage cameras, create playback sessions, embed streams)
- **Authentication**: Two methods: session cookie (admin dashboard) and API key (X-API-Key header for programmatic access). Show header example.
- **Step 1: Create an API Key**: POST /api/api-keys with body { name, scope, scopeId }. CodeBlock with curl example. Explain scoping (PROJECT = all cameras in project, SITE = cameras in site only).
- **Step 2: Create a Playback Session**: POST /api/cameras/{cameraId}/sessions with X-API-Key header. CodeBlock with curl. Explain response: { sessionId, hlsUrl, expiresAt }.
- **Step 3: Batch Sessions**: POST /api/playback/sessions/batch with { cameraIds: [...] }. Max 50 per request.
- **Step 4: Embed the Stream**: iframe snippet and hls.js snippet with CodeBlock.
- **Error Handling**: Common error responses (401 Unauthorized, 403 Feature disabled, 404 Camera not found, 429 Rate limited).

**2. `apps/web/src/app/admin/developer/docs/policies/page.tsx` -- Policies Guide:**
Sections:
- **Overview**: Policies control playback session behavior (TTL, viewer limits, domains).
- **Policy Levels**: System > Project > Site > Camera. Each level inherits from parent.
- **Resolution Order**: Per-field merge -- Camera overrides Site, Site overrides Project, Project overrides System. Only non-null fields override.
- **Configurable Fields**: TTL (seconds, controls session duration), Max Viewers (0 = unlimited), Domain Allowlist (array, empty = allow all, supports *.example.com), Allow No Referer (boolean), Rate Limit (per-API-key requests/min).
- **Examples**: CodeBlock showing policy creation via API and resolution behavior.

**3. `apps/web/src/app/admin/developer/docs/stream-profiles/page.tsx` -- Stream Profiles Guide:**
Sections:
- **Overview**: Stream profiles control how camera feeds are processed by FFmpeg.
- **Passthrough Mode**: `-c copy` -- no transcoding, lowest CPU. Best for H.264 cameras.
- **Transcode Mode**: Converts to H.264 with configurable resolution, FPS, bitrate. Required for H.265 cameras (browsers can't play HEVC natively).
- **Auto-Detection**: Platform auto-detects H.265 via ffprobe and applies transcoding profile.
- **Profile Settings**: Codec (auto/h264), Preset (ultrafast to veryslow tradeoff), Resolution, FPS, Video Bitrate, Audio Codec (AAC), Audio Bitrate.
- **Recommendations**: Table of recommended settings by use case (low bandwidth, high quality, mobile-optimized).

**4. `apps/web/src/app/admin/developer/docs/webhooks/page.tsx` -- Webhooks Guide:**
Sections:
- **Overview**: Receive HTTP POST notifications when camera events occur.
- **Event Types**: camera.online, camera.offline, camera.degraded, camera.reconnecting. Describe when each fires.
- **Creating a Subscription**: POST /api/webhooks with { name, url, events }. URL must be HTTPS.
- **Payload Format**: Show example JSON payload with cameraId, status, previousStatus, timestamp.
- **HMAC Verification**: Header format `X-Webhook-Signature: t=timestamp,v1=signature`. CodeBlock with Node.js verification example:
```javascript
const crypto = require('crypto');
const signature = crypto.createHmac('sha256', webhookSecret)
  .update(`${timestamp}.${rawBody}`)
  .digest('hex');
// Compare signature with v1 value using timing-safe comparison
```
- **Retry Behavior**: 5 attempts with delays ~1m, 5m, 30m, 2h, 12h. Failed after all attempts.
- **Best Practices**: Respond with 200 quickly, process async, verify HMAC, reject payloads > 5 min old.

**5. `apps/web/src/app/admin/developer/docs/streaming-basics/page.tsx` -- Streaming Basics Guide:**
Sections:
- **Overview**: How the SMS Platform processes camera streams.
- **RTSP**: What it is (Real Time Streaming Protocol), how cameras typically output RTSP, why browsers can't play RTSP directly.
- **HLS**: What it is (HTTP Live Streaming), how it works (m3u8 playlist + segments), why it's the delivery format (universal browser support).
- **The Pipeline**: Camera -> RTSP -> FFmpeg -> RTMP -> SRS -> HLS -> Browser. Explain each step briefly.
- **Codecs**: H.264 (universal browser support) vs H.265 (better compression, needs transcoding for browsers). AAC for audio.
- **Latency**: HLS has ~5-10 second latency (segment duration + buffering). WebRTC alternative for sub-second latency.
- **Key Terms Glossary**: Short definitions of RTSP, RTMP, HLS, SRS, FFmpeg, Transcode, Passthrough, fMP4, AES-128.
