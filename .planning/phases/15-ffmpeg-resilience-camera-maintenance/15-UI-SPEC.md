# Phase 15 — UI Design Contract

> Visual and interaction contract for the Camera table Status column upgrade (CAM-02) and the maintenance-mode row action (CAM-03). Backend work in this phase (RESIL-01..04) has no UI surface and is out of scope for this document. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope

This phase has a **narrow UI surface** — 90% of the work is backend. Only two user-facing changes:

1. **Composite Status column** on `apps/web/src/app/admin/cameras/components/cameras-columns.tsx` — one cell renders three horizontally-stacked icons: connectivity dot + recording dot + maintenance wrench. Each icon has a tooltip.
2. **Maintenance quick action** in the existing row dropdown — a single toggle entry ("เข้าโหมดซ่อมบำรุง" / "ออกจากโหมดซ่อมบำรุง") that opens a confirmation dialog, then calls the maintenance API.

Everything else (health check loop, SRS restart recovery, graceful shutdown, boot re-enqueue, flapping debounce, webhook suppression) is server-side and invisible to the operator. No new pages, no new components in `components/ui/`, no new tokens — this phase consumes the Phase 14 token system verbatim.

Source: CONTEXT.md D-16, D-17 + Claude's Discretion section.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `hsl(0 0% 100%)` light / `hsl(0 0% 9%)` dark | Table background, dialog surface |
| Secondary (30%) | `hsl(142 30% 97%)` light / `hsl(142 20% 15%)` dark | Table header, tooltip background |
| Accent (10%) | `hsl(142 71% 45%)` (green primary) | Online status dot (`bg-primary`), dialog confirm button when **exiting** maintenance |
| Destructive | `hsl(0 84% 60%)` | Recording indicator fill (`fill-red-500 text-red-500`), destructive confirm button when **entering** maintenance (stops stream) |
| Warning (amber) | `text-amber-500` / `text-amber-600` dark | Maintenance wrench icon when `maintenanceMode = true`, degraded/reconnecting status dot |
| Muted foreground | `text-muted-foreground` | Inactive recording dot, hidden-state placeholder width reservation |

Accent reserved for: online status dot (`bg-primary`), dialog "Exit maintenance" confirm button (non-destructive variant). **Never** on the recording dot, never on the maintenance wrench — those have their own semantic colors.

### Status Icon Color Matrix

| Icon | State | Class | Semantic |
|------|-------|-------|----------|
| Connectivity dot | online | `bg-primary` (green) | Accent |
| Connectivity dot | offline | `border border-muted-foreground bg-transparent` | Muted |
| Connectivity dot | degraded | `bg-amber-500` | Warning |
| Connectivity dot | connecting | `bg-primary animate-pulse` | Accent (animated) |
| Connectivity dot | reconnecting | `bg-amber-500 animate-pulse` | Warning (animated) |
| Recording dot | recording | `fill-red-500 text-red-500` | Destructive |
| Recording dot | not recording | `text-muted-foreground` (unfilled circle) | Muted |
| Maintenance wrench | in maintenance | `text-amber-600 dark:text-amber-500` | Warning |
| Maintenance wrench | not in maintenance | hidden (empty 14px slot reserved) | — |

**Why amber for both maintenance and degraded:** In this app's vocabulary amber always means "needs attention, not broken." Degraded = stream is flaky. Maintenance = operator parked it on purpose. They never appear simultaneously (maintenance filter excludes degraded cameras from health checks per D-02), so color collision is impossible.

**Why the maintenance icon slot stays reserved when hidden:** horizontal alignment across rows. If row A has maintenance and row B doesn't, their recording dots must still line up. Use `aria-hidden="true"` + `invisible` (preserves layout) instead of conditional mount.

Source: CONTEXT.md D-16, existing `CameraStatusDot` (`camera-status-badge.tsx:12-42`), existing recording icon pattern (`view-stream-sheet.tsx:75`).

**Icon choice rationale (`Wrench` vs `Tool` vs `Pause`):**
- `Pause` reads as "paused playback" — wrong semantic for "operator parked this for repair."
- `Tool` (lucide) is a wrench icon under an alias but less semantically clear.
- `Wrench` is the canonical maintenance/repair icon across SaaS admin UIs (Stripe, Vercel, GitHub Actions all use it for maintenance/admin modes).
- `Wrench` is not already used elsewhere in this codebase (grep confirmed: zero existing imports), so no vocabulary collision.

**New additions:** zero. Everything is composed from existing primitives.

### Composite Status Cell Structure

Replaces the single `<CameraStatusDot />` call at `cameras-columns.tsx:45` with a three-slot flex row:

```tsx
<div className="flex items-center gap-1" aria-label="Camera status">
  <Tooltip>
    <TooltipTrigger asChild>
      <CameraStatusDot status={camera.status} />
    </TooltipTrigger>
    <TooltipContent>{connectivityTooltip[camera.status]}</TooltipContent>
  </Tooltip>

  <Tooltip>
    <TooltipTrigger asChild>
      <Circle
        className={cn(
          "size-3",
          camera.isRecording
            ? "fill-red-500 text-red-500"
            : "text-muted-foreground"
        )}
        aria-hidden="true"
      />
    </TooltipTrigger>
    <TooltipContent>
      {camera.isRecording ? "กำลังบันทึก" : "ไม่ได้บันทึก"}
    </TooltipContent>
  </Tooltip>

  <Tooltip>
    <TooltipTrigger asChild>
      <Wrench
        className={cn(
          "size-3.5",
          camera.maintenanceMode
            ? "text-amber-600 dark:text-amber-500"
            : "invisible"
        )}
        aria-hidden={!camera.maintenanceMode}
      />
    </TooltipTrigger>
    {camera.maintenanceMode && (
      <TooltipContent>อยู่ในโหมดซ่อมบำรุง — ไม่แจ้งเตือน</TooltipContent>
    )}
  </Tooltip>
</div>
```

**Sizing contract:**

| Icon | Class | Rendered size |
|------|-------|---------------|
| `CameraStatusDot` | existing `h-2 w-2` | 8×8px |
| Recording `Circle` | `size-3` | 12×12px (stroke circle reads as 8px dot) |
| Maintenance `Wrench` | `size-3.5` | 14×14px (wrench is a denser glyph, 14px balances visual weight with the 8px dots) |
| Row height impact | none | All three fit within the existing `text-sm` row height |

**Gap:** `gap-1` (4px) — tight enough to read as a single cluster, loose enough that each icon has its own hit target for the tooltip trigger.

**Column width:** `size: 72` on the TanStack column definition (up from existing default). Accommodates the cluster + TanStack resize handle comfortably without crowding the Name column.

**Accessibility:**
- Parent `<div>` has `aria-label="Camera status"` — one meaningful label for screen readers, avoids three separate per-icon labels that read as "dot dot wrench."
- Individual icons have `aria-hidden="true"` — they're decorative; the tooltips carry the semantic.
- Tooltips are focusable via keyboard because `TooltipTrigger` accepts focus by default (Base UI primitive).
- Hidden maintenance icon uses `invisible` (layout preserved) + `aria-hidden="true"` — not `hidden` / unmount, which would shift other rows.

### Dropdown Action Insertion Point

In `cameras-columns.tsx` the existing `rowActions` array (line 121) is extended with a single entry, inserted between the recording toggle (line 129-133) and the Embed Code entry (line 134):

```tsx
{
  label: camera.maintenanceMode
    ? "ออกจากโหมดซ่อมบำรุง"
    : "เข้าโหมดซ่อมบำรุง",

```

`callbacks.onMaintenanceToggle` opens the appropriate AlertDialog (enter vs exit) based on the current `camera.maintenanceMode` value. Dialog dispatch happens at the page/container level (`cameras-page.tsx` or sibling), not inside the columns factory.

## Interaction Patterns

### Live Status Updates (existing behavior, unchanged)

- `useCameraStatus` WebSocket hook (`apps/web/src/hooks/use-camera-status.ts`) already dispatches per-row `camera:status` events. No changes to the hook.
- CONTEXT.md D-04 clarifies: "In-app status badge updates immediately (no debounce on UI state), only outbound notify/webhook is delayed." This contract is honored — the composite cell re-renders the instant a WebSocket event arrives.
- `maintenanceMode` is not pushed over WebSocket (it's a manual toggle, not a runtime-changing state). After the API call succeeds, the client optimistically updates the row via React Query's `setQueryData` or the page's local state — matches Phase 14 pattern.

### Enter Maintenance Flow

1. Operator clicks "เข้าโหมดซ่อมบำรุง" in row dropdown.
2. Dropdown closes, `AlertDialog` opens with title/body/destructive confirm.
3. On "เข้าโหมดซ่อมบำรุง" click:
   - Confirm button shows loading spinner (disabled state).
   - POST/PATCH to maintenance API.
   - On success: dialog closes, toast fires, row optimistically updates (`status = offline`, `maintenanceMode = true`, `isRecording = false` per D-13 side effects).
   - On error: dialog stays open, error toast fires, confirm button re-enabled.
4. Operator can re-attempt or cancel.

### Exit Maintenance Flow

1. Operator clicks "ออกจากโหมดซ่อมบำรุง" in row dropdown.
2. `AlertDialog` opens with title/body/default (non-destructive) confirm.
3. On "ออกจากโหมด" click:
   - Same loading / success / error pattern as enter.
   - On success: `maintenanceMode = false`, `status` stays `offline` (per D-14, no auto-restart). Operator must click Start Stream separately.
4. The success toast copy explicitly tells the operator to click Start Stream — prevents the "thought the camera was back" foot-gun.

### Destructive Action Summary

| Action | Confirmation Type | Button Variant | Reason |
|--------|-------------------|----------------|--------|
| Enter maintenance | AlertDialog | destructive | Stops a live stream + halts recording |
| Exit maintenance | AlertDialog | default | No destructive side effect; confirmation exists only to prevent accidental clicks in the dropdown |

Both flows use the same `AlertDialog` component — only the button variant and copy change.

## Debounce Implementation Note (non-UI)

CONTEXT.md D-04 left implementation of the 30s notification debounce to the researcher. The recommendation for the planner is:

**Use BullMQ delayed jobs, not a `setTimeout` map.** Rationale:
- `setTimeout` on a Node process dies on restart — the graceful-shutdown work (D-08) would kill any in-flight debounce, and on reboot the debounce window is lost. The 30s window **must** survive process restarts or the whole flapping defense is pointless.
- BullMQ has native delayed-job semantics. `StatusService.transition` enqueues a `notify-camera-event` job with `delay: 30_000` and a deterministic `jobId = camera:{cameraId}:notify`. A new transition inside 30s calls `queue.remove(jobId)` before enqueueing the fresh job — BullMQ treats this as debounce-by-replacement.
- `BullMQ jobId` dedup already underlies CONTEXT.md D-11; same pattern, same mental model.

This note lives in UI-SPEC.md only because CONTEXT.md asked the researcher to decide. It is informational for the planner, not a UI contract.

## Precedent This Phase Sets

No existing maintenance pattern exists in the app. This phase establishes:

1. **Composite status cells** — precedent for Phase 17/18 if they need multi-state indicators (e.g. recording timeline availability icons).
2. **Amber-as-maintenance** — reserves amber for "operator-parked" states across the app. Phase 17/18 should not reuse amber for anything else.
3. **Wrench-as-maintenance** — icon vocabulary. Future bulk-maintenance UI (CAM-04, deferred) should reuse Wrench.
4. **Thai-first destructive dialogs with bold side-effect callouts** — pattern for any future destructive action that has non-obvious downstream effects.
