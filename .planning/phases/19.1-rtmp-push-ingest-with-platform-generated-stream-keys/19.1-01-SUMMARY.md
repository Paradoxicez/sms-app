# Phase 19.1 Plan 01: Wave 1 Foundation Summary

Land every type, utility, error class, DTO, and SRS service helper that Waves 2–3 need to compile. Prisma schema gains the `ingestMode` / `streamKey` / `firstPublishAt` columns + the global `@@unique([streamKey])` constraint; live DB migrated through the `db:push` chain (camera_stream_url_unique → `prisma db push` → camera_push_fields → rls_apply_all). 14 Wave-0 `it.todo` scaffolds promoted to real passing vitest cases.

## What was built

### Prisma schema (Task 1)

Extended `Camera` with:

- `ingestMode  String   @default("pull")` — pull | push discriminator (D-04)
- `streamKey   String?`                   — nanoid(21) target, null while pull
- `firstPublishAt DateTime?`              — seeds `camera.push.first_publish` (D-21)
- `@@unique([streamKey])`                 — global; URL carries no orgId
- `@@index([ingestMode])`                 — cheap filter acceleration

`@@unique([orgId, streamUrl])` preserved for Phase 19's pull-mode dedup. `db:push` executed against `DATABASE_URL_MIGRATE`; all 3 columns + unique index + seeded rows verified in live DB (see Self-Check below).

### Stream-key primitives (Task 2)

`apps/api/src/cameras/stream-key.util.ts`:

| Export              | Contract                                                 |
| ------------------- | -------------------------------------------------------- |
| `generateStreamKey` | 21-char URL-safe nanoid; 10k calls yield 10k unique keys |
| `maskStreamKey`     | first-4 + `…` + last-4; collapses to `…` if ≤ 8 chars    |
| `streamKeyPrefix`   | first 4 chars only (for audit logs per D-21)             |
| `buildPushUrl`      | `rtmp://{host}:1935/push/{key}` canonical shape          |

`DuplicateStreamKeyError` mirrors `DuplicateStreamUrlError` — HTTP 409, `code: DUPLICATE_STREAM_KEY`, fixed user-facing message.

`CodecInfo` extended:

- `CodecInfoStatus` union gains `'mismatch'` (4 states total)
- New optional `CodecInfo.mismatchCodec?: string` for Plan 04 to record the codec the camera actually sent when passthrough mode sees non-H.264/AAC.

### DTOs + SRS service (Task 3)

- `CreateCameraSchema` and `BulkImportCameraSchema` now discriminate on `ingestMode` via `superRefine`: pull rows require a valid allowlisted `streamUrl`; push rows must leave it empty (D-13).
- `UpdateCameraSchema` wraps in `.strict()` so any `ingestMode` key in the body is rejected at the zod boundary — no silent strip.
- `OnForwardSchema` / `OnForwardDto` added at `apps/api/src/srs/dto/on-forward.dto.ts` for Plan 02's SRS forward-backend callback.
- `SrsApiService` gains two helpers:
  - `findPublisherClientId(streamPath)` — GET `/api/v1/clients`, match `client.url` endsWith `/{streamPath}` + publish-type filter; returns null on miss/unreachable.
  - `kickPublisher(clientId)` — DELETE `/api/v1/clients/{id}`, throws on non-2xx.

## Deviations from Plan

### [Rule 1 — Bug] `maskStreamKey` expected-value typo in plan

- **Found during:** Task 2 verification (RED phase for `maskStreamKey returns first-4 + ellipsis + last-4`).
- **Issue:** Plan dictated `expect(maskStreamKey('abcdefghijklmnopqrstu')).toBe('abcd…qrstu')` but the 21-char input has last-4 `'rstu'`, not `'qrstu'` (which would be last-5). The behavior section literally reads "first-4 + ellipsis + last-4", so the implementation (`${key.slice(0,4)}…${key.slice(-4)}`) was correct.
- **Fix:** Corrected the test expectation to `'abcd…rstu'`; added an inline comment showing the character breakdown to prevent future regression.
- **Files modified:** `apps/api/tests/cameras/stream-key.util.test.ts`
- **Commit:** `f8d3cea`
