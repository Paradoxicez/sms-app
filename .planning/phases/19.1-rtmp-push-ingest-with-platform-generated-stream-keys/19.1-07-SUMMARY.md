# Phase 19.1 Plan 07: Wave 4 Frontend B — ViewStreamSheet Push Surface

Compose the push-specific UI inside the existing camera detail sheet: a new Push URL card (owner-only), a codec-mismatch banner wired to the accept-auto-transcode flow, a first-publish empty state, and a rotate-key alert-dialog. Four new composites, integration into `view-stream-sheet.tsx`, and 10 passing tests across two `.spec.tsx` files. Parallel with Plan 06 in Wave 4 — no file overlap.

## What was built

### Four new composites (Task 1, commit `134a7c4`)

**PushUrlSection** (`push-url-section.tsx`)
- Card with heading `Push URL` + description `Paste this into your camera or encoder.` (UI-SPEC verbatim)
- Read-only `<Input aria-label="Push URL" className="font-mono text-xs">` containing the full streamUrl
- Two inline icon buttons (outline variant) wrapped in shadcn tooltips: `Copy` (Copy icon) + `Rotate` (RotateCw icon)
- `Setup guide →` link aligned right
- Copy handler writes to clipboard + toast "Copied to clipboard" (plus error toast on rejection)
- Rotate handler opens `<RotateKeyDialog>` (embedded, not prop-injected)

**CodecMismatchBanner** (`codec-mismatch-banner.tsx`)
- Amber `role="alert"` div; renders nothing if `normalizeCodecInfo(codecInfo)?.status !== 'mismatch'`
- Body interpolates `info.mismatchCodec ?? info.video?.codec ?? "an unsupported codec"` (3-step fallback ladder — keeps sentence grammatical when probe didn't name the codec)
- Two CTAs: `Enable auto-transcode` (default variant, calls `onAccept`) and `Dismiss` (outline variant, calls `onDismiss`)
- AlertTriangle icon with amber color + `mt-0.5` for optical alignment

**WaitingForFirstPublish** (`waiting-for-first-publish.tsx`)
- Dashed-border muted-bg Card, centered layout
- Loader2 icon at 60% opacity **without** `animate-spin` — per UI-SPEC this is a calm empty state, not a spinner
- Auto-hides when `firstPublishAt !== null` OR `status === 'online'`
- Troubleshooting checklist with 3 items (UI-SPEC-verbatim bullets): port reachability, codec recommendation, URL-paste correctness
- `See full guide →` link opens `/docs/push-setup` in new tab
- Uses `&apos;` HTML entity for apostrophe in "your camera's network" to satisfy react/no-unescaped-entities

**RotateKeyDialog** (`rotate-key-dialog.tsx`)
- Alert-dialog with phase-machine state `'confirm' | 'reveal'`
- Confirm body: title `Rotate push key?` + description referencing `disconnected immediately` + `new URL must be copied` (verbatim)
- Destructive CTA styled with `bg-destructive text-destructive-foreground hover:bg-destructive/90`
- On confirm: `POST /api/cameras/:id/rotate-key` with `credentials: 'include'`, preventDefault on click to avoid auto-close
- Success path swaps body to `<CreatedUrlReveal title="Key rotated" helperText="Old key invalidated. Update your camera to resume publishing.">` — the helper-text override is the whole point of the shared composite (Plan 05 handed us this prop explicitly)
- Error path: toast "Failed to rotate key. The old key is still valid — try again." + stays in confirm phase (the old key remains authoritative per threat_model T-19.1-ROTATE-RACE)
- `handleClose` resets phase to `'confirm'` so the next open starts fresh

### view-stream-sheet integration + 10 tests (Task 2, commit `fb23863`)

- **CameraRow type extension** (cameras-columns.tsx) — added four OPTIONAL fields: `ingestMode`, `streamKey`, `firstPublishAt`, `needsTranscode`. No runtime behavior change; existing pull-camera call-sites remain source-compatible.
- **ViewStreamContent composition** — Preview tab body now (top → bottom):
  1. `<CodecMismatchBanner>` (gated on `showMismatch`)
  2. `<HlsPlayer>` (existing)
  3. `<PushUrlSection>` (gated on `isPush`)
  4. `<WaitingForFirstPublish>` (gated on `isPush`)
  5. Existing Camera Info + ResolvedPolicyCard grid
- **Accept auto-transcode handler** — PATCHes `/api/cameras/:id` with `{ needsTranscode: true }` (reuses the Phase 19 update endpoint per plan's explicit fallback); on success, sets local `mismatchDismissed=true` + shows toast "Auto-transcode enabled. Retry publish from your camera."; on failure, error toast + banner stays visible.
- **Pull-camera behavior unchanged** — none of the push composites mount when `camera.ingestMode !== 'push'`; sheet matches pre-19.1 appearance.
- **ViewStreamContent now exported** — enables future integration tests to wrap it without the Sheet portal; default wrapper ViewStreamSheet behavior unchanged.
- **onRefresh prop** threaded through both the outer ViewStreamSheet and inner ViewStreamContent so callers can revalidate the camera record after a rotation.

## Deviations from Plan

### [Rule 3 — Blocking] CameraRow lacked push-camera fields

- **Found during:** Task 2 integration — TypeScript rejected `camera.ingestMode` / `camera.streamKey` / `camera.firstPublishAt` / `camera.needsTranscode` accesses inside ViewStreamContent.
- **Issue:** Neither Plan 06 nor Plan 07's files_modified list includes `cameras-columns.tsx`, but Plan 07 directly reads these fields from the sheet-provided camera. Plan 03 extended the backend Camera Prisma model with these fields and serializeCamera emits them on the owner payload, but the frontend `CameraRow` type was still pre-19.1. Without the type extension, the composition wiring fails typecheck and the whole plan is blocked.
- **Fix:** Added 4 OPTIONAL fields to `CameraRow` (each with `?` modifier). Zero runtime behavior change; every existing pull-camera call-site remains source-compatible. Added JSDoc comments referencing D-01/D-07/D-16/D-26 so future readers understand the ownership.
- **Files modified:** `apps/web/src/app/admin/cameras/components/cameras-columns.tsx`
- **Commit:** `fb23863`

### [Rule 2 — Critical functionality] Exported ViewStreamContent + added onRefresh prop

- **Found during:** Task 2 action step 3 (JSX composition) — the plan's snippet referenced `refreshCamera?.()` inside PushUrlSection's onRotated handler but neither the existing sheet nor the plan defined such a prop.
- **Issue:** After a rotation, the caller needs to be able to revalidate the camera payload (new streamUrl) — without this hook, the sheet would keep showing the stale pre-rotation URL until manual close+reopen. This is a plan-required flow ("triggers the parent's camera refetch").
- **Fix:** Added optional `onRefresh?: () => void` prop to both ViewStreamSheetProps and the inner ViewStreamContent. Wired through so `onRotated → onRefresh?.()` and the accept-auto-transcode success path also calls `onRefresh?.()`. Exported `ViewStreamContent` for future integration-test reachability (the plan's sample test imports it directly).
- **Files modified:** `apps/web/src/app/admin/cameras/components/view-stream-sheet.tsx`
- **Commit:** `fb23863`

| T-19.1-BANNER-BYPASS | accept (per plan) | `mismatchDismissed` is component-local state — next sheet mount re-reads camera.codecInfo and re-shows the banner if the backend probe hasn't cleared the mismatch. Defense in depth: Plan 04's probe processor also kicks the publisher, so a dismissed banner doesn't let a mismatched publish actually succeed. |
