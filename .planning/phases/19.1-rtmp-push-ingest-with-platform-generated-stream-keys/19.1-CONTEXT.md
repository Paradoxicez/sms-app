# Phase 19.1: RTMP push ingest with platform-generated stream keys — Context

**Gathered:** 2026-04-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Cameras / NVRs / encoders publish RTMP **directly** to our SRS on port 1935 using a **platform-generated stream key**, so that a developer can register a camera, copy the generated `rtmp://…/push/<key>` URL, paste it into their encoder, and have playback work end-to-end without ever running FFmpeg against an external RTSP/RTMP source (the Phase 19 "pull" model).

**In scope:**
- Stream-key generation, storage, rotation, and audit for push cameras
- New `push/<key>` SRS app path + `on_publish` authentication against the DB-resolved stream key
- Data-model extension: `ingestMode` enum + `streamKey` column alongside the existing `streamUrl`
- Add Camera UI: Pull/Push segmented control + post-save URL reveal in the dialog
- Bulk Import: `ingestMode` column in CSV + "Download URLs (CSV)" for generated push URLs
- Coexistence with existing Stream Profile system (Passthrough vs Transcode drives the FFmpeg path)
- Per-camera codec-mismatch consent flow (Passthrough profile + non-H.264/AAC → reject + prompt → user opts into auto-transcode)
- Push camera lifecycle (delete-while-publishing, maintenance mode behavior, key masking)

**Explicitly NOT in scope:**
- **RTMPS (TLS)** ingest — deferred to a future SRS v6→v7 migration phase once v7 is GA (late 2026). See PRE-DISCUSS-NOTES for rationale.
- New Stream Profile capabilities — reuse existing Passthrough/Transcode from v1.0
- Recordings, playback policy, session tokens — unchanged
- Org-level default codec-mismatch policy — only per-camera consent in this phase
- Test-publish probe action in the UI — existing status badge + codec column are sufficient
- Encrypted-at-rest stream keys — plain storage (SRS needs the raw value on `on_publish`), masked in responses and audit logs
- Multi-format copy affordance (FFmpeg CLI snippet / OBS JSON) — URL-only copy + docs link
- Block-delete while publishing — we kick the active publisher and hard-delete
- Soft-delete / grace-period key invalidation — rotation force-disconnects immediately

</domain>

<decisions>
## Implementation Decisions

### Stream-key & URL scheme

- **D-01:** Push URL format is `rtmp://{host}:1935/push/<key>`. `push` is a new SRS app distinct from `live/{orgId}/{cameraId}` (the existing internal pull publish target). No `orgId` in the URL — the key alone must resolve to a camera via DB.
- **D-02:** Stream-key format is **nanoid 21 chars** (URL-safe alphabet, 128-bit entropy). Use the `nanoid` npm package. No hyphens from the alphabet are required — the default URL-safe alphabet is fine. 21 chars was chosen over UUID v4 (shorter, no delimiter confusion when copied into encoder fields) and over a 12-char slug (insufficient entropy for an effectively-password credential).
- **D-03:** `on_publish` for the `push` app resolves the key with `prisma.camera.findFirst({ where: { streamKey } })` — a single lookup, no `orgId` filter. After resolution, `camera.orgId` + `camera.id` are used for all downstream routing (status transitions, refresh probe, recordings).

### Data model

- **D-04:** Extend `Camera` in `apps/api/src/prisma/schema.prisma`:
  - `ingestMode String @default("pull")` — enum-like, values `"pull" | "push"`. Not a Prisma `enum` so we avoid a migration when adding future modes.
  - `streamKey String?` — nullable; only populated when `ingestMode = "push"`.
  - Add `@@unique([streamKey])` — global unique, not per-org. Justified because the URL carries no `orgId`, so the key is the sole routing primitive. nanoid 21's 128-bit entropy makes collision statistically impossible, but the constraint is the DB-level guarantee.
- **D-05:** Keep `streamUrl` as a stored column for push cameras — write the full generated URL (`rtmp://{host}:1935/push/<key>`) at create time. This keeps API response shape uniform (pull and push both have `streamUrl`) and preserves the existing `@@unique([orgId, streamUrl])` from Phase 19 D-10c.
- **D-06:** Existing rows at migration time → `ingestMode = "pull"`, `streamKey = NULL`. No data migration beyond setting the default column value. `streamUrl` values unchanged.
- **D-07:** Stream-key stored **plain** in DB (SRS must receive the raw key in `on_publish` body, so we have to compare against plain values — no hashing is possible without a proxy in front of SRS, which is out of scope). Mitigate exposure by: (a) returning full URL only to the owning org, (b) masking to `<first-4>…<last-4>` in API responses for other contexts, (c) logging only the 4-char prefix in audit entries, (d) never rendering the key in non-owner UI surfaces.

### Add Camera UX

- **D-08:** Add a **segmented control** `[ Pull | Push ]` at the top of `camera-form-dialog.tsx`, above the name field. Selection drives the body:
  - **Pull** (default, backward-compatible): existing form — user types `streamUrl`.
  - **Push**: `streamUrl` input is hidden. A non-interactive hint block explains "We'll generate a push URL after you save. Configure your camera/NVR to publish to it (H.264+AAC recommended)." plus a link "Setup guide →".
- **D-09:** On submit for push mode: dialog **stays open**, switches to a "Created" state showing: generated URL + Copy button + short config hint + "Done" CTA. No navigation, no toast-and-close flow. User copies → clicks Done → dialog closes. User can still view/copy the URL later from the camera detail sheet.
- **D-10:** In-dialog help is **inline text + docs link only** — no accordion "Show examples" block, no tabbed encoder templates in the dialog. Depth moves to external docs.
- **D-11:** Copy affordance is **URL only** — single Copy button, no dropdown of formats (FFmpeg CLI snippet, OBS JSON, etc.). A "Setup guide" link next to Copy points users at a docs page with format examples.

### Bulk Import UX

- **D-12:** Bulk import CSV accepts a **mixed** template with a new `ingestMode` column. Existing rows (`rtsp://…`) implicitly infer `ingestMode = "pull"` if the column is absent — backward-compatible. Push rows set `ingestMode = "push"` and **leave `streamUrl` empty**; the server generates `streamKey` + `streamUrl` during import.
- **D-13:** Import DTO (`bulk-import.dto.ts`) refines per row:
  - `ingestMode = "pull"` → require `streamUrl` matching the Phase 19 allowlist (`rtsp|rtmp|rtmps|srt`).
  - `ingestMode = "push"` → require `streamUrl` to be empty/absent. Reject if the CSV supplies a streamUrl for a push row (defensive — prevents user confusion).
- **D-14:** After import, the result panel shows summary counts (imported / skipped / duplicate) — unchanged from Phase 19 — plus, when any push rows were created, a **"Download URLs (CSV)"** button that exports `name,streamUrl` for only the push rows of this import. No auto-download, no popup.

### SRS on_publish authentication

- **D-15:** Extend `parseStreamKey` in `srs-callback.controller.ts` to branch on `app`:
  - `app = "live"` → existing `live/{orgId}/{cameraId}` parser (no change).
  - `app = "push"` → treat `stream` as the streamKey; resolve via `prisma.camera.findFirst({ where: { streamKey }, include: { /* orgId */ } })`. If not found → return `{ code: 403 }` (SRS disconnects). If found → status transition + codec refresh, identical to the pull path.
- **D-16:** When the camera is resolved but the Stream Profile is **Passthrough** and the first successful probe (via SRS `/api/v1/streams`) shows a codec that is **not H.264/AAC**:
  - `on_publish` returns `{ code: 403 }` — publish rejected.
  - `Camera.codecInfo` is set to `{ status: "mismatch", reason: "Passthrough requires H.264/AAC, camera sent <codec>", probedAt }` — extends the Phase 19 tagged-union with a new `mismatch` status.
  - A per-camera banner appears in the camera detail sheet: "Camera is sending <codec>. Your profile is Passthrough (requires H.264/AAC). Switch to auto-transcode? This adds CPU/latency/bandwidth overhead." with Accept / Dismiss buttons.
  - On Accept → backend updates the camera's `streamProfileId` / `needsTranscode` to use the system default transcode profile (or user-chosen one — planner decides). Next publish attempt succeeds via the FFmpeg transcode path.
  - On Dismiss → camera stays in `mismatch` state; next publish re-triggers the same banner.

### Zero-transcode & Stream Profile integration

- **D-17:** Phase 19.1 does **not** introduce a new "zero-transcode vs transcode" architectural toggle. The existing Stream Profile (`Passthrough` vs `Transcode`) drives the path for push cameras, same as for pull:
  - **Push + Passthrough + H.264/AAC** → zero-transcode. SRS generates HLS directly from the RTMP publish. No FFmpeg in the path. This is the happy-path optimization.
  - **Push + Transcode profile** → FFmpeg is spawned (same BullMQ `stream` queue as pull) with **input URL = `rtmp://127.0.0.1:1935/push/<key>`** (SRS loopback), transcodes per profile, publishes output to `rtmp://127.0.0.1:1935/live/{orgId}/{cameraId}`. Reuses `FfmpegCommandBuilder`, `StreamProcessor`, `StreamService` — only the input URL source changes based on `ingestMode`.
  - **Push + Passthrough + non-compatible codec** → rejected until user consents to auto-transcode (D-16).
- **D-18:** Playback URL contract is preserved as `live/{orgId}/{cameraId}/index.m3u8` for **both** pull and push cameras. For push+passthrough this requires SRS to emit HLS at `live/{orgId}/{cameraId}` even though the RTMP was published to `push/<key>`. The planner/researcher chooses the exact mechanism: SRS `forward` directive, a backend-spawned FFmpeg `-c copy` loopback relay (push → live), or SRS `on_publish` hook returning a forward target. All three are viable; the tradeoff is CPU cost (relay ≈ negligible with `-c copy`) vs configuration complexity. **Requirement:** do not leak `streamKey` into any playback URL, and do not fork the playback controller or `hls.js` integration.

### Stream-key rotation

- **D-19:** Rotation action is exposed **only in the camera detail sheet** (not in the DataTable row dropdown). Reduces accidental rotation on a sensitive operation that requires user to reconfigure the encoder.
- **D-20:** Rotation flow: user clicks Rotate → confirmation dialog warning the active publish will be disconnected → server generates new nanoid, updates `Camera.streamKey` + `streamUrl` transactionally → then calls SRS `/api/v1/clients/{id} DELETE` to kick the current publisher using the old key → reveal new URL in the same dialog (same pattern as D-09). **Force-disconnect is immediate**, no grace period with dual keys.

### Audit events

- **D-21:** Push-specific audit events to emit (extending the Phase 14 audit interceptor):
  - `camera.push.key_generated` — on create of a push camera. Payload: `{ cameraId, streamKeyPrefix: key.slice(0, 4) }`. Never log the full key.
  - `camera.push.key_rotated` — on rotation. Payload: `{ cameraId, oldKeyPrefix, newKeyPrefix, rotatedBy }`.
  - `camera.push.publish_rejected` — on `on_publish` rejection due to unknown/rotated key OR codec mismatch on passthrough. Payload: `{ streamKeyPrefix, reason, clientIp }`. Security signal — surfaces brute-force patterns.
  - `camera.push.first_publish` — the first successful `on_publish` for a given camera (tracked via a boolean flag on the camera row, flipped on first success). Payload: `{ cameraId, codec, resolution }`. Not every publish — just first-time.

### Lifecycle

- **D-22:** Delete push camera while publishing → backend calls SRS `/api/v1/clients/{id} DELETE` to kick the publisher, then hard-deletes the row (existing cascade handles recordings, playback sessions). Do not soft-delete and do not block the deletion.
- **D-23:** Maintenance mode + push camera (Phase 15 pattern) → **accept** the publish, stream works normally, HLS is generated; only notifications and webhooks are suppressed by the existing `StatusService` maintenance gate. Rationale: maintenance in Phase 15 means "silence the operators", not "stop the stream". Push path follows the same semantics as pull.
- **D-24:** Double-publish on the same key (two clients publishing to `push/<key>` simultaneously) → rely on SRS default behavior. SRS rejects a second concurrent publisher on the same stream path; no custom logic required. Document this in the onboarding docs.

### Onboarding

- **D-25:** Verification UX is **the existing status + codec columns** (Phase 15 status icons + Phase 19 codec column). No "Test push" button. When the camera publishes successfully, the row's status transitions to online (broadcast over the WebSocket from the Phase 19 fix) and the codec column populates — that's sufficient signal.
- **D-26:** Empty-state guidance in camera detail (push, no publish yet) shows "Waiting for first publish…" plus a minimal troubleshooting checklist (port 1935 reachable, codec H.264+AAC recommended, link to setup guide). Avoid a full runbook in the UI — deep troubleshooting belongs in docs.

### Claude's Discretion

- Exact wording of the "Configure your camera" hint in the Add Camera dialog (English, concise, matching existing tone)
- Exact copy for the codec-mismatch consent banner — keep it honest about the CPU cost tradeoff
- Choice between SRS `forward` directive vs backend-spawned `-c copy` FFmpeg relay for push-passthrough (D-18) — researcher/planner evaluates tradeoffs
- Whether `bulk-import.dto.ts` should emit a clearer error when a user puts a value in `streamUrl` for an `ingestMode = "push"` row (recommend: explicit 400 with row index) — planner decides phrasing
- `system transcode profile` used when user consents to auto-transcode on codec mismatch (D-16): pick an existing profile that's reasonable for IP cameras (e.g., 720p H.264 4Mbps) or create a new `default-auto-transcode` profile — planner picks
- Migration strategy if the `@@unique([streamKey])` constraint collides with unusual seed data — same Claude's Discretion fallback as Phase 19 D-10 (dedup query vs fail-with-runbook)
- Index strategy: `Camera.streamKey` unique already implies an index, but a partial index `WHERE streamKey IS NOT NULL` is optional and not required for correctness

</decisions>

### Established Patterns
- **BullMQ `jobId` idempotency** — every new queue or new enqueue pattern uses `{queue}:{cameraId}` to auto-dedup. For push, rotation kick can reuse this (e.g., `rotate:{cameraId}` with a short TTL) or be done synchronously — planner decides.

- **Response masking for sensitive fields** — API Keys (v1.0) use a "copy once, mask forever" pattern. StreamKeys should follow the same contract: the owner sees the full URL in the camera detail and immediately post-create; other contexts see `<first-4>…<last-4>`.

<specifics>
## Specific Ideas

- The RTMPS deferral (PRE-DISCUSS-NOTES) is firm and load-bearing: do not let downstream discussions re-open it as a "while we're here" opportunity. The migration to SRS v7 is its own future phase and bundles RTMPS, native RTSP output, and any v7 breaking changes in one atomic swap.
- Stream-key ≈ password. Treat it like an API key: shown once, masked elsewhere, rotatable, audited. Never render in logs, never in non-owner UI.
- Stream Profile = Passthrough + codec mismatch flow (D-16) is the most subtle UX in the phase. The rejection has to be clear enough that a user doesn't assume "the platform is broken" when their first publish is disconnected. Banner copy matters; loop in user for wording during the UI sub-phase if one is created.
- The `push/<key>` → `live/{orgId}/{cameraId}` remap (D-18) is the architectural lynchpin that keeps the playback side untouched. If the researcher finds SRS cannot do dynamic forward, the backend `-c copy` loopback relay is the fallback — it's cheap (no codec work) but adds ~1 process per active push camera, same blast-radius as pull. Worth flagging if process count becomes a scaling concern.
- Phase 19's user quote in 19-CONTEXT §specifics — "เราต้องเป็นคนสร้าง url path แล้วเอาไปใส่กล้อง/NVR" — is the motivating statement for this whole phase. Design choices should serve that mental model: user gets a URL, pastes it, camera works.

</specifics>

<deferred>
## Deferred Ideas

- **RTMPS push ingest** — owned by the future SRS v6→v7 migration phase (late 2026). This phase ships plain RTMP only.
- **Multi-format copy** (FFmpeg CLI snippet, OBS JSON, encoder-specific presets) — docs-only in this phase; a richer "copy as X" dropdown can ship later if user data shows demand.
- **Org-level default codec-mismatch policy** — only per-camera consent now. An org-wide toggle ("always auto-transcode on mismatch") could ship in a later phase once real usage patterns surface.
- **Test-publish action** — rejected in favor of relying on the existing status badge + codec column + real-time WS updates. Revisit only if support tickets indicate users can't tell whether their push is working.
- **Encrypted-at-rest stream keys** — rejected (SRS needs raw). If compliance later demands encrypted credentials, the path is a reverse-proxy in front of SRS that decrypts on `on_publish`, not an app-level change.
- **Grace-period rotation** (keep old key alive for N minutes) — rejected; immediate force-disconnect. Reopen only if a concrete operational pain point emerges.
- **Push-specific onboarding wizard / walkthrough** — inline hint + docs link is the starting point. A dedicated wizard is overkill for v1 of this feature.
- **Transcode-on-push without consent (auto-upgrade silently)** — rejected. Users must opt in because it has real cost (CPU/latency/bandwidth).
- **Per-camera lifecycle states beyond online/offline/maintenance/recording** — e.g., a dedicated "awaiting first publish" badge. Explicitly kept to the existing 4-state set plus codec column; D-26 uses empty-state guidance in detail instead of a new badge.

</deferred>
