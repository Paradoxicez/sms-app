# Phase 19.1: RTMP push ingest with platform-generated stream keys — Research

**Researched:** 2026-04-23
**Domain:** RTMP push ingest on SRS v6 + NestJS service integration + platform-generated stream keys
**Confidence:** HIGH (every load-bearing decision — SRS `forward` path rewrite, `/api/v1/clients` kick, nanoid shape, SRS `/api/v1/streams` race, CSV mixed-mode DTO — is either verified against official docs or already established in code by Phases 19/15/14.)

## Summary

Phase 19.1 adds a **push** ingest path alongside Phase 19's pull path. Cameras publish RTMP directly to our SRS at `rtmp://host:1935/push/<key>`; the SRS `on_publish` callback resolves `<key>` to a Camera row, runs the same status-transition + codec-refresh pipeline the pull path already uses, and then (for passthrough profiles) forwards the stream to the canonical `live/{orgId}/{cameraId}` path so the existing playback URL contract (`/live/{orgId}/{cameraId}/index.m3u8`) is preserved. For transcode profiles, a new FFmpeg input URL is read from the loopback `rtmp://127.0.0.1:1935/push/<key>` instead of an external camera URL — everything downstream (`FfmpegCommandBuilder`, `StreamProcessor`, HLS emit) is untouched.

The research resolves 11 load-bearing unknowns. The most consequential finding is that **SRS v6's `forward` directive with a `backend` HTTP hook supports arbitrary target-path rewriting** — i.e., a publish to `push/<key>` can be forwarded to `rtmp://127.0.0.1:1935/live/{orgId}/{cameraId}` and the forwarded stream generates HLS at the target vhost's config. This makes D-18 a configuration problem, not a backend-spawned-FFmpeg problem, keeping the push + passthrough path zero-transcode and process-free.

**Primary recommendation:** Use SRS v6 `forward` with a `backend http://api:3003/api/srs/callbacks/on-forward` hook for the push→live remap. Fall back to a backend-spawned `ffmpeg -c copy` loopback relay only if operational testing reveals SRS forward reliability issues (unlikely — it's a first-class SRS feature since v2).

### Pattern 1: SRS Push→Live Remap via `forward` + Backend Hook (D-18)

**What:** When a client publishes RTMP to `push/<key>`, SRS POSTs to `/api/srs/callbacks/on-forward` asking where to forward. Our backend looks up the key → returns `rtmp://127.0.0.1:1935/live/{orgId}/{cameraId}`. SRS internally forwards the stream to that path, which is a second publish into the same SRS instance — the `live/...` stream then emits HLS at the canonical playback URL automatically (vhost-level HLS config applies to all apps).

**When to use:** Push + Passthrough profile (zero-transcode happy path). For Push + Transcode, the FFmpeg wrapper already handles the path (reads from `rtmp://127.0.0.1:1935/push/<key>`, writes to `live/...`), so the `forward` hook should **return empty urls** to suppress duplicate forwarding. The hook decides per-publish based on `camera.needsTranscode`.

**Why this over FFmpeg relay:** Zero extra processes, no CPU cost, handled entirely inside SRS. First-class SRS feature since v2. Path-rewriting is explicitly supported per SRS v6 docs `[CITED: https://ossrs.net/lts/en-us/docs/v6/doc/forward]`.

**Example srs.conf addition:**
```nginx
vhost __defaultVhost__ {
    # ...existing hls + http_hooks blocks unchanged...

    forward {
        enabled     on;
        backend     http://host.docker.internal:3003/api/srs/callbacks/on-forward;
    }
}
```

**Example backend handler:**
```typescript
// apps/api/src/srs/srs-callback.controller.ts
@Post('on-forward')
async onForward(@Body() body: OnForwardDto) {
  if (body.app !== 'push') {
    // Internal live/... forward target — do not recursively forward.
    return { code: 0, data: { urls: [] } };
  }
  const streamKey = body.stream;
  const camera = await this.systemPrisma.camera.findFirst({
    where: { streamKey },
    select: { id: true, orgId: true, needsTranscode: true },
  });
  if (!camera) {
    // Unknown key — reject publish by returning non-zero.
    // But on_publish is the auth chokepoint; on_forward firing
    // means on_publish already allowed. Return empty urls to
    // let the push stream exist at push/<key> without forwarding.
    return { code: 0, data: { urls: [] } };
  }
  if (camera.needsTranscode) {
    // Transcode profile handles forwarding via FFmpeg wrapper;
    // suppress SRS forward to avoid double-publish at live/.
    return { code: 0, data: { urls: [] } };
  }
  // Zero-transcode forward: push/<key> → live/{orgId}/{cameraId}
  const srsHost = process.env.SRS_HOST || '127.0.0.1';
  return {
    code: 0,
    data: {
      urls: [`rtmp://${srsHost}:1935/live/${camera.orgId}/${camera.id}`],
    },
  };
}
```

**Critical invariant:** The forwarded stream publishes back into SRS at `live/{orgId}/{cameraId}`, which fires a SECOND `on_publish` callback for `app=live`. This is the **existing** pull-path codec-refresh flow — the same StatusService transition + `enqueueProbeFromSrs` already runs. The `on_publish` `push` branch must **not** trigger the probe refresh (the live-branch re-fires and does it with ground truth). Verified: Phase 19's existing `on_publish` handler already idempotency-dedups via `jobId = "probe-{cameraId}-srs-api"`.

### Pattern 2: Stream Key Generation Utility

```typescript
// apps/api/src/cameras/stream-key.util.ts
import { customAlphabet } from 'nanoid';

// nanoid v3: CommonJS import works with NestJS + ts-node + module: "commonjs"
// Default URL-safe alphabet: A-Za-z0-9_- (64 chars → ~6 bits/char × 21 chars = 126 bits)
// The canonical nanoid() call uses the default alphabet at 21 chars.
import { nanoid } from 'nanoid';

const STREAM_KEY_LENGTH = 21;

export function generateStreamKey(): string {
  return nanoid(STREAM_KEY_LENGTH);
}

/** Mask for non-owner surfaces: first 4 + "…" + last 4 */
export function maskStreamKey(key: string): string {
  if (key.length <= 8) return '…'; // defensive
  return `${key.slice(0, 4)}…${key.slice(-4)}`;
}

/** Audit prefix — first 4 chars, safe for logs */
export function streamKeyPrefix(key: string): string {
  return key.slice(0, 4);
}

/** Build the full push URL from a key */
export function buildPushUrl(host: string, key: string): string {
  return `rtmp://${host}:1935/push/${key}`;
}
```

### Pattern 3: `on_publish` Branch for Push App

```typescript
// apps/api/src/srs/srs-callback.controller.ts — extend parseStreamKey
@Post('on-publish')
async onPublish(@Body() body: OnPublishDto) {
  const app = body.app ?? '';
  const stream = body.stream ?? '';

  // Push app: stream = streamKey, resolve via DB lookup (D-15)
  if (app === 'push') {
    const camera = await this.camerasService.findByStreamKey(stream);
    if (!camera) {
      await this.auditService.log({
        orgId: 'system', // unknown camera — system-level audit
        action: 'camera.push.publish_rejected',
        resource: 'camera',
        method: 'POST',
        path: '/api/srs/callbacks/on-publish',
        ip: body.ip,
        details: {
          streamKeyPrefix: streamKeyPrefix(stream),
          reason: 'unknown_key',
        },
      });
      return { code: 403 }; // SRS disconnects publisher
    }

    // D-23: maintenance mode doesn't block publish — suppression happens in StatusService.
    // D-16: codec-mismatch detection happens AFTER this on_publish returns { code: 0 }
    //       — we enqueue a probe (source: 'srs-api', delay 1000ms) and the probe writes
    //       status='mismatch' if passthrough + non-H.264/AAC.
    //       Note: by this point the publish is already allowed; rejection happens on
    //       NEXT publish after the user sees the banner. If we need to reject in-flight,
    //       we'd have to wait for probe before returning — rejected because it adds
    //       1s+ latency to every on_publish. Better: let it run, probe, flag, kick
    //       via /api/v1/clients DELETE if needsTranscode and profile=Passthrough.
    //       SEE: §Pitfalls for the mismatch-detection timing contract.

    await this.statusService.transition(camera.id, camera.orgId, 'online');
    await this.camerasService.enqueueProbeFromSrs(camera.id, camera.orgId, { delay: 1000 });
    await this.camerasService.markFirstPublishIfNeeded(camera.id, camera.orgId);
    return { code: 0 };
  }

  // Live app (internal forward target OR pull path) — existing behavior
  const { orgId, cameraId } = this.parseStreamKey(stream, app);
  // ... existing pull-path code unchanged ...
}
```

### Pattern 4: Rotate Stream Key

```typescript
// apps/api/src/cameras/cameras.service.ts
async rotateStreamKey(cameraId: string, userId: string): Promise<{ streamUrl: string }> {
  const camera = await this.tenancy.camera.findUnique({ where: { id: cameraId } });
  if (!camera) throw new NotFoundException();
  if (camera.ingestMode !== 'push') {
    throw new BadRequestException('Only push cameras can have their stream key rotated');
  }

  const oldKey = camera.streamKey!;
  const oldClientId = await this.srsApi.findPublisherClientId(`push/${oldKey}`);
  const newKey = generateStreamKey();
  const srsHost = process.env.SRS_PUBLIC_HOST || 'localhost';
  const newUrl = buildPushUrl(srsHost, newKey);

  // Transactional update in the tenancy session
  const updated = await this.tenancy.$transaction(async (tx: any) => {
    try {
      return await tx.camera.update({
        where: { id: cameraId },
        data: { streamKey: newKey, streamUrl: newUrl },
      });
    } catch (error) {
      if (error instanceof Prisma.PrismaClientKnownRequestError && error.code === 'P2002') {
        const target = (error.meta?.target as string[]) ?? [];
        if (target.includes('streamKey')) throw new DuplicateStreamKeyError();
        if (target.includes('streamUrl')) throw new DuplicateStreamUrlError(newUrl);
      }
      throw error;
    }
  });

  // Kick publisher AFTER DB commit so even if the kick fails, the new key is live.
  // D-20: best-effort, log on failure. SRS will eventually disconnect on key-miss next publish.
  if (oldClientId) {
    try {
      await this.srsApi.kickPublisher(oldClientId);
    } catch (err) {
      this.logger.warn(`Kick failed for client ${oldClientId} on rotate: ${(err as Error).message}`);
    }
  }

  await this.auditService.log({
    orgId: camera.orgId,
    userId,
    action: 'camera.push.key_rotated',
    resource: 'camera',
    resourceId: cameraId,
    method: 'POST',
    path: `/cameras/${cameraId}/rotate-key`,
    details: {
      oldKeyPrefix: streamKeyPrefix(oldKey),
      newKeyPrefix: streamKeyPrefix(newKey),
    },
  });

  return { streamUrl: updated.streamUrl };
}
```

### Anti-Patterns to Avoid

- **Don't embed orgId in the push URL.** Breaks D-01/D-03 — the key alone must resolve. A longer URL is not a security win; the key's entropy is the credential.
- **Don't hash the streamKey in the DB.** SRS delivers raw `stream` on `on_publish`; we need exact-match lookup. Hashing requires a reverse-proxy (out of scope).
- **Don't strip extensions from the `push` stream value in `parseStreamKey`.** The extension-strip logic is for `on_play` (HLS segments `cam-1.ts` / `cam-1-5.m4s`); `on_publish` always receives the canonical key. The push branch must bypass the strip (otherwise a stream key starting with `cam-1.m3u8` would be mangled — unlikely with nanoid but still wrong logic).
- **Don't write `ingestMode` as a Prisma enum.** CONTEXT D-04 explicitly uses `String @default("pull")` so future additions (`webrtc`, `srt`) don't require migrations.
- **Don't send the SRS `on_forward` hook a non-zero code for unknown keys.** The `on_publish` hook is the auth chokepoint — `on_forward` just decides where to forward. Returning `{ code: 0, data: { urls: [] } }` lets the stream exist (even if unauthenticated, which shouldn't happen because on_publish fires first and rejects).
- **Don't rely on SRS `/api/v1/streams` populated at `on_publish` fire-time.** Phase 19 found a ~1s delay before SRS registry reflects the new publisher. D-02's 1000ms delay on `enqueueProbeFromSrs` is already applied — reuse, don't remove.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Unique URL-safe ID generation | Custom `crypto.randomBytes().toString('base64url')` loop | `nanoid` | Battle-tested 128-bit entropy, single line, collision-resistant by design. |
| Stream-key collision detection | Application-level loop + re-check | Prisma `@@unique([streamKey])` + P2002 translation | DB-level guarantee is race-safe; app loop is not. Pattern already established in Phase 19 (D-11). |
| RTMP push→HLS path remap | Custom FFmpeg `-c copy` relay process | SRS `forward` with `backend` hook | Native SRS feature, zero extra processes, handles reconnects automatically. Relay is the documented fallback if forward misbehaves. |
| Publisher disconnect on key rotation | Kill FFmpeg process + reconnect loop | SRS `/api/v1/clients/{id} DELETE` | Official SRS API (CLAUDE.md §4). Kicks the exact client in one call. |
| CSV mixed-mode parsing | Raw `split(',')` | Existing `csv-parse` (API) + `xlsx` helper (web) | Handles quoted fields, BOM, different line endings. Already used by Phase 19 bulk-import. |
| Masking sensitive fields in API responses | Per-endpoint `if (!isOwner) redact()` | Response interceptor or `serializeCamera(camera, { owner })` helper | Mirrors v1.0 API-key mask pattern. Single choke point prevents leak via new endpoints. |

**Key insight:** Every load-bearing primitive for Phase 19.1 already has a canonical solution that either ships with SRS or is already in the codebase from Phase 19/15/14. This phase is nearly pure composition — the temptation to roll custom must be resisted.

## Runtime State Inventory

This phase is an **additive feature**, not a rename/refactor. However, because we add a column with a unique constraint + a new SRS config block, the runtime state inventory still matters.

## Common Pitfalls

### Pitfall 1: `on_publish` timing race against SRS `/api/v1/streams`
**What goes wrong:** Probe source='srs-api' reads SRS registry before the new publisher is registered there, returns null → codec column flips to 'failed' with "Stream not found".
**Why it happens:** SRS processes the `on_publish` callback in parallel with updating its internal stream registry. The registry update can lag ~1s.
**How to avoid:** Phase 19 already applies `delay: 1000` on `enqueueProbeFromSrs`. The push branch must use the SAME delay. Do not remove or shorten.
**Warning signs:** Failed probes immediately after publish that succeed on retry.
**Confidence:** HIGH — documented and fixed in commit `e1bd458` per the recent commit log (colon separator bug). `[VERIFIED: git log / Phase 19 D-02]`

### Pitfall 2: nanoid 5 ESM break on `module: "commonjs"`
**What goes wrong:** `import { nanoid } from 'nanoid'` in a TS file compiled to CommonJS → runtime `ERR_REQUIRE_ESM` because nanoid v5 is ESM-only and Node < 22.12 doesn't transparently `require()` ESM.
**Why it happens:** nanoid dropped CommonJS in v4+. Our `apps/api/tsconfig.json` sets `module: "commonjs"` and we're on Node 22.11.0.
**How to avoid:** Install `nanoid@3.3.11` (latest in the 3.x CommonJS line). It has identical API (`nanoid(21)` and `customAlphabet`), is actively maintained, and has no security deprecations. Alternative: upgrade to Node 22.12+ (adds `require()` of ESM out-of-the-box) and then v5 works.
**Warning signs:** `ERR_REQUIRE_ESM` in `pnpm --filter @sms-platform/api start` or unit tests.
**Confidence:** HIGH `[CITED: https://github.com/ai/nanoid issue #365; nodejs.org 22.12 release notes]` `[VERIFIED: npm registry — nanoid@3.3.11 exists]`

### Pitfall 3: SRS `forward` recursion on live → live
**What goes wrong:** If the `on_forward` backend hook returns a URL for **every** publish (not just `app=push`), the forwarded `live/{orgId}/{cameraId}` publish fires `on_forward` again, and the backend responds with yet another URL — infinite loop.
**Why it happens:** `forward` fires for all publishes at the vhost level.
**How to avoid:** The backend handler must branch on `body.app`. `app === 'push'` → return forward target. `app === 'live'` (internal target) → return `{ code: 0, data: { urls: [] } }` to suppress forward. Also: FFmpeg-transcode output publishes to `live/...` — same empty-urls response applies, same code path.
**Warning signs:** SRS log shows repeated `on_forward` calls for the same stream; CPU spike on SRS; HLS never starts.
**Confidence:** HIGH `[CITED: https://ossrs.net/lts/en-us/docs/v6/doc/forward]` — explicit in docs that empty urls disables forward.

### Pitfall 4: Codec-mismatch detection timing (D-16)
**What goes wrong:** `on_publish` must return `{ code: 0 }` quickly (SRS disconnects after ~3s). But we can't know the codec until `/api/v1/streams` reflects the publisher (~1s delay). So "reject publish on codec mismatch" (D-16) can't happen IN the on_publish callback.
**Why it happens:** Architectural — probe is async, callback must be fast.
**How to avoid:** Two-step flow:
  1. `on_publish` returns `{ code: 0 }` (allows publish), enqueues probe with `delay: 1000`.
  2. Probe worker reads `/api/v1/streams`. If camera's profile is Passthrough AND codec ≠ H.264/AAC → writes `codecInfo.status = 'mismatch'` + emits `camera.push.publish_rejected` audit event + calls `srsApi.kickPublisher(clientId)` to disconnect the now-unwanted stream.
  3. Banner shows on next page load via WebSocket broadcast (Phase 19 already broadcasts codecInfo updates per `stream-probe.processor.ts:73`).
**Critical:** D-16 says "publish rejected" but the honest UX is "publish briefly accepted, then kicked on codec detection, banner tells user why next time." The 19.1-UI-SPEC banner copy ("Camera is sending {codec}. Your profile is Passthrough…") already reflects this — the camera DID send something, we just don't accept it.
**Warning signs:** User's encoder reports "connection OK" for ~1 second, then disconnects — without a banner they'd be confused. Banner + audit event are essential signal.
**Confidence:** MEDIUM — the architectural approach is sound and matches Phase 19's async-probe pattern, but the "kick on mismatch" step is new code. Needs explicit test coverage (§Validation Architecture).

### Pitfall 5: `forwardRef` cycle already exists between `CamerasModule` and `SrsModule`
**What goes wrong:** Adding a NEW service that depends on both modules (e.g., a "stream-key service" that needs camerasService + srsApi + callbacks) can re-trigger the cycle.
**Why it happens:** `SrsModule` → `CamerasModule` (via `forwardRef` per `srs.module.ts:17`). `CamerasModule` → `StreamsModule` → `SrsModule`.
**How to avoid:** Keep the new `on-forward` endpoint INSIDE `SrsCallbackController` (existing controller, already in the forwardRef chain). Keep key generation in `cameras.service.ts` (no additional inter-module dep). New `kickPublisher` method lives on `SrsApiService` (already exported from `SrsModule`).
**Warning signs:** NestJS DI error "Nest can't resolve dependencies of CamerasService/SrsApiService — circular?"
**Confidence:** HIGH — pattern already documented in `srs-callback.controller.ts:22-27` comment.

### Pitfall 6: Key masking missed in one response serializer
**What goes wrong:** Camera list endpoint masks the key, but bulk-import result endpoint returns unmasked keys — a user on a shared screen shows the list safely, then triggers bulk import and the full key appears.
**Why it happens:** Masking logic scattered across multiple response paths instead of centralized.
**How to avoid:** Create `serializeCamera(camera, { perspective: 'owner' | 'masked' })` helper. Every controller method that returns a Camera goes through it. The only `owner`-perspective surfaces are: `GET /cameras/:id` (detail sheet), `POST /sites/:id/cameras` (create response), `POST /cameras/:id/rotate-key` (reveal response). Everything else (list, bulk-import result, audit log) is masked.
**Warning signs:** Grep for `camera.streamKey` outside the helper → any hit is a potential leak.
**Confidence:** HIGH — v1.0 API-key pattern (FIX-02 context) already follows this.

### Pitfall 7: Existing `@@unique([orgId, streamUrl])` conflict with generated push URLs
**What goes wrong:** Push camera creation generates `streamUrl = rtmp://host:1935/push/<key>`. If key collides (impossibly small with nanoid-21) OR if `SRS_PUBLIC_HOST` changes such that two push cameras in the same org get the same URL → P2002 on `(orgId, streamUrl)`.
**Why it happens:** The unique constraint is on streamUrl, not streamKey. Technically impossible collision (key is globally unique), but the URL template includes `host` — if host changes, old push cameras keep the old URL, new ones get new URL, no collision. Safe.
**How to avoid:** Keep `streamUrl` in sync with `streamKey` via a single code path (transactional `camera.update({ streamKey, streamUrl })`). Test: changing `SRS_PUBLIC_HOST` doesn't corrupt existing rows (they keep their URL; new rows use new host).
**Warning signs:** P2002 on `streamUrl` during push camera create — would signal that the host changed and an old row's URL happens to match the new one (vanishingly rare).
**Confidence:** MEDIUM — relies on `SRS_PUBLIC_HOST` being stable in production.

### Example 1: Extended parseStreamKey with push branch
```typescript
// Source: extends existing apps/api/src/srs/srs-callback.controller.ts:192-214
private parseStreamKey(
  stream: string,
  app: string,
): { mode: 'push' | 'live' | 'unknown'; streamKey?: string; orgId?: string; cameraId?: string } {
  // Push branch (D-15): `app = "push"`, `stream = <nanoid-21>`
  if (app === 'push') {
    // Do NOT strip extensions — on_publish always receives canonical key.
    // Extension strip is only for on_play (HLS segments).
    return { mode: 'push', streamKey: stream };
  }

  // Live branch (existing): `app = "live"`, `stream = "{orgId}/{cameraId}"` OR app = "live/{orgId}"
  const fullPath = app ? `${app}/${stream}` : stream;
  const parts = fullPath.replace(/^live\//, '').split('/');
  if (parts.length >= 2 && parts[0] && parts[1]) {
    let cameraId = parts[1];
    const extMatch = cameraId.match(/\.(m3u8|ts|m4s|mp4|flv)$/);
    if (extMatch) {
      cameraId = cameraId.slice(0, -extMatch[0].length).replace(/-\d+$/, '');
    }
    return { mode: 'live', orgId: parts[0], cameraId };
  }
  return { mode: 'unknown' };
}
```

```typescript
/** Find the SRS client_id currently publishing the given stream path. */
async findPublisherClientId(streamPath: string): Promise<string | null> {
  // streamPath e.g. "push/abc123..." or "live/{orgId}/{cameraId}"
  try {
    const res = await fetch(`${this.baseUrl}/api/v1/clients`);
    const result = await res.json();
    const clients: any[] = result?.clients || [];
    // Match on url field which includes the app/stream path.
    const match = clients.find((c: any) => {
      const url = c.url ?? '';
      return url.endsWith(`/${streamPath}`) && c.type === 'fmle-publish'; // or 'publish'
    });
    return match?.id ?? null;
  } catch (err) {
    this.logger.warn(`findPublisherClientId failed: ${(err as Error).message}`);
    return null;
  }
}

/** Kick a client by id via SRS /api/v1/clients/{id} DELETE. */
async kickPublisher(clientId: string): Promise<void> {
  const res = await fetch(`${this.baseUrl}/api/v1/clients/${clientId}`, {
    method: 'DELETE',
  });
  if (!res.ok) {
    throw new Error(`SRS kick failed: ${res.status} ${res.statusText}`);
  }
  this.logger.log(`Kicked SRS client ${clientId}`);
}
```

## Open Questions

1. **Should `on_publish` for unknown stream keys (404 case) log an audit event under `orgId: 'system'` or skip audit entirely?**
   - What we know: D-21 requires `camera.push.publish_rejected` with `clientIp`. The audit table has `orgId` NOT NULL.
   - What's unclear: Does the existing audit schema accept a 'system' sentinel orgId, or will FK constraint reject it?
   - Recommendation: Check existing audit entries for non-org-scoped events (e.g., Phase 14 FIX-01 super-admin). If 'system' org exists, use it. Otherwise, add a nullable `orgId` to audit table (out of scope — use a fallback orgId of the system organization row or skip the audit for unknown-key rejections).

2. **Should the `on-forward` hook verify the streamKey AGAIN, or trust that `on_publish` already did?**
   - What we know: on_publish fires first for a given publish; on_forward fires right after.
   - What's unclear: Is there a race where on_forward arrives before on_publish completes?
   - Recommendation: SRS docs strongly imply on_publish→on_forward sequence. Recommend verifying once in on_publish, then on_forward trusts the key (re-querying is idempotent but wasteful). Add a test that simulates the two callbacks in rapid succession to validate.

3. **Which existing Stream Profile is the "system transcode profile" that D-16 Accept flips to?**
   - What we know: `StreamProfile` table has `isDefault: Boolean` and `codec: 'auto'` as defaults.
   - What's unclear: Every org has its own StreamProfile list. Is there a cross-org shared "always transcode" profile?
   - Recommendation: **Option A (preferred):** Reuse the org's `isDefault: true` profile IF its codec ≠ 'copy'. **Option B:** Create (or find) a profile named `'default-auto-transcode'` per-org during onboarding. **Option C:** Add a new `type: 'auto-transcode'` field. Planner decides; A is the lightest.

| V3 Session Management | no | Sessions are handled by Better Auth (existing); push stream key is not a session. |

### Known Threat Patterns for SRS push ingest + stream-key auth

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Stream-key brute force | Spoofing | 128-bit entropy (nanoid 21) makes brute force infeasible. Rate-limiting on the SRS callback path is extra defense (Throttler already wired on API but SRS callbacks use `@SkipThrottle()` — may need revisit). Audit event `camera.push.publish_rejected` surfaces the pattern. |
| Key leak via log | Information disclosure | Audit entries log only 4-char prefix (D-21). Key never appears in error messages. Owner UI surfaces full URL; all other surfaces mask. |
| Replay with captured key after rotation | Spoofing | Rotation immediately kicks active publisher via `/api/v1/clients/{id} DELETE` (D-20). New publish attempts with old key → `on_publish` 403 (D-15). No grace period (D-20 rejected). |
| Denial of service via unauthorized publish spam | Denial of service | SRS rejects second concurrent publisher on same path (D-24). Unknown keys hit `on_publish` → DB query + 403 → no stream resources allocated. If abuse rate is high, consider Phase 19+ additional rate-limit at the SRS `http_hooks` path via nginx. Out of scope for this phase. |
| Cross-org stream-key collision | Privilege escalation | `@@unique([streamKey])` global constraint prevents identical keys across orgs. DB-level guarantee. |
| Info disclosure in codec-mismatch banner | Information disclosure | Banner shows only the **detected codec string** (e.g., "H.265"), not the full stream metadata or RTMP tcUrl. Already a normalized display value from `SrsApiService.getStream`. |
| Tenancy bypass via on_publish callback | Elevation | SRS callback runs without CLS context — `enqueueProbeFromSrs` already uses `systemPrisma` (RLS-bypass) per Phase 19 commit. Any NEW callback-context DB read must follow the same pattern. **Do not use tenancy client in callback handlers.** |

### Specific hardening recommendations

- **Rate-limit the `on_publish` endpoint** on unknown-key 403 responses. Currently `@SkipThrottle()` applies — consider a per-IP throttle on rejected publishes. Not strictly in CONTEXT scope; flag for discussion if brute-force events become visible in audit.
- **Timing attack on key lookup:** `findFirst({ streamKey })` is a DB equality lookup. PostgreSQL B-tree index lookup time is O(log N) but doesn't vary with key content — timing attack is not viable. No special mitigation needed.
- **Stream key in URL bar / browser history:** The reveal dialog shows the full URL in an `<input readOnly>`. Users copying to OBS don't see it in a URL bar. Acceptable risk.
