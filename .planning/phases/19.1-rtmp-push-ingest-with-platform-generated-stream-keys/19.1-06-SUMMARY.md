# Phase 19.1 Plan 06: Wave 4 Frontend Dialogs Summary

Wave 4 frontend A â€” extend camera-form-dialog and bulk-import-dialog with push-mode UI. Consumes the IngestModeToggle + CreatedUrlReveal composites from Plan 05, wires the new POST /cameras (push branch) + POST /cameras/bulk-import response shape from Plan 03 into the UI, and introduces a new 'result' step in bulk-import for one-time push-URL CSV download. 17 new tests across 2 `.spec.tsx` files; full web suite 306 passed / 1 skipped / 0 failed across 47 files.

## What was built

### camera-form-dialog push mode (Task 1, commit `7076b1b`)

- **State additions**: `ingestMode: 'pull' | 'push'` (default `'pull'`), `phase: 'form' | 'reveal'` (default `'form'`), `createdUrl: string`. Hidden when editing (per D-01 / UI-SPEC).
- **Two-phase render** â€” `DialogContent` swaps body based on `phase`:
  - `phase === 'form'`: standard form + `<IngestModeToggle>` above Name (create-mode only). When `ingestMode === 'push'`, the streamUrl `<Input>` is replaced by a hint block with UI-SPEC verbatim copy:
    - Heading: `We'll generate a push URL after you save.`
    - Body: `Configure your camera or encoder to publish to the generated URL. H.264 video + AAC audio are recommended for zero-transcode delivery.`
    - Link: `Setup guide â†’` (right-aligned `text-xs text-muted-foreground`, opens `/docs/push-setup` in new tab)
  - `phase === 'reveal'`: renders `<CreatedUrlReveal url={createdUrl} title="Camera created" onClose={handleRevealDone} />`. Done â†’ `onSuccess()` â†’ `onOpenChange(false)` â†’ `resetForm()`.
- **Submit branches**:
  - Edit mode: PATCH unchanged (streamUrl + site fields, no ingestMode).
  - Push create: POST with `{ name, description?, tags?, location?, streamProfileId, ingestMode: 'push' }` â€” **no streamUrl**. On 2xx `response.streamUrl`, flip to reveal phase (do NOT call onSuccess/onOpenChange yet).
  - Pull create: POST with `{ ...pullFields, ingestMode: 'pull' }`. Close dialog on success.
- **Error branch**: `ApiError(409)` now distinguishes `DUPLICATE_STREAM_URL` (existing copy) from `DUPLICATE_STREAM_KEY` (new copy "A camera with this push key already exists. Please try saving again.").
- **Mode switch** (`handleIngestModeChange`) clears `streamUrl` + error on either direction â€” prevents stale state from leaking between modes.
- **Submit gating** (`canSubmit`): push mode requires only `name` (+ siteId for create); pull keeps Phase 19 validation (name + streamUrl + streamUrlError === null + siteId).

### bulk-import-dialog push extensions (Task 2, commit `c22cac1`)

- **Type**: `CameraRow.ingestMode?: 'pull' | 'push'` (optional â€” backward compat for pre-19.1 CSVs).
- **Parsing**:
  - `mapHeaders` recognizes `ingestMode` column via `normalizeHeader` (strips `[_\-\s]` + lowercase) with alias `mode`. `ingest_mode` is handled by the normalization.
  - `rowFromValues` + `parseJSON` populate the field; any value other than `'push'` (case-insensitive) falls back to `'pull'`.
- **validateRow** â€” per-mode branch:
  - Push: `streamUrl` non-empty â†’ error `Push rows must leave streamUrl empty â€” a URL will be generated.` (UI-SPEC verbatim); empty â†’ OK.
  - Pull: Phase 19 behavior preserved (empty â†’ "Stream URL is required"; invalid prefix â†’ "Must be rtsp://, rtmps://, rtmp://, or srt://"; invalid URL â†’ "Invalid URL â€” check host and path").
- **annotateDuplicates** â€” push rows short-circuit to `{ duplicate: false, duplicateReason: undefined }` before any map lookup. Pull-row dedup continues as before.
- **Sample CSV** includes `ingestMode` column with one `push` row example (`push-cam-1,,push,Encoder feed (URL generated on save),indoor,,`). Upload hint updated from `Columns: name, streamUrl, description, â€¦` to `Columns: name, streamUrl, ingestMode, description, â€¦`.
- **handleImport payload** omits `streamUrl` for push rows, always sends `ingestMode`. Response response shape extended with `cameras?: ImportedCamera[]` (captured in new `importedCameras` state).
- **Toast cascade** (priority order matters â€” push/mixed cases are checked first):
  - All-push (`pushCount > 0 && pullCount === 0`): `Imported {N} push cameras. Download URLs to configure your encoders.`
  - Mixed (`pushCount > 0 && pullCount > 0`): `Imported {N} cameras ({P} push, {Q} pull). {R} skipped as duplicates.`
  - Pull-only existing branches (success / partial / all-duplicate / failed) â€” unchanged.
- **New 'result' step** â€” only entered when `pushCount > 0`. Keeps dialog open so user can click `<PushUrlsDownloadButton>`. Pull-only imports close immediately as before.
- **PushUrlsDownloadButton** (new inline component):
  - Label `Download push URLs (CSV)` â†’ `Downloaded` (disabled) for 3s post-click.
  - Builds CSV client-side: `name,streamUrl` header + one row per push camera with RFC 4180 double-quoted escaping.
  - Triggers download via `URL.createObjectURL(blob)` + anchor click + `URL.revokeObjectURL`.
  - Tooltip: `One-time download â€” view individual URLs later from each camera.`

## Deviations from Plan

### [Rule 1 â€” Bug] base-ui Tooltip renders duplicate role=button for PushUrlsDownloadButton

- **Found during:** Task 2 GREEN step (tests `renders Download push URLs (CSV) button` + `Download button click` failing with `Found multiple elements with the role "button" and name /Download push URLs \(CSV\)/i`).
- **Issue:** Plan sample used `screen.findByRole('button', { name: /Download push URLs \(CSV\)/i })`. base-ui's `Tooltip` implementation wraps its trigger in a focus-guard `<span role="button">` sibling, so `getByRole('button', { name })` returns two matches and throws. Identical pattern is documented in Plan 05's deviation log for the `CreatedUrlReveal` composite where `Copy URL`/`Copied` button encountered the same duplicate-role issue (different fix there â€” `aria-live` worked around a different symptom).
- **Fix:** Narrow test queries to `screen.getAllByText(/Download push URLs \(CSV\)/i)` to match either element by text, and for the click test use `.closest('button')` + array `.find()` to pick the real `<button>` element. The component itself is unchanged â€” only test queries adjusted.
- **Files modified:** `apps/web/src/app/admin/cameras/components/__tests__/bulk-import-dialog-push.spec.tsx`
- **Commit:** `c22cac1`

### [Rule 2 â€” Missing critical functionality] DuplicateStreamKey error branch

- **Found during:** Task 1 implementation (reading Plan 03 SUMMARY threat register T-19.1-SK-COLLISION).
- **Issue:** Plan 06 sample code only handled `DUPLICATE_STREAM_URL`. Plan 03 ships a 409/DUPLICATE_STREAM_KEY response on the ~2^-64 nanoid collision path, but the form-dialog error branch would fall through to the generic "Failed to create camera" message â€” misleading since retry resolves it.
- **Fix:** Added a dedicated `err.code === 'DUPLICATE_STREAM_KEY'` branch surfacing "A camera with this push key already exists. Please try saving again." This mirrors the UI-SPEC's defensive-case copy table entry for 409 on push camera creation.
- **Files modified:** `apps/web/src/app/admin/cameras/components/camera-form-dialog.tsx`
- **Commit:** `7076b1b`

### [Rule 2 â€” Missing critical functionality] bulk-import 'result' step to keep dialog open

- **Found during:** Task 2 implementation (reviewing UI-SPEC Â§"Bulk-import result panel extension (D-14)").
- **Issue:** Plan sample assumed the existing post-import flow (immediately `onOpenChange(false)` + close) would still apply. But the D-14 contract requires the user to SEE the `PushUrlsDownloadButton` in the result panel before the dialog closes â€” closing immediately gives the user no time to click Download.
- **Fix:** Introduced a third step `'result'` (in addition to `'upload'` / `'preview'`). Pull-only imports still auto-close on success (Phase 19 UX unchanged â€” blast radius scoped). Push imports stop at the result step with a Done button; `onSuccess()` still fires so the parent table refreshes in the background while the dialog is still open.
- **Files modified:** `apps/web/src/app/admin/cameras/components/bulk-import-dialog.tsx`
- **Commit:** `c22cac1`
