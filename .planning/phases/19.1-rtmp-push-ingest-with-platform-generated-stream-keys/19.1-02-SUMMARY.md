# Phase 19.1 Plan 02: Wave 2 Backend A — SRS Callback Push Branch Summary

Land the SRS callback layer for RTMP push ingest: `on_publish` branches on `app='push'` (DB-resolved stream key → transition + probe + first-publish audit, or `{code:403}` + rejected audit), a new `on_forward` endpoint implements the push→live passthrough remap with a recursion guard, and `config/srs.conf` gets the `forward { backend ... }` directive pointing at it. 11 scaffolded `it.todo` tests flip to real passing tests; build stays green; 3 `TODO(Plan 19.1-03)` stubs bridge compile until Plan 03 fills them in.

## What was built

### Controller (Task 1)

`apps/api/src/srs/srs-callback.controller.ts`:

- **`parseStreamKey`** rewritten as a discriminated union return:
  - `{ mode: 'push', streamKey }` — app=`'push'`, stream IS the canonical key. NO extension-strip (the `.m3u8 | .ts | .m4s | .mp4 | .flv` regex is gated inside the live branch so push keys like `abc.m3u8` pass through literally).
  - `{ mode: 'live', orgId?, cameraId? }` — preserves existing parsing exactly (live/`{orgId}`/`{cameraId}` incl. extension-strip + `-{seq}` suffix trim).
- **`onPublish` push branch** (new):
  1. `findByStreamKey(streamKey)` on tenancy-bypass CamerasService helper (Plan 03 owns the real lookup).
  2. If not found → `auditService.log({ orgId: 'system', action: 'camera.push.publish_rejected', details: { streamKeyPrefix: first4, reason: 'unknown_key' }, ip })` + `return { code: 403 }`. Full key never logged.
  3. If found → `statusService.transition(cameraId, orgId, 'online')` (maintenanceMode does NOT short-circuit — StatusService downstream gate handles suppression per D-23).
  4. `enqueueProbeFromSrs(..., { delay: 1000 })` so SRS `/api/v1/streams` populates before the worker fetches.
  5. `markFirstPublishIfNeeded(cameraId, orgId, { clientIp })` → if this was the first time, emit `camera.push.first_publish` audit tagged with the camera's orgId.
  6. Return `{ code: 0 }`.
- **`onForward` endpoint** (new — D-18):
  - Parses body via `OnForwardSchema` (Plan 01). Invalid → `{ code: 0, data: { urls: [] } }`.
  - Recursion guard: `if (app !== 'push') return empty urls` — prevents the SRS live→live loop documented in RESEARCH Pitfall 3. `resolveForwardTarget` is NOT called in this path (test-asserted).
  - `resolveForwardTarget(streamKey)` → null (unknown) or transcode=true → empty urls.
  - Passthrough: `rtmp://${SRS_HOST ?? '127.0.0.1'}:1935/live/${orgId}/${cameraId}` returned as the single URL.
- **Live branch preserved** — existing `transition + enqueueProbeFromSrs(delay:1000)` flow unchanged. All other handlers (`on_unpublish` / `on_play` / `on_stop` / `on_hls`) updated to branch on `parsed.mode` before consuming `orgId/cameraId`; push-mode on those events is a `{ code: 0 }` debug-log no-op (HLS playback still flows through the forwarded live path).

- **Commit:** none (environment setup, not code).

### [Rule 1 — Type Safety] AuditService constructor param made optional

- **Found during:** Task 1 — reconciling the new constructor signature with existing 3-arg / 5-arg test instantiations.
- **Issue:** Plan specified a required `auditService: AuditService` 6th arg. Three existing tests construct the controller with fewer positional args (`callbacks.test.ts`: 3, `on-play-verification.test.ts`: 3, `srs-callback.test.ts`: 5). Making the 6th arg required would force unrelated test edits across the repo.
- **Fix:** Declared the parameter as `auditService?: AuditService` (optional). All push-branch audit calls use `this.auditService?.log(...)` so the undefined-path is safe at runtime. In production DI, Nest still binds the real AuditService (AuditModule import wires it). The 6-arg test file I wrote passes an explicit `{ log: vi.fn() }` mock so the `.log` assertions still run.
- **Impact:** Zero — all acceptance criteria still pass (`auditService.log` call assertions still work in the new Plan 02 tests). No regression in the existing SRS tests (53 passed).
- **Commit:** `088c8c7`.
