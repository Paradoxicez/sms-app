# Phase 21.1: Active-job collision fix for hot-reload restart - Context

**Gathered:** 2026-04-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Make profile-edit PATCH actually trigger FFmpeg restart on cameras whose BullMQ `start` job is currently in active+locked state ‚Äî the common state after boot recovery. Phase 21 left the surface area correct (audit log, response fields, UI toasts, 409 protection all working) but the runtime restart cycle silently no-ops in this case because `enqueueProfileRestart`'s remove-then-add pattern fails on locked active jobs.

Post-fix expected behavior: profile edit on a running camera produces an FFmpeg PID change within ~5 seconds (gracefulRestart grace + signal latency).

**In scope:**
- Add Redis pub/sub signal mechanism between `enqueueProfileRestart` (publisher) and `StreamProcessor` (subscriber per active job)
- Wire 4 mitigations against signal-mode-specific edge cases (lifecycle, retry race, concurrent dedup, test coverage)
- Hybrid test layer (mock unit + real Redis integration)

**Out of scope (preserved from Phase 21):**
- D-03 jobId pattern `camera:{id}:ffmpeg`
- D-04 dedup by jobId for wait/queued state
- D-05 `gracefulRestart` helper (reused as-is)
- D-07 audit-at-enqueue ordering
- B-1 collision guard in `camera-health.service.ts`
- DELETE 409 protection, UI toasts, all surface-level Phase 21 contracts

**Out of scope (deferred):**
- Apply pub/sub fix to `startStream` / `stopStream` / status notify queue ‚Äî those have correct behavior for their intent (see `<deferred>`).
- Admin endpoint for targeted force-restart ‚Äî current PATCH mechanism + standard deploy restart cover the operational need.

</domain>

<decisions>
## Implementation Decisions

### D-12: Fix approach ‚Äî Pub/sub signal

Use Redis pub/sub channel `camera:{cameraId}:restart` per camera as the signaling mechanism between `enqueueProfileRestart` (publisher) and `StreamProcessor` (subscriber per active job).

**Mechanism:**
- `enqueueProfileRestart` checks `existingJob.isActive()`. If active ‚Üí publish signal to `camera:{cameraId}:restart` channel with `{ profile: newProfile }` payload. If wait/queued/none ‚Üí fall through to existing remove-then-add path (which works for those states).
- `StreamProcessor.process()` opens a Redis subscriber via `ioredis.duplicate()` at job start, subscribes to its camera's channel, calls `gracefulRestart` on signal, then lets the worker's natural error handling re-spawn FFmpeg with new profile data on the next attempt.

**Rejected alternatives (recorded for `<deferred>`):**
- B. `job.update()` + external SIGTERM ‚Äî `StreamProcessor.process()` destructures `job.data.profile` once at function entry; updating data via `Job.updateData()` would not flow into the running worker without a refactor. Indirect fail-and-retry semantics make code harder to read.
- C. Split jobIds (`:ffmpeg:start` vs `:ffmpeg:restart`) ‚Äî would require redesigning B-1 collision guard from Plan 04 and add risk of two FFmpeg processes for one camera running concurrently.
- D. Cooperative shutdown flag with worker polling ‚Äî adds permanent polling overhead per online camera and worse latency than pub/sub.

### D-12 mitigations (bundled into the fix)

The `<deferred>` issue analysis identified 4 risks specific to pub/sub. All 4 mitigations are part of this fix, not optional:

- **Mitigation 1 ‚Äî Subscriber lifecycle:** `StreamProcessor.process()` wraps the worker body in `try { ...subscribe } finally { sub.quit() }`. NestJS `OnModuleDestroy` hook reaps any leaked subscribers via existing Phase 15 ResilienceService graceful shutdown.
- **Mitigation 2 ‚Äî Worker retry race safety net:** On subscriber `'subscribe'` event ready, the processor reads the current profile from Prisma and compares fingerprint vs `job.data.profile`. If different ‚Üí call `gracefulRestart` immediately. Catches PATCHes that landed during the worker death+retry window.
- **Mitigation 3 ‚Äî Concurrent PATCH dedup:** `StreamProcessor` keeps `private restartingCameras = new Set<string>()`. Signal handler is a no-op if the cameraId is already in the set; adds on entry, removes on completion.
- **Mitigation 4 ‚Äî Test coverage:** integration test layer with real Redis (see D-14).

### D-13: Fix scope ‚Äî Strict, only `enqueueProfileRestart`

Apply the pub/sub fix only to `streams.service.ts:enqueueProfileRestart` (lines 206-210). The other 3 remove-then-add sites in the codebase are NOT defective:

- `streams.service.ts:100-110` (`startStream`) ‚Äî intent is "start a stopped camera". If a job is already active, dedup-as-no-op is correct behavior.
- `streams.service.ts:281-287` (`stopStream`) ‚Äî intent is "stop the camera". Has explicit log + transition('offline') fallback when `remove()` fails. Worker reaps naturally.
- `status.service.ts:87-91` (notify debounce) ‚Äî different queue (notifyQueue), short-lived debounced jobs, irrelevant to FFmpeg lifecycle.

No helper extraction (`signalActiveOrEnqueue()`) ‚Äî premature abstraction with no second caller.

### D-14: Test strategy ‚Äî Hybrid

Two test layers required. Pure mock alone caused Phase 21 to miss this defect.

**Unit (mock-based, fast feedback):**
- `ioredis-mock` or `vi.mock('ioredis')` for the pub/sub primitive
- BullMQ mock as in Phase 21 patterns
- Coverage: subscribe wiring, signal handler, mutex dedup, fingerprint safety-net path, finally-block cleanup
- Existing Phase 21 test patterns (e.g., `profile-restart-dedup.test.ts`) extend naturally

**Integration (real Redis, ~3-5 tests):**
- Spin up a real BullMQ worker against `sms-app-redis-1` test DB index
- Spawn a stub long-running "FFmpeg" (sleep loop) so the job stays in active+locked state
- Publish a signal via the production `enqueueProfileRestart` code path
- Assert: stub process killed, new process spawned with new profile data, full cycle <5s
- Reuse Plan 04's `ffmpeg-graceful-restart.test.ts` patterns for spawn-stub timing

Integration tests document the exact failure mode that Phase 21 unit tests missed.

### D-15: Rollout ‚Äî Standard restart

No special rollout procedure. Docker Compose deploy already triggers the conditions Phase 21.1 needs:

1. `docker compose restart api` (or rebuild + `pnpm start:prod`) sends SIGTERM to node process
2. Phase 15 `ResilienceService.onApplicationShutdown` (10s grace) kills all FFmpeg children
3. Process exits ‚Üí all BullMQ active+locked jobs are released back to wait state
4. New process boots, runs Phase 15 BootRecoveryService ‚Üí re-enqueues every camera
5. New workers process jobs with the new code (subscriber attached)

This means D-12 Mitigation 2 (fingerprint safety net on subscribe) covers the narrow case where a PATCH lands between the OLD worker dying and the NEW worker subscribing ‚Äî so deploy-time PATCHes are still honored.

Plan 21.1 will document this expected deploy behavior in the SUMMARY but adds NO ops procedure, NO admin endpoint, NO maintenance-window requirement. The defect fix and the deploy mechanism compose naturally.

### Claude's Discretion

- Exact Redis channel name format (suggested `camera:{cameraId}:restart` but planner can refine)
- Subscriber registration hook (process() body vs OnModuleInit + per-camera dispatch ‚Äî planner picks based on existing StreamProcessor structure)
- Test file naming and location (extend existing `tests/streams/` patterns)
- Exact log/telemetry strings for the new code path

## Existing Code Insights

### Reusable Assets

- **`FfmpegService.gracefulRestart(cameraId, graceMs)`** (Plan 04, `ffmpeg.service.ts`) ‚Äî already does SIGTERM ‚Üí 5s poll ‚Üí SIGKILL. Subscriber callback calls this directly; no new kill logic needed.
- **`ioredis` client + `.duplicate()` API** ‚Äî already a project dependency; `duplicate()` produces a fresh connection suitable for blocking subscribe operations without starving the main client.
- **NestJS `OnModuleDestroy` hook** ‚Äî already used by Phase 15 ResilienceService for graceful shutdown; processor cleanup hooks into the same pattern.
- **`AuditService.log({ action: 'camera.profile_hot_reload', ... })`** ‚Äî Phase 21 D-07 audit row writer; behavior unchanged.
- **`profile-fingerprint.util.ts:fingerprintProfile()`** (Plan 02) ‚Äî used by Mitigation 2 safety net to compare DB profile vs `job.data.profile`.
- **Plan 04's `ffmpeg-graceful-restart.test.ts`** ‚Äî spawn-stub + timing-assertion pattern to extend for new integration tests.

### Established Patterns

- **BullMQ jobId scheme `camera:{id}:ffmpeg`** ‚Äî single jobId per camera regardless of job name (`start` vs `restart`). Phase 21.1 preserves this; signal carries the variation that `add()` would have carried.
- **Audit-at-enqueue (D-07)** ‚Äî write audit row BEFORE the queue operation so intent is captured even if the runtime path fails. Pub/sub fix continues this: signal published AFTER audit row exists.
- **Worker capture of profile at job start** ‚Äî `StreamProcessor.process()` destructures `job.data.profile` once at function entry. Phase 21.1 does NOT change this; pub/sub signal injects fresh profile via the signal payload, and gracefulRestart + retry brings worker into the new code path.
- **`ResilienceService.onApplicationShutdown`** (Phase 15) ‚Äî 10s grace, kills FFmpeg children. D-15 depends on this for clean rollout.

### Integration Points

- **Publisher:** `streams.service.ts:enqueueProfileRestart` (existing method, modified at lines 206-210 ‚Äî branch on `existingJob.isActive()`).
- **Subscriber:** `stream.processor.ts:process()` (existing method, wrapped with try/finally + subscribe block).
- **Module wiring:** `streams.module.ts` ‚Äî inject Redis client into `StreamProcessor` (already imported transitively via BullMQ).

<specifics>
## Specific Ideas

- **`gracefulRestart` is mostly idempotent** ‚Äî it reads `runningProcesses.get(cameraId)` first; second call after kill is a no-op. Mitigation 3 (in-process Set) is belt + suspenders, not load-bearing.
- **Test reproducer:** the BKR06 + SD640 + 11 PATCHes scenario from 21-VALIDATION.md is the canonical regression test target. After Phase 21.1, repeating that exact sequence MUST result in FFmpeg PID changes matching the audit row count.
- **D-15 implies no migration code** ‚Äî boot recovery already re-enqueues every running camera on startup, so the deploy itself is the migration.

</specifics>
