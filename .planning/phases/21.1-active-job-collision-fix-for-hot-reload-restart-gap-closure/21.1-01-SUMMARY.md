## Decisions Made

- **Reuse `REDIS_CLIENT` symbol from `api-keys.service.ts`** rather than introducing a new injection token or factoring a shared `RedisModule`. Three modules already use this exact pattern (api-keys, dashboard, account). D-13 forbids out-of-scope refactors.
- **Channel name `camera:{cameraId}:restart`** — per-camera channel, mirrors the `camera:{id}:ffmpeg` jobId scheme so the subscriber (Plan 02) can derive its channel name without a separate registry.
- **Payload includes both `profile` and `inputUrl` plus `fingerprint`** — Plan 02's subscriber Mitigation 2 (fingerprint safety net) compares the published fingerprint against the DB profile fingerprint to catch races during worker death+retry windows. `inputUrl` is included so the subscriber doesn't need to re-derive it from `cam.ingestMode + cam.streamKey`.
- **`continue` rather than nested `if/else`** — keeps the active-path's "skip queue.add entirely" semantics explicit and visually obvious. The defect being fixed is precisely that `queue.add()` on a locked-active jobId is silently a no-op; `continue` makes the skip-on-purpose clear to future readers.
- **Reordered profile/inputUrl computation above the branch** — required because the active-path needs them in the publish payload. Audit-at-enqueue (D-07) is still preserved because the audit row is still written first; only the queue lookup moved down.
- **`typeof existingJob.isActive === 'function'` mock-compat guard** — preserves all existing Phase 21 unit tests without modification. Plan 03 will add new mocks that DO implement `isActive()`.
