## Subscriber Lifecycle (exact sequence)

1. `process(job)` enters → defensive guards (empty-data, push+passthrough) run BEFORE any subscriber logic.
2. If guards pass and `this.redis` is defined: `subscriber = this.redis.duplicate()` (a fresh ioredis connection sharing config but independent of the main client; blocking subscribe will not starve other Redis usage).
3. Channel name `camera:{cameraId}:restart` is computed (template literal, byte-identical to Plan 01's publisher format).
4. `subscriber.on('message', ...)` handler is wired (deferred — fires only when publisher signals).
5. `subscriber.subscribe(channel, callback)` is called inside an `await new Promise<void>((resolve) => { ... })` wrapper so subscribe-ready is awaited before spawn (the safety net resolves inside the success path of the callback). On subscribe-ready: `runFingerprintSafetyNet` is fired-and-forgotten via `.then(resolve)`, so the spawn path does not block on its completion.
6. The Phase 21 `if (job.name === 'restart') { ... } else { ... }` branch + `await startStream(...)` runs inside the new `try` block.
7. Whether `startStream` resolves naturally OR rejects from a signal-driven kill, the `finally` runs `await subscriber.unsubscribe().catch(() => {})` then `await subscriber.quit().catch(() => {})`. Both errors are swallowed because the subscriber is being torn down anyway — Redis transient errors during quit are not actionable.

## Fingerprint Safety Net (exact role)

The pub/sub mechanism has a fundamental signal-loss window: a PATCH that arrives between OLD-worker-death (e.g. SIGTERM during deploy) and NEW-worker-subscribe (after BullMQ retry + new `process()` invocation) would publish to a channel with no subscriber and the message is silently dropped. The fingerprint safety net closes this window:

1. NEW worker boots (post-deploy or post-FFmpeg-error retry).
2. NEW worker calls `subscriber.subscribe(channel, callback)`. The callback fires when subscription is established (Redis ack received).
3. Inside the callback: `runFingerprintSafetyNet` reads `Camera + StreamProfile` via `SystemPrismaService.camera.findUnique({ where: { id: cameraId }, include: { streamProfile: true } })`.
4. Builds `dbProfileInput` = the 7 FFmpeg-affecting fields from the DB (`codec`, `preset`, `resolution`, `fps`, `videoBitrate`, `audioCodec`, `audioBitrate`).
5. Computes `dbFingerprint = fingerprintProfile(dbProfileInput)` and `jobFingerprint = fingerprintProfile(jobProfile as any)`.
6. If `dbFingerprint !== jobFingerprint`: a PATCH was missed during the death-and-retry window → call `gracefulRestart(cameraId, 5_000)` immediately. The kill rejects this attempt's `startStream`, BullMQ retries with the next attempt, and on the next attempt's subscribe-ready the fingerprints match (because the missed PATCH already updated the DB row that the next attempt reads via Phase 15 BootRecoveryService re-enqueue path).

D-15 documents this as the deploy-time scenario: "standard restart triggers BootRecoveryService re-enqueue, ResilienceService graceful FFmpeg shutdown, and Mitigation 2 covers the deploy-window-PATCH case."

## restartingCameras Set Lifecycle

- **Field:** `private restartingCameras = new Set<string>()` on `StreamProcessor` class (process-local; not Redis-backed because two API processes won't share one camera's BullMQ active+locked state — BullMQ jobIds enforce single-worker ownership).
- **Signal-handler entry:** `if (this.restartingCameras.has(cameraId)) { logger.debug('...ignored — already restarting'); return; }` — immediate no-op.
- **Signal-handler add:** `this.restartingCameras.add(cameraId);` then a `try { ... await gracefulRestart ... } finally { this.restartingCameras.delete(cameraId); }`.
- **Safety-net check:** identical pattern: `if (!this.restartingCameras.has(cameraId)) { this.restartingCameras.add(cameraId); try { ... } finally { this.restartingCameras.delete(cameraId); } }` — the safety net additionally guards "is the signal handler currently restarting?" before joining the restart in-flight.

The Set deletes happen in `finally` blocks so an exception inside `gracefulRestart` (which itself never throws per the FfmpegService contract — but defensive) cannot leave a stale `cameraId` in the set, which would cause subsequent legitimate signals to be permanently no-op'd.

## Why `let { inputUrl, profile } = job.data` (instead of `const`)

The signal handler receives a JSON payload `{ profile, inputUrl, needsTranscode, fingerprint }` from the publisher (Plan 01). When the handler updates `profile` and `inputUrl` from the payload, those values must flow into the eventual `await this.ffmpegService.startStream(cameraId, inputUrl, outputUrl, profile, needsTranscode)` at the bottom of `process()`. With `const` destructure, TypeScript binds the names immutably and the assignment fails at compile time. With `let`, the local variables are reassigned and the post-retry spawn picks up the new values.

This is also why the safety net's `gracefulRestart` is sufficient on its own: even if the local variable update was missed, the kill rejects this attempt's `startStream`, BullMQ retries with `job.data` re-read from BullMQ storage, and the next attempt's subscribe-ready safety net sees matching fingerprints (because the publisher writes audit row → publishes → and the next attempt's DB read sees the latest state).
