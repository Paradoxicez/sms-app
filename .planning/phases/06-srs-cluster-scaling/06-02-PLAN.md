2. Create ClusterHealthService (`apps/api/src/cluster/cluster-health.service.ts`):
- Inject PrismaService (direct, not tenancy), SrsApiService, ClusterGateway, Logger
- `async checkNode(nodeId: string): Promise<void>`:
  a. Load node from DB
  b. If node.role === ORIGIN:
     - Try `srsApiService.getSummaries(node.apiUrl)`
     - On success: extract `data.self.cpu_percent` as cpu (%), `data.self.mem_percent` as memory (%), `data.self.srs_bytes_sent_total` as bandwidth, count from `srsApiService.getClients(node.apiUrl)` as viewers, `data.self.srs_uptime` as uptime
     - Try `srsApiService.getVersions(node.apiUrl)` for srsVersion
  c. If node.role === EDGE (nginx):
     - Try fetch `${node.hlsUrl}/health` with 5s timeout. Success = healthy.
     - Try fetch `${node.hlsUrl}/nginx_status` to parse active connections. Parse the nginx stub_status response: look for `Active connections: N` line, extract N as viewers.
     - cpu/memory not available from nginx -- leave null or 0
  d. On success:
     - Reset missedChecks to 0, set lastHealthAt to now()
     - If previous status was OFFLINE or DEGRADED, set status to ONLINE (auto-recovery per D-14)
     - Update health metrics (cpu, memory, bandwidth, viewers, uptime, srsVersion)
     - Broadcast node:health via gateway
     - If status changed, broadcast node:status via gateway
  e. On failure:
     - Increment missedChecks
     - If missedChecks >= 3: set status OFFLINE (per D-12)
     - Else if missedChecks >= 1: set status DEGRADED
     - If status changed, broadcast node:status via gateway
     - Broadcast node:health with null metrics

- `async startHealthChecks(): Promise<void>`: called from onModuleInit. For each node in DB, add a BullMQ repeatable job with key `health-${nodeId}`, repeat every 10000ms (10 seconds per D-12).
- `async addHealthCheck(nodeId: string)`: add repeatable job for a newly created node
- `async removeHealthCheck(nodeId: string)`: remove repeatable job when node is deleted

```typescript
async regenerateAndReloadSrs(): Promise<void> {
  // ... existing config generation and file write ...

  // Reload origin SRS
  try {
    await this.srsApiService.reloadConfig();
    this.logger.log('SRS origin configuration reloaded successfully');
  } catch (error) {
    this.logger.warn('Failed to reload SRS origin config', error);
  }

  // Reload all edge nodes (per D-07)
  try {
    const edges = await this.clusterService.getOnlineEdges();
    for (const edge of edges) {
      try {
        // For nginx edges: config changes affect origin only (nginx proxies to origin).
        // But nginx config itself may need reload if nginx.conf changed.
        // For now, log that edge config is served via API -- edge pulls on next health check.
        this.logger.log(`Edge node ${edge.name} will pick up config changes via origin`);
      } catch (err) {
        this.logger.warn(`Failed to notify edge ${edge.name}`, err);
      }
    }
    // Increment configVersion on all nodes to signal config change
    await this.clusterService.incrementConfigVersion();
  } catch (error) {
    this.logger.warn('Failed to propagate config to edges', error);
  }
}
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Backend -> Edge node health endpoint | Backend polls nginx /health and /nginx_status |
| Client -> Edge nginx -> Origin SRS | HLS traffic flows client to edge to origin, token validated at origin |
| PlaybackService -> ClusterService | Session routing depends on accurate health data |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-06 | Tampering | Health metrics manipulation | accept | Health endpoints on internal Docker network, not public. Edge nginx stub_status restricted to internal IPs. |
| T-06-07 | Denial of Service | All edges marked offline | mitigate | Fallback to origin when no edges online -- playback continues on origin |
| T-06-08 | Information Disclosure | nginx stub_status | mitigate | stub_status restricted to Docker network ranges (172.16.0.0/12, 10.0.0.0/8) via nginx allow/deny |
| T-06-09 | Repudiation | Health check results | accept | Health metrics logged via Logger, no audit trail needed for internal monitoring |
</threat_model>
