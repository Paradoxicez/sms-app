## Spacing Scale

Declared values (must be multiples of 4). Inherited from Phase 14, no additions:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline label↔helper spacing |
| sm | 8px | Form field internal spacing (label→input, helper→input), progress bar row gap |
| md | 16px | Default field stack, card content padding-x, section content row gap |
| lg | 24px | Card body padding, section header→body gap |
| xl | 32px | Between sections (Profile → Security → Plan & Usage) |
| 2xl | 48px | Page top/bottom padding |
| 3xl | 64px | Reserved — not used in this phase |

Exceptions:
- Avatar circular asset sits at 96px × 96px (multiple of 4 and sits outside the spacing-token system as a media dimension)
- Strength bar height: 4px (matches `xs` token)
- Progress bar height: 8px (matches `sm` token — larger than default 4px so threshold color changes are visible)

## Color

Inherited from Phase 14. No new base tokens:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `hsl(0 0% 100%)` / `hsl(0 0% 9%)` dark | Page background, section card surface |
| Secondary (30%) | `hsl(142 30% 97%)` / `hsl(142 20% 15%)` dark | Card header background, disabled input, progress bar track |
| Accent (10%) | `hsl(142 71% 45%)` | Primary CTA (`Save changes`, `Upload avatar`, `Change password`), focus ring, progress bar fill at safe threshold |
| Destructive | `hsl(0 84% 60%)` | `Remove avatar` button, progress fill at critical threshold, field error text |

Accent reserved for:
- Primary CTA buttons inside each section form (`Save changes`, `Change password`, `Upload avatar`)
- Focus ring on inputs / buttons
- Progress bar fill when usage `< 80%` (safe)
- `✓` feature-enabled icon in feature flags list

Never used for: plan name text, plain section headings, informational Contact admin message, avatar fallback background (keeps existing `bg-primary/10` for initials per `sidebar-footer.tsx`).

### Progress Bar Threshold Colors (new for this phase)

Usage percentage = `used / max * 100`. Thresholds apply to all 4 limit-capped metrics (cameras, viewers, bandwidth, storage). API calls progress bar is not rendered (no max per D-15).

| Range | Fill color | Label suffix color | Semantic |
|-------|-----------|--------------------|----------|
| 0–79% | `bg-primary` (accent green) | `text-muted-foreground` | Safe — plenty of headroom |
| 80–94% | `bg-amber-500` | `text-amber-600` | Warning — approaching limit |
| 95–100% | `bg-destructive` | `text-destructive` | At/over limit — contact admin |

Track color: `bg-muted` (secondary surface) in all states.

Rationale: 80% / 95% aligns with existing `MetricBar` component thresholds (70/90) used for Cluster Nodes (14-UI-SPEC §Column Specs). Plan & Usage uses 80/95 because customer-facing quotas should warn later and alarm later than ops-facing CPU/memory readouts.

### Password Strength Bar Colors (new for this phase)

zxcvbn produces scores 0-4. Per D-13, compressed to 3 levels:

| zxcvbn score | UI level | Bar segments filled (of 3) | Fill color | Label |
|--------------|----------|----------------------------|-----------|-------|
| 0, 1 | Weak | 1 | `bg-destructive` | "Weak" |
| 2, 3 | Medium | 2 | `bg-amber-500` | "Medium" |
| 4 | Strong | 3 | `bg-primary` | "Strong" |

Track for empty segments: `bg-muted`. Three equal-width segments with 4px gap between.

### Feature Flag Icons

| State | Icon | Color |
|-------|------|-------|
| Enabled | Lucide `Check` | `text-primary` (accent green) |
| Disabled | Lucide `X` | `text-muted-foreground` (no destructive — absence, not failure) |

│                                                │

```

### Avatar Component Spec

- Size: 96px × 96px (`Avatar` with custom class `size-24`; existing Avatar primitive supports arbitrary size via className override)
- Image source: `user.image` (Better Auth session → passed as prop)
- Fallback: initials from display name, matching existing `sidebar-footer.tsx` logic (`bg-primary/10 text-primary`, uppercase, max 2 chars)
- Upload affordance (D-09: no client crop):
  - Primary button "Upload new avatar" (variant: `default`) with Lucide `Upload` icon — triggers hidden `<input type="file" accept="image/jpeg,image/png,image/webp">`
  - Secondary button "Remove" (variant: `ghost`, `text-destructive`) with Lucide `Trash2` icon — only shown when `user.image` exists (D-10)
  - After file selected but before upload: show filename in mono 12px beneath buttons, and replace "Upload new avatar" with "Uploading..." + spinner during request
- Drag-and-drop is NOT required for v1 (D-09 says simple); buttons only is sufficient

### Display Name Form (D-06)

- Single field, `Input` primitive
- Initial value: `user.name` from Better Auth session
- Validation (client):
  - `required`: must not be empty or whitespace-only
  - `min length 2`, `max length 100`
- `Save changes` button disabled when field is empty, unchanged, or saving
- Loading state: button label becomes `Saving...`, spinner optional, button stays disabled
- Success: Sonner toast (see Copywriting § below), form becomes pristine (no longer dirty), session refresh triggers re-render

### Avatar Validation (D-08)

Client-side pre-flight before upload:
- Size > 2MB → error toast "Image too large. Maximum 2 MB." — do not send request
- MIME not in `[image/jpeg, image/png, image/webp]` → error toast "Unsupported format. Use JPEG, PNG, or WebP." — do not send request
- Server-side re-validation (canonical; backend returns 400 with body `{ error: "..." }` on failure)

Upload request: `POST /api/users/me/avatar` (multipart/form-data, field name `file`)
Remove request: `DELETE /api/users/me/avatar`

│  Label: Confirm new password                   │
│  [ Input type=password ....................]   │

│                                                │
│  [ Change password ]  ← right-aligned          │
└────────────────────────────────────────────────┘
```

### Password Strength Bar (D-13)

- Three equal-width horizontal segments, 4px tall, rounded-full, with 4px gaps
- As user types in "New password":
  - zxcvbn runs on each keystroke, debounced 150ms
  - Fills segments per table in Color section above
  - Label text to the right of the bar (`text-sm font-medium`): "Weak" / "Medium" / "Strong"
- When input is empty, all 3 segments show `bg-muted` and label shows `text-muted-foreground` "Enter a password"
- **Accessibility:** wrap the strength bar + label in a `<div aria-live="polite" aria-atomic="true">` so screen readers announce level changes without being chatty

### Validation

| Field | Rule | Error copy |
|-------|------|------------|
| Current password | Required | "Enter your current password." |
| New password | Required, min 8 chars (D-12) | "Password must be at least 8 characters." |
| New password | Must differ from Current | "New password must be different from your current password." |
| Confirm new password | Must match New password | "Passwords do not match." |

All errors render as `text-xs text-destructive` below the field with `mt-xs` (4px).

`Change password` button disabled until all three fields are non-empty and client validation passes.

### Submit Behavior (D-11, D-14)

- Calls `authClient.changePassword({ currentPassword, newPassword, revokeOtherSessions: true })`
- On 400 from server (wrong current password): show error toast, clear only the Current password field, keep focus on it
- On success:
  - Sonner toast (see Copywriting § below)
  - Clear all 3 password fields
  - Reset strength bar to empty state

### Progress Row Spec (for each of 4 capped metrics)

- Single row per metric, two-line:
  - Line 1: `Label` (left, 14/500) + spacer + `{used} / {max} {unit}` (right, 14/500, `tabular-nums`)
  - Line 2: Progress track 8px tall, full-width, rounded-full
- Percentage label ("32%") right-aligned inside or adjacent to bar, body/regular/muted-foreground when safe, amber or destructive at thresholds
- Use shadcn `Progress` primitive; wrap fill color in className that branches on computed threshold

### API Calls Row Spec (D-15: count only, no max)

- Single-line, Label left, `{count}` right in `tabular-nums`
- No bar
- Helper hint below in `text-xs text-muted-foreground`: "Month-to-date"

### Features List Spec

- Ordered: Recordings, Webhooks, Map view (stable order regardless of Package.features JSON key order)
- Each row: icon (4×4 w-4 h-4, with `mr-sm` 8px gap) + label (14/500)
- Enabled row: `text-foreground`; disabled row: `text-muted-foreground`

| Case | Behavior |
|------|----------|
| Org has no package assigned (null `packageId`) | Show section with: `<h3>No plan assigned` + `<p>Contact your administrator to assign a plan.` — suppress Usage and Features blocks |
| Endpoint returns error | Show compact inline error block: AlertTriangle icon + "Couldn't load plan details. Retry" (link triggers re-fetch). Retain Profile/Security unaffected |
| Values are zero (e.g. 0/25 cameras) | Render normally with `0%`, all-muted track, no special copy |

## Interaction Patterns

### Form Validation Strategy

- Use **react-hook-form + zod** (established pattern, 14-UI-SPEC §Component Inventory + admin camera form references)
- `mode: 'onBlur'` for field-level validation (error shows when user leaves field), with `reValidateMode: 'onChange'` (fixing a dirty-error field re-validates live)
- Password strength bar is decoupled from form validation — it's visual feedback only and never blocks submit (D-12: no HIBP, no mixed-case requirement)

### Keyboard Flow

Expected tab order on `/app/account` (tenant):

1. (Sidebar dropdown is behind focus — not on this page)
2. Display name input
3. Save changes button (Profile)
4. Upload new avatar button
5. Remove avatar button (if present)
6. Current password input
7. New password input
8. Confirm new password input
9. Change password button (Security)
10. Retry link (if Plan & Usage errored)

Enter key inside a field submits the enclosing section form. Section forms are independent `<form>` elements, so Enter in Profile only submits Profile.

### Focus Rings

All interactive elements use the accent focus ring already declared in `globals.css` (`:focus-visible` with `hsl(142 71% 45%)` at 2px, offset 2px). No per-element override.

### Dirty-State Tracking

- Profile form: `Save changes` button disabled when `name` unchanged AND avatar unchanged
- Security form: `Change password` button disabled until all 3 fields non-empty and validation passes
- Navigating away with unsaved Profile changes does NOT prompt (low stakes; form is persistent per session)

### ARIA

| Surface | ARIA |
|---------|------|
| Page heading | `<h1>` only one per page |
| Section cards | `<section aria-labelledby="profile-heading">` wrapping each card; H2 inside has matching id |
| Strength bar | Wrapper `role="status" aria-live="polite" aria-atomic="true"` |
| Progress bars | `<Progress>` primitive renders ARIA via base-ui automatically; include `aria-label="Cameras usage, 8 of 25"` |
| Field errors | `aria-describedby` on input → id of error `<p>`; error `<p>` has `role="alert"` only when appearing after a submit attempt (not on every keystroke) |
| Avatar upload button | Hidden `<input type="file">` + visible button pattern; button labelled "Upload new avatar", file input `aria-label="Choose image file"` |
