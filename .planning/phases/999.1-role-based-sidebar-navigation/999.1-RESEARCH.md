# Phase 999.1: Role-based Sidebar Navigation - Research

**Researched:** 2026-04-15
**Domain:** Next.js 15 App Router authorization + Better Auth role/org session + role-gated navigation for a SaaS platform portal vs tenant portal split
**Confidence:** HIGH (all key patterns confirmed by direct code inspection; library facts cited to official docs)

## Summary

This phase is primarily an **App Router refactor**, not a new-feature build. The backend already ships every endpoint required: `POST /api/organizations/:orgId/users` (create-with-credentials), `GET` list members, `PATCH` role, `DELETE`, plus `GET /api/organizations/:orgId/features` behind `useFeatures`. Better Auth is already wired with the `organization` + `admin` client plugins, giving us `User.role` (`admin` | `user`), `Session.activeOrganizationId`, `Member.role` (`admin` | `operator` | `developer` | `viewer`), `authClient.organization.list()` / `setActive()`. The existing `SidebarNav` component is a template for splitting — not throwing away. `/admin/layout.tsx` is a working template for the new `/app/layout.tsx`.

The real work is: (1) create `/app` route tree with its own layout + `TenantNav`, (2) move or alias tenant-scope pages (`cameras`, `projects`, `sites`, `stream-profiles`, `policies`, `recordings`, `map`, `dashboard`, `audit-log`, `developer/*`) under `/app`, (3) trim `PlatformNav` to platform-only items, (4) build Users/Team pages in both portals using existing backend endpoints, (5) swap the sign-in redirect to role-aware, (6) delete the `/dashboard` stub. Role-gating and feature-gating on the sidebar are pure client-side visibility filters; backend guard hardening is explicitly deferred (D-12) but flagged as follow-up.

**Primary recommendation:** Reuse `SidebarNav` structure verbatim — extract into two sibling components `PlatformNav` and `TenantNav` that share a common presentational primitive. Do NOT introduce Next.js middleware for auth — the existing layout-based client-side `authClient.getSession()` pattern is what the rest of the app uses; adding middleware just for this phase would be inconsistent and Better Auth's session cookie is not reliably readable in Edge middleware without an extra server call. Use Next.js **Route Groups** `(platform)` and `(tenant)` only if you want shared root-level styling; since the URL prefixes `/admin` and `/app` are hard requirements (D-01) and already differ, plain folder routing is simpler. The existing backend UsersController is already `SuperAdminGuard`-gated, which is correct for the `/admin/users` page but **wrong for the `/app/team` page** — the tenant Team page needs a new endpoint OR the guard needs relaxing to also accept "Member.role='admin' in :orgId" (see Open Questions).

### Pattern 1: Layout-based authorization (existing — reuse verbatim)

**What:** `useEffect` in `layout.tsx` calls `authClient.getSession()`, redirects on fail, sets active org.
**When to use:** Every protected tree (`admin/`, `app/`).
**Example (mirror of `apps/web/src/app/admin/layout.tsx`):**

```tsx
// apps/web/src/app/app/layout.tsx (NEW)
"use client";
import { useEffect, useState } from "react";
import { useRouter } from "next/navigation";
import { authClient } from "@/lib/auth-client";
import { TenantNav } from "@/components/nav/tenant-nav";
import { Skeleton } from "@/components/ui/skeleton";

export default function AppLayout({ children }: { children: React.ReactNode }) {
  const router = useRouter();
  const [ctx, setCtx] = useState<{
    name?: string; email?: string; orgId: string; orgName: string; memberRole: string;
  } | null>(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    async function check() {
      try {
        const session = await authClient.getSession();
        if (!session.data?.user) { router.replace("/sign-in"); return; }
        // Wrong portal
        if (session.data.user.role === "admin") {
          router.replace("/admin");
          return;
        }
        // Need active org
        let activeOrgId = session.data.session?.activeOrganizationId;
        const orgs = await authClient.organization.list();
        if (!orgs.data || orgs.data.length === 0) {
          // No memberships — bounce with toast
          router.replace("/sign-in?error=no-org");
          return;
        }
        if (!activeOrgId) {
          await authClient.organization.setActive({ organizationId: orgs.data[0].id });
          activeOrgId = orgs.data[0].id;
        }
        const activeOrg = orgs.data.find(o => o.id === activeOrgId) ?? orgs.data[0];
        // Resolve member role for active org (need backend call — see below)
        const role = await fetchMyMemberRole(activeOrg.id); // GET /api/organizations/:id/members/me
        setCtx({
          name: session.data.user.name,
          email: session.data.user.email,
          orgId: activeOrg.id,
          orgName: activeOrg.name,
          memberRole: role,
        });
      } catch {
        router.replace("/sign-in");
      } finally {
        setLoading(false);
      }
    }
    check();
  }, [router]);

  if (loading || !ctx) return <div className="flex min-h-screen items-center justify-center"><Skeleton className="h-8 w-32" /></div>;

  return (
    <div className="flex min-h-screen">
      <TenantNav userName={ctx.name} userEmail={ctx.email} orgName={ctx.orgName} orgId={ctx.orgId} memberRole={ctx.memberRole} />
      <main className="flex-1 md:p-8 p-4 pt-18 md:pt-8">{children}</main>
    </div>
  );
}
```

Source: adapted directly from `apps/web/src/app/admin/layout.tsx` [VERIFIED: code inspection].

### Pattern 4: Role-gated direct URL renders empty state (not 404, not redirect)

Per UI-SPEC: Operator navigates directly to `/app/policies` → page.tsx detects role mismatch and renders the **"You do not have access to this page"** empty state, NOT a redirect. This means every role-gated tenant page needs a lightweight role check at the top of its server/client component. Suggested helper hook:

```tsx
// apps/web/src/hooks/use-require-tenant-role.ts
export function useRequireTenantRole(allowed: TenantRole[]) {
  const { memberRole, loading } = useTenantContext(); // context supplied by /app/layout.tsx
  const allowedNow = !loading && allowed.includes(memberRole);
  return { allowedNow, loading, memberRole };
}
```

Pages render `<RoleGateEmptyState />` when `!allowedNow && !loading`.

### Pattern 5: Active organization context via React Context

**What:** `/app/layout.tsx` resolves active org + member role once and shares via Context so every `/app/*` page can consume without refetch.
**Why:** D-09 + performance; avoids every page re-calling `organization.list()` + `members/me`.

```tsx
// apps/web/src/components/tenant-context.tsx
const TenantCtx = createContext<{orgId: string; orgName: string; memberRole: TenantRole} | null>(null);
export function useTenantContext() { const v = useContext(TenantCtx); if (!v) throw new Error("outside tenant layout"); return v; }
```

### Anti-Patterns to Avoid

- **One filtered sidebar for both personas:** explicitly rejected by memory and D-01. The prior implementation conflated super admin + org admin into one tree — this is the bug being fixed. Do not reintroduce it by e.g. "show/hide" flags in a single nav array.
- **Server-side role check via middleware.ts:** tempting but inconsistent with existing pattern + requires new `/api/auth/get-session` fetch per request. Skip.
- **Duplicating menu definition across desktop + mobile variants:** existing `SidebarNav` defines nav items once and renders via `NavContent` shared between `aside` and `Sheet`. Mirror that in both new components.
- **Checking role with `includes` on a string union from Better Auth without narrowing:** `User.role` is `string | null | undefined` in Better Auth types. Narrow to `admin | user` explicitly (`session.data.user.role === "admin"`).
- **Feature-disabling that also prevents direct URL hit:** UI-SPEC requires the page to still render (showing **"Recordings are not included in your plan"** empty state). Do not short-circuit the route file.
- **Renaming Member.role values:** `admin | operator | developer | viewer` is already persisted in the DB. Keep the wire values; only change UI labels.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Auth session / role / active org | Custom cookie + JWT parser | `authClient.getSession()` + `authClient.organization.list()` + `setActive()` | Better Auth already handles signing, refresh, organization plugin — already wired in `auth-client.ts` |

| Feature gating | Re-fetch `/api/organizations/:id/features` in every page | `useFeatures(orgId)` hook (existing) | Caches per component; feed it into TenantContext if you want single fetch per session |
| Mobile nav overlay | Custom drawer | shadcn `Sheet` component | Already used in `SidebarNav` — reuse identically |
| Toast feedback | Custom snackbar | `sonner` (already installed) | `toast.success("User created. They can sign in now.")` — one line |
| Active-link styling | Recomputed per-link | `pathname.startsWith(item.href)` with exact-match exception for `/app/dashboard` | Exact rule from UI-SPEC Interaction Contract; existing SidebarNav has the pattern [VERIFIED: sidebar-nav.tsx:114] |

## Common Pitfalls

### Pitfall 1: Double-redirect loop between `/admin` and `/app`

**What goes wrong:** `/admin/layout.tsx` redirects non-admin to `/app/dashboard`; if `/app/layout.tsx` then decides something is off (e.g. stale session), it bounces back to `/sign-in` → user re-authenticates → back to `/admin` → back to `/app` → loop.
**Why it happens:** Both layouts run independently; neither knows about the other's state. Transient session states (active org not yet set, stale session during logout) can flip-flop.
**How to avoid:**
- Use `router.replace` (not `push`) for cross-portal redirects — no history entry to flap (UI-SPEC mandates this).
- Add a "in-flight" guard: if `searchParams.get('from') === 'admin'`, skip the same-reason bounce once.
- Set active org **before** rendering the redirect; ensure `setActive` completes before layout concludes portal mismatch.
**Warning signs:** Browser dev-tools shows rapid-fire navigations; `router.replace` called twice in one render.

### Pitfall 2: `authClient.getSession()` is cached per render cycle but not across mounts

**What goes wrong:** After `signOut()`, stale session data in closure can briefly show sidebar before redirect fires.
**Why it happens:** Better Auth's React client caches session state in a store; `getSession()` returns the store value immediately and refetches in background.
**How to avoid:** Use `authClient.useSession()` reactive hook in layouts where you need live updates, or explicitly invalidate via `await authClient.signOut()` which clears the store.
**Warning signs:** Sidebar flickers during sign-out; user sees "wrong portal" banner when signing out.

### Pitfall 3: Member role not fetched — only User.role available from session

**What goes wrong:** `session.data.user.role` is `"user"` (platform-level). To know if the user is `admin | operator | developer | viewer` inside their active org, you need `Member.role` — which is **not on the session**. You must fetch it.
**Why it happens:** Better Auth's organization plugin exposes `organization.list()` and active org ID, but not the caller's member role on the active org, in the session directly.
**How to avoid:**
- Add a **new backend endpoint** `GET /api/organizations/:orgId/members/me` returning `{ userId, role, joinedAt }` OR reuse existing `listMembers` and filter client-side (cheap — org has few members).
- Cache result in TenantContext so child pages don't re-fetch.
**Warning signs:** Every page makes its own `listMembers` call; role check always returns undefined.

### Pitfall 4: `UsersController` is `SuperAdminGuard`-only — tenant Team page will 403

**What goes wrong:** D-19 says Org Admin uses tenant Team page to manage team. But `apps/api/src/users/users.controller.ts:25` applies `@UseGuards(SuperAdminGuard)` to the entire controller. Org Admin (User.role='user') hitting `POST /api/organizations/:orgId/users` will get 401 "Super admin access required".
**Why it happens:** Phase 01 built the controller only for super admin use; no Member.role guard existed.
**How to avoid:**
- **Option A (minimal):** Create a new `OrgAdminGuard` that checks `Member.role === 'admin' in :orgId` and mount a parallel controller `/api/organizations/:orgId/team/*` that uses it. Super admin keeps `/users`, Org Admin gets `/team`.
- **Option B:** Relax existing `SuperAdminGuard` on `UsersController` to `SuperAdminOrOrgAdminGuard` (accepts either User.role='admin' OR Member.role='admin' for the specified orgId).
- Either way, this is a **real backend change** — D-20 says "backend endpoints already support this" which is **not quite true** — it supports super-admin-acting-on-any-org but not org-admin-acting-on-own-org. Planner must surface this.
**Warning signs:** Tenant Team page "Add Team Member" returns 401; curl with Org Admin session fails on POST.

> **CRITICAL for planner:** This contradicts CONTEXT.md D-20 ("backend ready, just wire up"). Research says D-20 is inaccurate — backend DOES need a guard change. Either expand phase scope or add a small backend task.

### Pitfall 5: Feature gate race — tenant sidebar renders before features load

**What goes wrong:** `useFeatures(orgId)` returns `loading: true` initially with `features: {}`. If TenantNav renders unconditionally, Recordings/Webhooks/Map/API Keys/Audit Log all momentarily **hide** (because `isEnabled` returns false while loading), then pop in after fetch resolves.
**Why it happens:** `isEnabled(key) => features[key] === true` returns false for unloaded state.
**How to avoid:**
- Gate the sidebar render on `features.loading === false` (show existing `Skeleton` per UI-SPEC loading pattern).
- Or: default to **showing** the item while loading, then hide if feature resolves false (less clean; causes "pop out").
- Safer: block at layout level — don't render `<TenantNav>` until both session and features are ready.
**Warning signs:** First paint of sidebar has fewer items than second paint.

### Pitfall 6: Next.js folder conflict with route group `(dashboard)` still present

**What goes wrong:** D-23 deletes `/dashboard` stub. If the folder `apps/web/src/app/(dashboard)/` still exists when the planner creates `/app/dashboard/`, the URL `/dashboard` (without `/app`) still resolves (route groups are invisible in URL — `(dashboard)/page.tsx` maps to `/`, not `/dashboard`). [VERIFIED: reading the folder — `(dashboard)/page.tsx` is actually the root `/` page]
**Why it happens:** Route group `(dashboard)` wraps `/` — naming is misleading. The actual path `/dashboard` was never occupied; the admin layout redirects `/dashboard` which is a 404-returning-to-signin currently.
**How to avoid:**
- Re-read `apps/web/src/app/(dashboard)/page.tsx` and `layout.tsx` before deletion — this is the **root route** `/`, not `/dashboard`.
- After deletion, the root `/` will 404 unless you add `apps/web/src/app/page.tsx` that redirects based on role (mirror of `sign-in/page.tsx`'s post-login decision).
**Warning signs:** After removing `(dashboard)/`, visiting `http://localhost:3000/` 404s. Add a new root `app/page.tsx` that pushes to `/admin` or `/app/dashboard` based on session.

### Pitfall 7: Moving pages breaks relative imports

**What goes wrong:** Moving `/admin/cameras/page.tsx` to `/app/cameras/page.tsx` is physically a folder move, but all pages import via `@/...` (alias) which is depth-invariant. Relative `../` imports are rare in this project but exist (grep first).
**How to avoid:** Grep `from "\.\./\.\./` under `apps/web/src/app/admin/` before moving.
**Warning signs:** TypeScript errors after move.
