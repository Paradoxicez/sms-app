## Intentionally-dropped assertions (with rationale)

All drops land because the landed UI-SPEC (see `999.1-UI-SPEC.md` + Plan 02/04 summaries) diverged from the Wave 0 stub authors' assumptions.

| Original assertion | Why dropped | Evidence |
|--------------------|-------------|----------|
| `platform-users-page` expects `<label>Role filter` + `fetch(...role=operator)` | The page aggregates users via `authClient.organization.list() × GET /api/organizations/:orgId/users` per Plan 04 — there is no server-side role filter and no client-side role-filter UI. | `src/app/admin/users/page.tsx` (no `role=` query); `999.1-04-SUMMARY.md` §"Cross-org aggregation" |
| `platform-users-page` expects "Add Platform User" button + User.role `{admin, user}` option | Button copy is "Create User"; dialog uses the shared `CreateUserSchema` member-role enum `{admin, operator, developer, viewer}`. Minting User.role='admin' is out-of-scope for AUTH-04 and would violate the shared-schema decision. | `src/app/admin/users/page.tsx:90`; `src/app/admin/users/components/create-platform-user-dialog.tsx:228-232`; `src/lib/validators/create-user.ts` |
| `tenant-nav` expects "Settings" link for admin | UI-SPEC landed with Sites replacing Settings in the Cameras group; no Settings route exists. | `src/components/nav/tenant-nav.tsx:70` (Sites in CAMERAS group) |
| `team-page` hook `@/hooks/use-current-member` | Hook never existed — implementation uses `@/hooks/use-current-role` (returns `{userRole, memberRole, activeOrgId, activeOrgName, loading}`). | `src/hooks/use-current-role.ts` |
