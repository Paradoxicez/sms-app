<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser form → POST /api/organizations/:orgId/users | Untrusted input; validated client-side (Zod) and server-side (Zod DTO + OrgAdminGuard from Plan 01) |
| Org Admin action → DELETE :orgId/users/:userId | Can only target members of own org (backend guard enforces) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-999.1-03 | E (Elevation of privilege) / cross-tenant write | AddTeamMemberDialog | mitigate | Dialog posts to `{activeOrgId}` from session; Plan 01 OrgAdminGuard rejects if session user is not admin of that org. Client can't spoof orgId because activeOrgId comes from Better Auth session, not user input. |
| T-999.1-10 | E (Elevation of privilege) / role escalation | Role Select in team dialog | mitigate | Select only exposes 4 Member roles; cannot pick User.role='admin'. Backend CreateUserSchema enum also rejects anything outside the 4. Defense in depth. |
| T-999.1-11 | S (Spoofing) | Self-removal | mitigate | Team table hides Remove action for self; backend should ALSO reject self-removal (flagged as FOLLOW-UP — out of scope this phase per D-12). |
| T-999.1-12 | I (Information disclosure) | Platform Users aggregation | accept | Super admin legitimately sees all users; aggregation reveals org membership. No data leak outside admin role. |
| T-999.1-13 | T (Tampering) | Password field | mitigate | Password sent over HTTPS to backend which hashes (Better Auth). Never logged client-side. Minimum length enforced both sides. |
</threat_model>
