# Phase 999.1: Role-based Sidebar Navigation - Discussion Log

> **Audit trail only.** Decisions captured in CONTEXT.md.

**Date:** 2026-04-15
**Phase:** 999.1-role-based-sidebar-navigation
**Mode:** discuss

## Context Entered

User hit blocker during Phase 01 UAT — regular user login → no sidebar, redirected to `/dashboard` stub. Traced flow:
- `sign-in/page.tsx:74` pushes ALL logins to `/admin`
- `/admin/layout.tsx:27-30` redirects `User.role !== 'admin'` → `/dashboard`
- `/dashboard` = plain welcome stub with no sidebar

This is exactly the 999.1 backlog gap.

## Discussion

### Round 1 — Framing (my initial misread)
I presented gray areas assuming the choice was between "2 tiers (Super/Org)" vs "4 tiers (Super/Admin/Op/Viewer)". Offered pragmatic 2-tier to unblock UAT faster.

### Round 2 — User correction (critical reframe)
User clarified: this is SaaS.
- Super Admin = platform operator (us)
- Org Admin = tenant (customer who rents the service)
- Prior implementation wrongly conflated both into one `/admin` sidebar
- Correct architecture: two separate portal surfaces, not one filtered sidebar

Saved as feedback memory `saas_role_architecture.md` — this is load-bearing for future phases.

### Round 3 — Final decisions

| Question | Decision |
|----------|----------|
| Portals | Two separate: `/admin` (platform) and `/app` (tenant) |
| Feature gate UI | Hide menu items when package disables feature (per Phase 01 UAT-8) |
| Tenant roles | 4 tiers using existing `Member.role` enum (admin/operator/developer/viewer) — Org Admin manages their own team's roles |
| User management | Closed system, create-with-credentials flow. Both portals need Users page (currently missing in both). No public self-register, no email invitation in 999.1 |

### Scope refinement

Expanded from "sidebar filter" (original roadmap framing) to:
1. Create `/app` route group + layout + sidebar
2. Trim `/admin` sidebar to platform-only
3. Role + feature aware menu in `/app`
4. User management UI in BOTH portals
5. Login redirect routing
6. Backend role guards — flagged as out-of-scope but surfaced as follow-up

## Deferred (captured, out of 999.1)

- Email invitation flow (token + email service)
- Super admin tenant impersonation
- Multi-org switcher
- Billing/usage UI
- Upsell CTA on hidden features

## Key Design Decision

Super Admin does NOT see tenant-scope features. No "admin view all" sidebar. Clean separation. Future impersonation feature is how super admin can "enter" a tenant, if needed.
