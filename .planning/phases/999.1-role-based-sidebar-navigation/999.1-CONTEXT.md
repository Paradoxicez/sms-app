# Phase 999.1: Role-based Sidebar Navigation - Context

**Gathered:** 2026-04-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Build the missing tenant portal (`/app`) and trim the platform portal (`/admin`) so that SaaS architecture is correctly reflected in routing, navigation, and user management. Two separate portal surfaces — not one filtered sidebar.

**In scope:**
- New `/app` route group (tenant portal) with its own layout + sidebar
- Trim `/admin` sidebar to platform-only items
- Role-aware menu (Org Admin / Operator / Developer / Viewer) inside tenant portal
- Feature-aware menu (hide items when org package disables feature)
- User management UI in both portals (closed system — create with credentials, no self-register)
- Login redirect routing (`User.role='admin'` → `/admin`, `User.role='user'` → `/app`)
- Active organization context in tenant portal

**Out of scope (deferred to later phases):**
- Email invitation flow (token + email service)
- Super admin tenant impersonation
- Multi-org switcher (assume user belongs to one org primarily; auto-select first)
- Billing / usage UI in platform portal
- Moving page source code (menu + route rewrites only; page components stay where they are unless required)

</domain>

<decisions>
## Implementation Decisions

### Portal architecture
- **D-01:** Two separate route groups with independent layouts: `/admin` = platform portal (super admin), `/app` = tenant portal (org users)
- **D-02:** `User.role='admin'` → platform portal only. `User.role='user'` → tenant portal only. Hard redirect if wrong portal is accessed
- **D-03:** Super admin does NOT see tenant-scope features (cameras/projects/sites/recordings/policies/developer) in their sidebar. If a tenant issue needs platform intervention, that is a future "impersonate tenant" feature — not in this phase.

### Platform portal (`/admin`) menu
- **D-04:** Menu items: Organizations, Packages, Cluster Nodes, SRS Engine, Platform Audit Log, Users (platform-level user management)
- **D-05:** Platform Dashboard shows cross-org aggregates (total cameras/bandwidth/cluster health). Platform Audit Log shows platform-level events (org created, package changed, super admin actions)
- **D-06:** Remove from `/admin`: Projects, Cameras, Sites, Stream Profiles, Stream Engine (keep as Cluster Nodes), Policies, Recordings, Map, Developer section, Dashboard (org-scoped), Audit Log (org-scoped)

### Tenant portal (`/app`) menu
- **D-07:** Menu items: Dashboard (org-scoped), Cameras, Map, Projects, Sites, Stream Profiles, Recordings*, Policies, Team (users in org), API Keys, Webhooks, Audit Log (org-scoped). Items marked * are feature-gated.
- **D-08:** Tenant Dashboard shows org-scoped data only (their cameras, their bandwidth, their alerts)
- **D-09:** All `/app/*` routes require active organization context. Resolve from `session.activeOrganizationId` on layout mount. If user belongs to multiple orgs, auto-select first (no switcher in this phase)

### Role tiers inside tenant
- **D-10:** Four tenant tiers map to existing `Member.role` enum values: `admin` | `operator` | `developer` | `viewer`
- **D-11:** Role → permitted menu matrix:
  - **Org Admin** (`admin`): all tenant menu items
  - **Operator** (`operator`): Dashboard, Cameras (view+control), Map, Recordings*, Audit Log. NOT Projects/Sites/Stream Profiles/Policies/Team/API Keys/Webhooks
  - **Developer** (`developer`): Dashboard, Cameras, Map, API Keys, Webhooks, Audit Log. NOT Projects/Sites/Stream Profiles/Recordings/Policies/Team
  - **Viewer** (`viewer`): Dashboard (read), Cameras (view only), Map, Recordings* (view), Audit Log (read). NOT any write action
- **D-12:** Menu filtering is client-side visibility only. Backend MUST still enforce per-route guards (out of scope for this phase — note in FOLLOW-UP that Nest guards should be audited for Member.role checks)

### Feature-aware filtering
- **D-13:** When org's package disables a feature, corresponding menu items are **hidden** (not disabled/greyed). Aligns with Phase 01 UAT-8 requirement
- **D-14:** Feature → menu mapping: `recordings: false` → hide Recordings; `webhooks: false` → hide Webhooks; `map: false` → hide Map; `apiKeys: false` → hide API Keys; `auditLog: false` → hide Audit Log
- **D-15:** Fetch org features via existing `useFeatures(orgId)` hook on tenant layout mount; cache per session

### User management (closed system)
- **D-16:** Closed system — no public self-registration. Users are created by admins only
- **D-17:** Create-with-credentials flow: admin fills form (email, name, password, role), user can log in immediately. No email invitation / magic link in this phase
- **D-18:** Platform portal Users page: super admin can create platform-level users (`User.role='admin'`) and org admins (`User.role='user'` + `Member.role='admin'` in target org). Super admin can list all users across all orgs with filter.
- **D-19:** Tenant portal Team page: Org Admin can create/invite users within their own org. They choose role from `admin | operator | developer | viewer`. Cannot promote anyone to `User.role='admin'` (that's platform-level)
- **D-20:** Existing backend endpoints already support this: `POST /api/organizations/:orgId/users` (create with credentials). Frontend just needs to wire up

### Login & redirect flow
- **D-21:** After successful login: if `User.role='admin'` → push `/admin` (not `/admin/organizations` specifically; let /admin/page.tsx handle landing). If `User.role='user'` → push `/app/dashboard`
- **D-22:** `/admin/layout.tsx` checks `User.role === 'admin'` else redirect `/app/dashboard`. `/app/layout.tsx` checks `User.role === 'user'` AND has active org else redirect `/sign-in`
- **D-23:** Remove the `/dashboard` stub route (current welcome page) — replaced by `/app/dashboard`

### Claude's Discretion
- Org creation flow: super admin creates org → separately creates first Org Admin via Users page, OR combined form with optional "First Admin email/password/name" fields. Planner decides based on simpler UX.
- Sidebar visual design — reuse existing `SidebarNav` component styling, split into `PlatformNav` + `TenantNav` components with same look
- Active organization resolution: if `session.activeOrganizationId` is null but user has memberships, call `authClient.organization.setActive(firstOrgId)` on mount (same pattern as current admin layout)

</decisions>

<specifics>
## Specific Ideas

- SaaS vocabulary: "Super Admin" = platform operator, "Tenant" = customer org. The UI should reflect this mental model — super admin doesn't feel like they can "use" the product, they feel like they're running it
- Closed system: think B2B SaaS where sales onboards customers, not a public signup funnel
- Prior attempt conflated the two personas into one `/admin` sidebar — root cause of Phase 01 UAT-8 being blocked and the "regular user sees no sidebar" bug
- Feature gating should feel clean: if org hasn't bought recordings, the menu item simply isn't there. No upsell clutter in 999.1 (may add later)

</specifics>

### Reusable Assets
- `SidebarNav` component + all icon imports + responsive Sheet menu — can be duplicated/split into `PlatformNav` and `TenantNav` with the same look & feel

- `authClient` (Better Auth) — `getSession()`, `organization.list()`, `organization.setActive()`, `signOut()`

- Existing `SuperAdminGuard` — protects platform endpoints
- Existing UI components: Button, Avatar, DropdownMenu, Sheet, Separator, Dialog (for user create form), Table (for user list) — shadcn/ui library

- Route groups: `(auth)`, `(dashboard)`, `admin` already exist — add `app` as a new route group (or similar)
- Layouts check session via `authClient.getSession()` in useEffect, redirect on fail — reuse this pattern in new tenant layout
- Forms use zod schemas in `dto/*.ts` files — reuse CreateUserSchema for client-side validation too (already shared)
- Page structure: each admin section has `page.tsx` + `components/*-table.tsx` + dialogs — follow same structure for Team page

### Integration Points
- Move route targets in `sign-in/page.tsx` — role-aware push after login
- Update `/admin/layout.tsx` to restrict menu items (remove tenant-scope items from platform sidebar)
- Create `/app/layout.tsx` mirroring admin layout pattern
- New `/app/page.tsx` or redirect to `/app/dashboard`
- Team (users) pages: `/app/team/page.tsx` for tenant, `/admin/users/page.tsx` for platform
- Remove or repurpose `/(dashboard)/page.tsx` stub
