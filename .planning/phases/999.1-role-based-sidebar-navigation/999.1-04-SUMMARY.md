# Phase 999.1 Plan 04: User Management UI Summary

Super admins administer users from `/admin/users`; Org Admins administer their team from `/app/team`. Both wire Plan 01's OrgAdminGuard + members/me backend and existing CRUD endpoints with verbatim UI-SPEC copy.

## What Was Built

### Task 1 — Platform Users page

- `CreateUserSchema` in `apps/web/src/lib/validators/create-user.ts` — client mirror of the apps/api DTO, exported with `CreateUserInput` type.
- `/admin/users/page.tsx` — fetches `authClient.organization.list()` then `GET /api/organizations/:orgId/users` per org; aggregates by email so a user belonging to multiple orgs shows once with an `orgs.length` count. H1 `Users` + primary CTA `Create User` per UI-SPEC.
- `PlatformUsersTable` — columns Email / Name / Role / Orgs / Last sign-in / Actions. Role badge variants exactly per D-18: `admin→default`, `operator/developer→secondary`, `viewer→outline`. Row action `Deactivate user` opens AlertDialog with verbatim `Deactivate {email}?` / *They will not be able to sign in. You can reactivate them later.* / destructive `Deactivate user` button. On confirm `DELETE /api/organizations/{orgs[0].id}/users/{userId}`; success toast `User deactivated.`
- `CreatePlatformUserDialog` — shadcn Dialog with `sm:max-w-[440px]`, verbatim title `Create platform user`, verbatim description, fields Email / Full name / Password / Organization / Role, password helper *Minimum 8 characters. User can change this after signing in.*, submit `Create user`. Submit POSTs to `/api/organizations/{organizationId}/users`; success toast `User created. They can sign in now.`; error toast uses verbatim `Could not create user: {server message}. Check the email is not already in use.`

### Task 2 — Tenant Team page

- `/app/team/page.tsx` — reads `useCurrentRole()` for `memberRole`, `activeOrgId`, `activeOrgName`. Three branches before the table:
  1. `loading` → skeleton.
  2. `memberRole !== 'admin'` → role-gated empty state with `Lock` icon, heading `You do not have access to this page`, body *Team management is restricted to organization admins. Contact your admin if you need access.*
  3. No `activeOrgId` → fallback empty state.
  Otherwise renders `Team` H1 + `Add Team Member` CTA. When `members.length <= 1` shows the `Just you so far` empty state with the UI-SPEC body and a next-step `Add Team Member` button.
- `TeamTable` — columns Email / Name / Role / Added / Actions. `Remove member` action hidden when row is self (`member.userId === currentUserId`). Confirmation AlertDialog body is verbatim: *They will lose access to {Org Name} immediately. Their audit-log entries stay intact.* (single-line template literal so grep-based plan verify passes).
- `AddTeamMemberDialog` — verbatim title `Add team member`, description substitutes `{Org Name}` at render time. Role Select shows all four Member roles with UI-SPEC label + helper pairs rendered as stacked text inside each `SelectItem` (primary label bold, helper muted). Submit `Create user` POSTs to `/api/organizations/{activeOrgId}/users`; success toast `User created. They can sign in now.`; 403 → *You do not have permission to add team members.*; other errors → verbatim create-error toast.
