<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SRS stream registry → probe worker | Trusted (loopback HTTP); we assume `/api/v1/streams` is not tampered with. |
| Probe worker → codec payload written to DB | Trusted; the worker writes camera.codecInfo via tenancy or system client as already established in Phase 19. |
| Probe worker → SRS DELETE kick | Self-initiated; no external input. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-19.1-PUB-SPOOF | Spoofing | Codec-mismatch kick flow | mitigate | Passthrough profile + non-H.264/AAC → publisher is disconnected within ~1s of publish. Combined with Plan 02's on_publish 403 for unknown keys, a publisher can't both spoof a key AND bypass codec enforcement. |
| T-19.1-SK-LEAK | Information disclosure | Probe-mismatch audit payload | mitigate | Audit details use `camera.streamKey.slice(0, 4)` — never the full key. Detected codec is a short display string ("H.265", "AAC"), not the full SRS stream metadata. |
| T-19.1-KICK-LOOP | Denial of service | If an encoder retries immediately after kick with same non-H.264 codec | accept | Encoder will be kicked again every ~1s until user accepts auto-transcode or changes codec. Not a DoS on the platform (bounded work, single camera). User sees the mismatch banner in the UI and takes action. |
| T-19.1-FORWARD-LOOP | Denial of service | StreamsService push+passthrough no-op | mitigate | Early return ensures the stream queue is never enqueued for push+passthrough cameras, avoiding a scenario where FFmpeg tries to read from push/key while SRS forward also runs. Single responsibility per camera mode. |
</threat_model>
