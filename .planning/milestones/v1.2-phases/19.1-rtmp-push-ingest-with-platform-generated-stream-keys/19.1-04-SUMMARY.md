# Phase 19.1 Plan 04: Wave 2 Backend C — Push Routing + Codec-Mismatch Detection

Wire the two remaining backend decisions for push ingest: (1) StreamsService routes push cameras to the SRS loopback instead of their external publish URL, and (2) StreamProbeProcessor detects on-the-wire codec mismatch on passthrough profiles, kicks the offending publisher, and emits a prefix-only audit event. Also clears the 10 pre-existing `stream-probe.test.ts` failures that Plan 01 deferred.

## What was built

### Task 1 — StreamsService push routing (D-17)

`StreamsService.startStream(cameraId)` now branches on `camera.ingestMode`:

| Mode                  | inputUrl                                               | Queue                  |
| --------------------- | ------------------------------------------------------ | ---------------------- |
| `pull` (any profile)  | `camera.streamUrl` (external RTSP/RTMP — unchanged)    | enqueued as before     |
| `push` + transcode    | `rtmp://127.0.0.1:1935/push/${camera.streamKey}`       | enqueued (loopback in) |
| `push` + passthrough  | n/a — early return, SRS `forward` handles the remap    | **NOT enqueued**       |

Rationale:
- **push+passthrough short-circuits** because SRS's native `forward` block already rewrites `push/<key>` → `live/<orgId>/<cameraId>` at the protocol layer; spawning FFmpeg on top would create a duplicate-reader conflict and waste a BullMQ concurrency slot.
- **push+transcode** reads from the SRS loopback because the encoder only knows the external publish URL; inside the platform, that URL is not routable — only SRS can read it.
- `buildStreamJobData` in `apps/api/src/resilience/job-data.helper.ts` applies the same routing so BootRecoveryService, CameraHealthService, and SrsRestartDetector all re-enqueue push cameras with the loopback input URL. Without this change, a boot-time recovery would resurrect push cameras with an unreachable input.

### Task 2 — Codec-mismatch detection + kick + audit (D-16, D-21)

`StreamProbeProcessor.process` gains a mismatch branch inside the `source='srs-api'` success path. After pulling `{video, audio}` from `/api/v1/streams`, it looks up the camera via `systemPrisma.camera.findUnique` to read `ingestMode`, `streamKey`, and `needsTranscode`:

- **push + passthrough + video ≠ H.264/AVC OR audio ≠ AAC** →
  1. Write `codecInfo.status='mismatch'` with `mismatchCodec` = first offending codec (video precedence).
  2. Look up the publisher via `srsApi.findPublisherClientId('push/<streamKey>')`.
  3. If found, call `srsApi.kickPublisher(clientId)` — SRS closes the RTMP connection, encoder sees an error, user is forced to act.
  4. Emit `camera.push.publish_rejected` audit with `details = { streamKeyPrefix: key.slice(0,4), reason: 'codec_mismatch', detectedVideo, detectedAudio }`.
  5. Return — DO NOT fall through to the success write.

- **push + transcode** + any codec → success (FFmpeg transcodes anything).
- **pull** → existing ffprobe path unchanged.

Codec matching uses case-insensitive regex: video accepts `h264 | h.264 | avc | avc1` (SRS has been observed returning all four); audio requires exact `aac`.

## Deviations from Plan

### [Rule 2 — Critical functionality] Extended push routing to `buildStreamJobData`

- **Found during:** Task 1 verification.
- **Issue:** The plan's action section edited only `StreamsService.startStream`, but `apps/api/src/resilience/job-data.helper.ts` (used by BootRecoveryService, CameraHealthService, SrsRestartDetector) constructs `StreamJobData` via its own helper that still read `camera.streamUrl`. At API boot or after an SRS restart, push cameras would be re-enqueued with an unreachable external RTMP URL and fail the FFmpeg attempt until max-retry offline.
- **Fix:** Mirrored the ingest-mode branch in `buildStreamJobData` so all four call sites emit identical payloads.
- **Files modified:** `apps/api/src/resilience/job-data.helper.ts`
- **Commit:** `03a3b52`

### [Rule 1 — Bug] `StreamProbeProcessor.statusGateway` forced DI crashed every unit test

- **Found during:** Attempt to run Plan 01's deferred failures.
- **Issue:** The Phase 19 gateway-broadcast landed without making `statusGateway` optional. Any unit test that constructed the processor without injecting a gateway mock hit `TypeError: Cannot read properties of undefined (reading 'broadcastCodecInfo')`.
- **Fix:** Added `@Optional()` to `statusGateway` + `auditService` constructor params. Broadcast + audit sites now use optional-chaining and explicit null-guards. Production DI behavior unchanged because both modules are `@Global()`.
- **Files modified:** `apps/api/src/streams/processors/stream-probe.processor.ts`
- **Commit:** `a09f632`

### [Rule 1 — Bug] `tests/cameras/stream-probe.test.ts` assertions used stale copy

- **Found during:** Same deferred-failures pass.
- **Issue:** 5 `normalizeError` assertions and 1 srs-api-not-found assertion asserted against the pre-Phase-19-UX short literals. The dictionary had been rewritten to user-friendly longer copy but the tests weren't updated.
- **Fix:** Aligned test expectations to the current dictionary and renamed test descriptions to describe behavior (phrasing) rather than quoting the exact literal.
- **Files modified:** `apps/api/tests/cameras/stream-probe.test.ts`

| Threat ID             | Disposition | Status                                                                                                                                                                                                                                                                                                                                                 |
| --------------------- | ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| T-19.1-PUB-SPOOF      | mitigate    | Mismatch branch kicks the publisher within ~1s of probe completion (probe queue fires from on_publish). | T-19.1-SK-LEAK        | mitigate    | Audit details use `camera.streamKey.slice(0, 4)`; full-key presence asserted absent in the JSON-stringified payload test. Detected codec display string is short and has no secret content.                                                                                                                                                             |
| T-19.1-KICK-LOOP      | accept      | As per plan's threat model: encoder retrying with the same bad codec gets kicked again every probe cycle. Bounded work (one camera, ~1s/cycle, audit log grows linearly with retries). User sees the banner and resolves out-of-band. Not an attack vector, just a UX artifact.                                                                         |
| T-19.1-FORWARD-LOOP   | mitigate    | `startStream` early-return for push+passthrough prevents FFmpeg from ever being spawned against `push/<key>`, so the only reader of that stream path is SRS `forward`. Duplicate-reader conflict impossible.                                                                                                                                            |
