# Phase 19.1 Plan 00: Wave 0 Scaffolding Summary

Land the dependency + test-infrastructure baseline for Phase 19.1 — install nanoid v3 (CJS-compatible), add an idempotent migration seeding `ingestMode='pull'` on existing Camera rows, and create 14 vitest scaffolds with 72 `it.todo` entries so every subsequent plan runs its verify command against a real test file.

## What was built

- **nanoid@3.3.11** added as an explicit runtime dependency of `@sms-platform/api` (previously a transitive dep only). v3.x locked in to preserve CommonJS import compatibility on Node 22.11 — v5 is ESM-only and would trip `ERR_REQUIRE_ESM` inside the NestJS build pipeline.
- **`camera_push_fields/migration.sql`** — an idempotent SQL migration that seeds `ingestMode='pull'` on every pre-existing Camera row before Plan 01's `prisma db push` adds the column. Guards the `UPDATE` with `information_schema.columns` so the file is a no-op on first run (column not yet present) and on every subsequent run (all rows already seeded). `db:push` chain now runs it between `prisma db push --accept-data-loss` and `rls_apply_all/migration.sql`.
- **`push-camera.fixture.ts`** — exports `createPushCameraFixture(orgId, siteId, prisma, overrides?)` signature returning `{ camera, streamKey, pushUrl }`. Stub throws; Plan 01 wires the real implementation once `ingestMode` + `streamKey` columns land.
- **14 test scaffolds** with 72 total `it.todo` entries across `apps/api/tests/{cameras,srs,streams}/`, each entry tagged with the implementing Plan number and relevant decision ID from `19.1-CONTEXT.md`. Every file is a clean vitest-collectable module (`describe` + `it.todo` only — no assertion helpers yet).
