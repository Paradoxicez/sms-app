## Pre-locked decisions (not asked during session)

From `PRE-DISCUSS-NOTES.md` (dated 2026-04-23, carried in before session):

- **RTMPS ingest → deferred** to a later SRS v6→v7 migration phase. Rationale: SRS v6 has no native RTMPS; v7 has it from v7.0.56+ but is still flagged Unstable as of 2026-04-23. v7 GA planned for late 2026. Rejecting nginx/stunnel proxy in favor of bundling RTMPS + native RTSP output + v7 upgrade into one future phase.

## Foundation

### URL scheme for push endpoint

| Option | Description | Selected |
|--------|-------------|----------|
| `push/<opaque-key>` new SRS app | Opaque key resolved via DB. No orgId/cameraId leak. Branch in parseStreamKey + new app. | ✓ |
| `live/{orgId}/{cameraId}` reuse | Reuse existing path, streamKey = cameraId. Zero SRS change. Leaks orgId+cameraId. Rotation awkward. | |
| `live/{orgId}/<opaque-key>` hybrid | Coexists with pull but parseStreamKey needs fallback DB lookup → complexity. | |

**User's choice:** `push/<opaque-key>` (recommended).

### Stream-key format

| Option | Description | Selected |
|--------|-------------|----------|
| nanoid 21 chars | URL-safe alphabet, 128-bit entropy, shorter than UUID. `nanoid` lib. | ✓ |
| UUID v4 (36 chars) | Familiar, `crypto.randomUUID`. Longer, hyphens as delimiter. | |
| Short 12-char slug | Compact, ~72-bit entropy. Guessable at scale. | |

**User's choice:** nanoid 21 chars (recommended).

### Data model columns

| Option | Description | Selected |
|--------|-------------|----------|
| ingestMode + streamKey, streamUrl stored | Add enum + nullable column. streamUrl stored (denormalized for uniform response). | ✓ |
| ingestMode + streamKey, streamUrl computed | Compute streamUrl on-the-fly for push. Saves storage, makes serializer mode-aware, unique may drift. | |
| Polymorphic `source` JSON | Drop streamUrl, polymorphic. Flexible but breaks Phase 19 constraint + query patterns + type safety. | |

**User's choice:** ingestMode + streamKey + stored streamUrl (recommended).

### streamKey uniqueness scope

| Option | Description | Selected |
|--------|-------------|----------|
| Global unique | `@@unique([streamKey])`. Required because URL has no orgId for on_publish resolution. | ✓ |
| Per-org unique | `@@unique([orgId, streamKey])`. Would require orgId embedded in URL. | |

**User's choice:** Global unique (recommended).

## Zero-transcode (reformulated after user clarification)

**Clarification note:** The initial framing of zero-transcode as an independent architectural toggle was rejected. The user pointed out that Stream Profile (Passthrough / Transcode, validated in v1.0) is the existing mechanism and Phase 19.1 should not introduce a parallel control. Questions were reformulated around how push integrates with the existing Stream Profile system.

### FFmpeg input for push + Transcode profile

| Option | Description | Selected |
|--------|-------------|----------|
| `rtmp://127.0.0.1:1935/push/<key>` | FFmpeg reads from SRS loopback. Reuses pull orchestration, only input URL differs. | ✓ |
| FFmpeg triggered by on_publish hook | Callback spawns FFmpeg. Diverges from BullMQ pattern, loses resilience. | |
| SRS native transcode directive | CLAUDE.md bans this (libx264 only, no API control). | |

**User's choice:** `rtmp://127.0.0.1:1935/push/<key>` (recommended).

### Codec mismatch on Passthrough profile

| Option | Description | Selected |
|--------|-------------|----------|
| Reject publish with clear error | 403 on on_publish + codecInfo.status = 'failed' + reason. User sees UI. | |
| Auto-upgrade to transcode default | Silent switch. Surprising CPU bill. | |
| Accept + warning in UI | Silent success/failure based on SRS behavior. Not predictable for API consumers. | |
| **Auto-upgrade with user consent** | Hybrid: first publish reject → notify → user approves → switch to transcode on next publish. | ✓ (user "Other") |

**User's choice:** "Auto-upgrade เป็น transcode default ตามข้อที่ 2 แต่ว่าให้เตือน user ว่าจะเกิดอะไรขึ้น ให้ user รับทราบ user สามารถยินยอมหรือไม่ยินยอมก็ได้."
**Notes:** Auto-upgrade requires explicit consent, not silent. See follow-up question for consent flow design.

### Consent flow for codec mismatch

| Option | Description | Selected |
|--------|-------------|----------|
| First publish reject + notify + per-camera consent | UI banner + Accept/Dismiss → profile update → next publish uses transcode path. | ✓ |
| Org-level default policy | Settings surface for Reject / Auto-transcode / Ask. Wider scope. | |
| Pre-publish consent (save-time) | Upfront on Passthrough pick. Clicks without context (no codec probe yet). | |

**User's choice:** First publish reject + per-camera consent (recommended).

### Playback URL preservation

| Option | Description | Selected |
|--------|-------------|----------|
| Keep `live/{orgId}/{cameraId}` contract | SRS forward rule or cheap FFmpeg `-c copy` relay maps push→live. Planner picks mechanism. | ✓ |
| Expose `push/<key>/index.m3u8` | Leaks key in playback URL; breaks uniform contract. | |

**User's choice:** Keep contract (recommended).
