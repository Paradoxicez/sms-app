# Phase 19.1 Plan 03: Wave 2 CamerasService Push Surface Summary

Complete the CamerasService push-ingest surface: createCamera/bulkImport/deleteCamera push branches, rotateStreamKey (D-19/D-20), serializeCamera masking chokepoint (D-07), real findByStreamKey / markFirstPublishIfNeeded / resolveForwardTarget implementations (consumed by Plan 02 in the parallel worktree), POST /api/cameras/:id/rotate-key controller endpoint, real push-camera fixture, and 26 passing unit tests across 6 files with a 21-char nanoid leak-detector invariant.

## What was built

### CamerasService push branches (Task 1)

- **`createCamera`** — `ingestMode === 'push'` branch: generates 21-char nanoid via `generateStreamKey()`, composes `rtmp://{SRS_PUBLIC_HOST}:1935/push/{key}` via `buildPushUrl()`, stores both, emits `camera.push.key_generated` audit with `streamKeyPrefix` (4 chars, never full key). P2002 on `streamKey` → `DuplicateStreamKeyError` (409); P2002 on `streamUrl` → `DuplicateStreamUrlError` (409). Pull path unchanged from Phase 19.
- **`bulkImport`** — each push row gets a server-generated key + URL inside the tenancy transaction; pull rows keep Phase 19 URL-dedup semantics. Response shape extended with `cameras: { id, name, ingestMode, streamUrl }[]` for the UI CSV download (D-14). Mixed batches insert all rows in one transaction.
- **`rotateStreamKey(cameraId, userId)`** — tenancy 404 / BadRequestException on pull / resolve old SRS client_id → DB rotate in one call → best-effort kick → `camera.push.key_rotated` audit with `oldKeyPrefix` + `newKeyPrefix` (4 chars each). Returns `{ streamUrl }`.
- **`deleteCamera`** — push branch calls `SrsApiService.findPublisherClientId('push/<key>')` + `kickPublisher()` before hard-delete. Kick failure is logged but does not block deletion (admin must always be able to remove a camera).
- **Three Plan-02-consumer helpers** (`findByStreamKey`, `markFirstPublishIfNeeded`, `resolveForwardTarget`) — implemented with `systemPrisma` to bypass RLS (SRS callbacks have no CLS context). `markFirstPublishIfNeeded` uses `firstPublishAt: null` in WHERE clause for atomic idempotency.
- **`updateCamera`** — defensively strips any `ingestMode` key from the payload even though `UpdateCameraSchema.strict()` already rejects it. Belt-and-suspenders per D-01 (mode immutable post-create).
- **Constructor** — 2 new OPTIONAL params (`SrsApiService`, `AuditService`) appended after existing `systemPrisma`. All 4 existing positional-arg test harnesses remain source-compatible.
- **CamerasModule** — imports `forwardRef(SrsModule)` + `AuditModule`.

### Controller + util + fixture (Task 2)

- **`serialize-camera.util.ts`** — new file. `serializeCamera<T>(camera, { perspective })` passes pull cameras through, returns push cameras unchanged for `owner`, masks `streamKey` and substitutes the mask into `streamUrl` for `masked`. Single chokepoint per RESEARCH Pitfall 6.
- **`cameras.controller.ts`** — `POST /api/cameras/:id/rotate-key` routes `req.user.id` through `findCameraById` (tenancy 404) then `rotateStreamKey`. `findAllCameras` + `findCameraById` now map returns through `serializeCamera(c, { perspective: 'owner' })` so the masking contract is active on every outbound surface today.
- **`push-camera.fixture.ts`** — real implementation. Replaces the Plan 00 stub that threw `Error('not implemented — Plan 01 wires this')`.

## Threat Register Status

| Threat ID | Disposition | Status |
|-----------|-------------|--------|
| T-19.1-SK-LEAK | mitigate | `camera.push.key_generated` and `camera.push.key_rotated` payloads carry only `streamKeyPrefix(key)` (first 4 chars). `push-audit.test.ts` ships the LEAK DETECTOR invariant — `expect(JSON.stringify(call[0])).not.toMatch(/[A-Za-z0-9_-]{21}/)` — asserting no 21-char nanoid substring appears in any payload. |
| T-19.1-KICK-UNAUTH | mitigate | `POST /cameras/:id/rotate-key` is behind `AuthGuard` + `findCameraById` (tenancy 404 on cross-org). Service `rotateStreamKey` throws `BadRequestException` on pull cameras. `rotate-stream-key.test.ts` verifies both the cross-path 400 and the positive-path auth integration via controller. |
| T-19.1-TENANCY-BYPASS | accept | `findByStreamKey`, `resolveForwardTarget` use `systemPrisma` — intentional. SRS callbacks have no CLS context; the 128-bit entropy of the key is the tenancy primitive. `camera.orgId` is then derived from the lookup result. |
| T-19.1-CSV-INJECT | mitigate | Plan 01's `BulkImportCameraSchema.superRefine` rejects push rows with non-empty `streamUrl`. `bulkImport` additionally generates the key server-side — a malicious client cannot influence the push URL even if it bypasses zod. `bulk-import-push-service.test.ts` covers both paths. |
| T-19.1-ROTATE-RACE | accept | Kick happens AFTER the DB commit — new key is authoritative even if SRS kick fails. Any continuing publish against the old key will be rejected by `on_publish` on the next publish cycle. `rotate-stream-key.test.ts` "tolerates kick failure — new key stays live" verifies this explicitly. No grace period per D-20. |
