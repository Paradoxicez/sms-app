## BKR06 Reproducer Recap (per 21-VALIDATION.md)

The defect this test layer guards against:

> Boot recovery enqueued 'start' job at 11:16:08 with `videoBitrate: 2000k`. 11 subsequent PATCHes between 11:18 and 11:22 changed `SD640.videoBitrate` to `2500k`/`3000k`/`2500k`/... Final DB state: `SD640.videoBitrate = 2500k`, `AuditLog count(camera.profile_hot_reload) = 11`. BullMQ active job state: still `name: "start"`, `data.profile.videoBitrate: "2000k"`, same `processedOn: 11:16:08`. FFmpeg PID 14013 unchanged.

The integration test compresses this to: **11 sequential `mainRedis.publish('camera:integration-cam-1:restart', ...)` calls** while a hung-startStream simulates the live FFmpeg holding the BullMQ lock. Without the fix → `invocationCount === 0` (the publishes hit nothing). With the fix → `invocationCount === 11` (each publish triggers gracefulRestart). The test asserts the latter.

This is the exact regression target Phase 21 needed but lacked.

## Skip-on-no-Redis Implementation

Plan 03's Task 2 originally proposed an async detection in `beforeAll`. That pattern does NOT work with vitest's `describe.skipIf` because the skip predicate is evaluated synchronously at file collection (before `beforeAll` runs). First test run with the async pattern: suite skipped even though `redis-cli ping` succeeded.

Fix applied as a deviation (Rule 3 — blocking issue): replace async detection with a synchronous bash `/dev/tcp` probe at module load. Probe code:

```typescript
function detectRedisSync(): boolean {
  try {
    execSync(
      `bash -c 'exec 3<>/dev/tcp/${REDIS_HOST}/${REDIS_PORT} && exec 3<&- && exec 3>&-' 2>/dev/null`,
      { timeout: 1_000, stdio: 'ignore' },
    );
    return true;
  } catch {
    return false;
  }
}
const isRedisAvailable = detectRedisSync();
```

This relies only on bash builtins (`/dev/tcp`) — no extra deps, runs in <50ms when Redis is up, returns false in <1s when down. The async pattern is preserved for the `beforeAll`'s connection setup (which only runs when `isRedisAvailable === true`).

## Decisions Made

See `key-decisions` in the frontmatter. Highlights inline:

- **Sync Redis probe via `execSync('bash -c exec 3<>/dev/tcp/...')`** — required because vitest evaluates `describe.skipIf()` synchronously at file collection. Discovered as a Rule 3 deviation when the initial async-in-beforeAll pattern caused the entire integration suite to skip even when Redis was reachable. The sync probe replaces 50+ lines of timing-fragile alternatives.
- **Subscriber-spy pattern via `redis.duplicate` override** — Plan 03 noted that `capturedSub.status === 'end'` assertions might be flaky against `ioredis-mock`. The mitigation: spy on `unsubscribe`/`quit` directly via wrapped functions that delegate to originals. This decouples the test from `ioredis-mock`'s internal status enum and asserts on what we actually care about (lifecycle invocation count).
- **50ms graceful + 100ms publish loop in BKR06 reproducer** — chosen so each iteration completes before the next publishes. With Mitigation 3 dedup, if the graceful was longer than the loop interval, multiple publishes during in-flight would dedup down to 1 invocation per "burst", not 11. The 50/100ms split is the tightest faithful timing that proves all 11 distinct restarts fire.
- **Reset `gracefulRestart` mock inside the BKR06 test body** — the beforeEach mock is `mockResolvedValue(undefined)` which doesn't increment a counter. Inside the BKR06 test we reassign to `vi.fn(async () => { invocationCount++; ... })`. This local-scope reassignment lets us count just the BKR06 path's invocations cleanly.

## Deviations from Plan

**Rule 3 (auto-fix blocking issue) — async Redis detection in `beforeAll` did not work as planned.**
- **Found during:** Task 2 first test run (the suite reported `1 skipped` even though `redis-cli ping` from the worktree succeeded).
