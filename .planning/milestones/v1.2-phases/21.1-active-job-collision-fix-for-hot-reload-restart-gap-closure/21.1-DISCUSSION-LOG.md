## Fix approach (D-12)

| Option | Description | Selected |
|--------|-------------|----------|
| A. Pub/sub signal | Redis pub/sub channel per camera. Worker subscribes on job start, gracefulRestart on signal. Preserves B-1, no polling, real-time | ✓ |
| B. job.update() + SIGTERM | Patch job.data + external kill, retry respawn. Requires StreamProcessor refactor for re-read | |
| C. Split jobIds | Simplest enqueue-side but breaks B-1 collision guard from Plan 04, risk of two FFmpeg per camera | |
| D. Cooperative flag | Redis key + worker polling. No B-1 impact but permanent polling overhead and worse latency | |

**User's choice:** A (Pub/sub signal) with all 4 mitigations bundled
**Notes:** User asked for detailed pros/cons of each option and rationale for recommendation A. Followed up by asking how much A's listed weaknesses can be reduced. Recommendation accepted after seeing residual risk table:
- Subscriber lifecycle: 90% mitigated by try/finally + sub.quit() + NestJS OnModuleDestroy
- Worker retry race: 95% mitigated by fingerprint diff check on subscribe ready event
- Test integration cost: 70% mitigated by hybrid layer (mock + real Redis container)
- Concurrent PATCH dedup: 100% mitigated by in-process `restartingCameras: Set<string>`

| Extract helper signalActiveOrEnqueue() | DRY but no second caller — premature abstraction | |

**User's choice:** Strict
**Notes:** Audited 4 remove-then-add patterns in codebase before asking. Confirmed only `enqueueProfileRestart` has incorrect behavior on active+locked state — the other 3 sites (`startStream`, `stopStream`, status notify debounce) have correct behavior for their respective intents. Decision avoids over-engineering and keeps blast radius small.

## Rollout (D-15)

**Initial framing rejected by user** — first asked Live deploy + safety net vs Force restart all FFmpeg vs Manual sweep. User asked for detailed pros/cons before answering, prompting reframing.

**Reframe:** Realized Docker Compose deploy mechanism (per CLAUDE.md "Docker Compose, single server, self-hosted") inherently force-restarts everything via Phase 15 graceful shutdown + boot recovery. Strategy A and B converge in practice for this project.

| Option | Description | Selected |
|--------|-------------|----------|
| A. Standard restart (no special handling) | Docker Compose restart + Phase 15 graceful shutdown + boot recovery handle force-restart automatically | ✓ |
| B. Maintenance window | Schedule deploy off-hours — over-engineering for project's deployment style | |
| C. Admin endpoint POST /admin/restart-all-streams | Adds new admin surface — scope creep | |

**User's choice:** A (Standard restart)
**Notes:** Plan 21.1 documents expected deploy behavior in SUMMARY but adds no ops procedure, no admin endpoint, no maintenance-window requirement. The Mitigation 2 fingerprint safety net (from D-12) covers the narrow window where a PATCH lands between OLD worker death and NEW worker subscriber attach.
