<objective>
Fix two bugs blocking camera-card snapshot generation:

1. **Auth bug.** SRS config has `hls_ctx on` + `hls_ts_ctx on`, so any HTTP GET of `/live/{org}/{cam}.m3u8` triggers `on_play` to authorize. SnapshotService currently passes an unauthenticated URL to FFmpeg → SRS returns 403 → FFmpeg sees "End of file" → snapshot fails silently. Fix: mint a PlaybackService session inside refreshOne, use the tokenized hlsUrl as the FFmpeg input.
2. **Stale-stream guard bug.** on_hls only fires snapshot when `seq_no === 0`. Cameras already streaming when the fix deploys have seq_no in the thousands (BKR05=21447, BKR06=803) and never get a snapshot until republished. Fix: broaden the guard to `seq_no === 0 || !(await snapshotService.hasSnapshot(cameraId))`.

Purpose: Camera cards display real thumbnails for ALL online cameras — including those already streaming when the fix deploys — and the snapshot pipeline survives SRS hls_ctx auth.

Output: snapshot.service.ts (auth + hasSnapshot helper), srs-callback.controller.ts (broadened guard), callbacks.test.ts (updated mock + new cases).
</objective>

```typescript
// Quick task 260426-06n: broaden the guard so cameras already publishing
// when this fix deploys (seq_no in the thousands) still get a catch-up
// snapshot. The OR short-circuits: seq_no===0 fires immediately without a
// DB hit; for seq_no>0 we ask SnapshotService whether a thumbnail already

// no further refreshes are triggered until the bulk-refresh-all endpoint

}
```

```typescript
if (this.snapshotService) {
```
