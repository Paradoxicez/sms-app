# Quick Task 260426-06n: Snapshot auth via PlaybackService token + on_hls catch-up guard

PlaybackService-backed authenticated SRS URL plus an OR-guarded on_hls
catch-up trigger so cameras already publishing when the fix deploys still
get a thumbnail without republishing.

## What Changed

### `apps/api/src/cameras/snapshot.service.ts`

- Added `ModuleRef` constructor injection for lazy `PlaybackService`
  resolution. Mirrors the pattern in `cameras.controller.ts:54-63` exactly
  — CamerasModule does NOT import PlaybackModule (cycle:
  Cameras → Playback → Cluster → Srs → Cameras), so `ModuleRef.get(...,
  { strict: false })` walks the global scope at first call to resolve it.
- `refreshOne()` now mints a real playback session via
  `getPlaybackService().createSession(camera.id, camera.orgId)` and
  passes `session.hlsUrl` verbatim to FFmpeg. The unauthenticated
  `http://${SRS_HOST}:8080/live/${orgId}/${cameraId}.m3u8` URL is gone.
- New public method `hasSnapshot(cameraId)` — single
  `findUnique({where:{id},select:{thumbnail:true}})` returning
  `Boolean(row?.thumbnail)`. Used by the on_hls callback to decide
  whether a mid-stream catch-up snapshot is needed.
- JSDoc updated to reflect the auth path: "Source: tokenized SRS HLS URL
  minted via PlaybackService.createSession — required because srs.conf
  has hls_ctx on…".
- All preserved: per-camera `inFlight` Set dedup, 5s bulk debounce,
  fire-and-forget swallow contract, `refreshAllForOrg` concurrency=3
  fan-out.

### `apps/api/src/srs/srs-callback.controller.ts`

- `onHls` snapshot guard widened from `seq_no === 0` to
  `seq_no === 0 || !(await this.snapshotService.hasSnapshot(cameraId))`.
- Whole block gated on `if (this.snapshotService)` so optional-controller
  legacy test constructors that pass `undefined` for snapshotService stay
  compatible (without the gate, `!(await x?.fn())` would yield
  `!undefined === true` and always trigger).
- Archive branch (recording, quota, path-traversal guard, archiveSegment
  call, archiveMetrics) untouched.

### `apps/api/tests/srs/callbacks.test.ts`

- `mockSnapshotService` extended with
  `hasSnapshot: vi.fn().mockResolvedValue(false)`.
- Existing test renamed and tightened: "does NOT trigger snapshot refresh
  on subsequent segments (seq_no > 0) when thumbnail already exists" —
  sets `hasSnapshot.mockResolvedValue(true)`, asserts no refresh AND
  asserts `hasSnapshot` was consulted exactly 3 times.
- 3 new test cases:
  1. `triggers snapshot when seq_no > 0 AND no thumbnail exists yet`
     (catch-up path).
  2. `skips snapshot when seq_no > 0 AND thumbnail already exists`
     (steady state).
  3. `skips hasSnapshot DB lookup when seq_no === 0` (cheap-path proof,
     OR short-circuits).
