<objective>
Replace the camera-form-dialog's hardcoded `<SelectItem value="">Default</SelectItem>` fallback with an explicit, required Stream Profile selection UX. This realigns the frontend with the Phase 21 backend semantics shipped in quick task 260426-07r (commit 813be0a) where `streamProfileId` resolution is `selected → org isDefault → 409` rather than a free-text "Default" placeholder.

Purpose: After 260426-07r, sending `streamProfileId: null` from the client is silently coerced server-side to the org's isDefault profile. The current UI implies "Default" is a valid choice and obscures which concrete profile gets applied — a footgun for tenants with multiple profiles and a soft-fail when the org has zero profiles. This task makes the choice explicit, enforces it client-side (friendlier than waiting for a 409), and gives orgs with zero profiles a guided path to create one.

Output:
- Hardcoded `Default` SelectItem removed
- Auto pre-selection of org's isDefault profile on create
- Inline required validation when org has profiles but user hasn't picked one
- Empty-state amber callout + disabled Save when org has 0 profiles
- Amber warning above Select for legacy edit-mode cameras with `streamProfileId === null`
- Five new vitest+RTL test cases extending the existing dialog test file
</objective>
