# Quick Task 260426-07r: Backend Default Profile Semantic Alignment Summary

Backend semantic fix so cameras with `streamProfileId=null` resolve to their org's `isDefault=true` profile at FFmpeg-job-enqueue time (instead of the hardcoded `{codec:'auto', audioCodec:'aac'}` last-resort), and so deleting an isDefault profile is blocked with HTTP 409 + a distinct error message when other profiles exist in the same org.

## What Was Built

### 1. `streams.service.ts` — null-profile org-default fallback (lines 87-126)

Replaced the single ternary that mapped `camera.streamProfile` truthy/falsy directly to either the embedded profile fields or the hardcoded `{codec:'auto', audioCodec:'aac'}` with a three-branch resolution:

1. `camera.streamProfile` truthy → use embedded profile fields (unchanged behavior, with `?? undefined` coercion of nullable schema fields to satisfy the `StreamProfile` interface).
2. `camera.streamProfile` falsy AND `this.systemPrisma` injected → query `systemPrisma.streamProfile.findFirst({ where: { orgId: camera.orgId, isDefault: true } })`. If a row returns, build the 7-field profile from it and emit the log line `camera ${id} streamProfileId is null — using org default profile ${defaultRow.id} (${defaultRow.name})`. If no row, emit warn-log and fall through to the hardcoded fallback.
3. Defensive: `systemPrisma` not injected (test-only path) → fall through to the hardcoded fallback with no log noise.

Added `import { StreamProfile } from './ffmpeg/ffmpeg-command.builder'` so the local `let profile: StreamProfile` annotation aligns with the downstream `StreamJobData['profile']` shape (which has optional fields per the codec builder's contract).

### 2. `stream-profile.service.ts` — isDefault delete precondition (lines 109-149)

Added pre-fetch + new precondition BEFORE the existing Phase 21 D-10 usedBy check:

1. `findUnique({ id }, select: { id, orgId, isDefault })` — pre-fetch target row.
2. If `target` is null → throw `NotFoundException('Stream profile not found')` (added to `@nestjs/common` import alongside existing `ConflictException`).
3. If `target.isDefault === true` → `count({ orgId, id: { not: target.id } })`. If `otherCount > 0` → throw `ConflictException('Set another profile as default before deleting this one. Run PATCH /api/stream-profiles/:id with isDefault=true on the profile you want to promote, then retry delete.')` (plain string message, distinguishable from D-10's object-form usedBy by both shape AND message prefix).
4. If `otherCount === 0` → fall through (delete the org's only profile is allowed; runtime hardcoded fallback covers cameras until a new profile is created).
5. Existing D-10 `camera.findMany` + `usedBy` check + `prisma.delete` preserved verbatim.

Greppable verification: `grep -n "Set another profile as default" apps/api/src/streams/stream-profile.service.ts` returns exactly 1 match (line 144).

**2. [Rule 1 - Type] Initial `let profile: {…} | {…}` annotation rejected nullable Prisma schema fields**

- **Found during:** Task 1 `tsc --noEmit` verification
- **Issue:** First implementation used a strict literal type for `profile` requiring all 7 fields as non-null strings/numbers. But `StreamProfile` schema marks `preset`, `resolution`, `fps`, `videoBitrate`, `audioBitrate` as nullable, so direct assignment from `defaultRow.preset` (etc.) emitted TS2322 (`'string | null' is not assignable to 'string'`).
- **Fix:** Imported the existing `StreamProfile` interface from `ffmpeg-command.builder.ts` (which models these as `?: string | undefined`) and coerced nullable Prisma values via `?? undefined`. Aligns the runtime shape with the codec builder's downstream consumer.
- **Files modified:** `apps/api/src/streams/streams.service.ts` (added import line 12, simplified `let profile: StreamProfile`)
