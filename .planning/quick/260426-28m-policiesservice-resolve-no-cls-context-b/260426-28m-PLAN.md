<objective>
Make `PoliciesService.resolve(cameraId)` resilient to background callers that have no active CLS context (e.g. fire-and-forget snapshot job → `PlaybackService.createSystemSession` → `policiesService.resolve`). When called outside any CLS context, the tenancy extension never emits `set_config('app.is_superuser','true')` or `set_config('app.current_org_id', ...)`, so RLS closes-by-default and the camera lookup at `policies.service.ts:155` returns null, throwing `NotFoundException("Camera ${cameraId} not found")` at line 161. This is the same RLS regression class fixed for 6 services in 260420-oid; PoliciesService was missed there because at audit time `resolve()` had no background callers.

Purpose: Close the residual snapshot-pipeline failure that survived 260426-0m4 (which only covered `PlaybackService.createSystemSession`'s own camera lookup). The next call inside `createSystemSession` — `await this.policiesService.resolve(cameraId)` — still injects `TENANCY_CLIENT` and runs without a CLS bypass, so it 404s and the snapshot job dies. Mirror the in-file `onModuleInit` precedent (lines 60-63) to wrap the resolve body in `cls.run({IS_SUPERUSER:'true'}, ...)` ONLY when there is no existing CLS context — HTTP callers (which already set ORG_ID via AuthGuard) keep their original tenant-scoped behavior, no double-bypass.

- `apps/api/src/policies/policies.service.ts` — `resolve()` becomes context-aware: detects empty CLS context, wraps the entire body (camera findUnique + policy findMany + sort/merge) in `cls.run({IS_SUPERUSER:'true'},...)`, emits a single debug log when the bypass is taken. HTTP-caller path unchanged.

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-260426-28m-01 | Information Disclosure | PoliciesService.resolve cross-org read after the fix | mitigate | The bypass is taken ONLY when CLS is empty (`!isActive() || (!ORG_ID && !IS_SUPERUSER)`). Authenticated HTTP users always have ORG_ID set by AuthGuard (line 51 of auth.guard.ts) — they will NOT take the bypass branch. The only callers reaching the bypass are background workers / boot lifecycle code (snapshot job, system session creation), which have no caller-supplied orgId to leak from. The resolve() output is policy data scoped to a SPECIFIC `cameraId` argument; resolved policies are not user-supplied tenant content (they are platform-level rules), so even if a wrong cameraId is passed by a misbehaving worker, the data returned is the resolved-policy aggregation, not a foreign org's confidential rows. |
| T-260426-28m-02 | Elevation of Privilege | Background worker tricks resolve() into running with IS_SUPERUSER='true' | mitigate | The bypass is gated on absence of any CLS signal — an authenticated request with ORG_ID set will NEVER take this branch (the `hasCtx` check is positive-signal). A malicious caller would need to call resolve() OUTSIDE any AuthGuard'd path, which already implies the call originated from server-side worker code (no external attacker reach). The IS_SUPERUSER flag is set inside a NEW `cls.run` scope and is therefore scoped to ONLY the resolve() body — it does not leak upward into any caller's CLS scope. |
