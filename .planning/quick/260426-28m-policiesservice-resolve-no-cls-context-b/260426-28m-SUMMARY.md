# Quick Task 260426-28m: PoliciesService.resolve no-CLS-context bypass — Summary

**One-liner:** Make PoliciesService.resolve resilient to background callers with no active CLS context by wrapping the body in `cls.run + IS_SUPERUSER='true'` when CLS is empty (snapshot pipeline → PlaybackService.createSystemSession → policiesService.resolve was 404'ing under RLS), HTTP callers unchanged.

## What changed

`apps/api/src/policies/policies.service.ts` — `resolve()`:

**Before** (excerpt):
```typescript
async resolve(cameraId: string): Promise<ResolvedPolicy> {
  const camera = await this.prisma.camera.findUnique({ ... });
  if (!camera) throw new NotFoundException(`Camera ${cameraId} not found`);
  // ... policy findMany + sort + scalar merge + domains merge ...
  return { ...resolved, sources };
}
```

**After** (excerpt):
```typescript
async resolve(cameraId: string): Promise<ResolvedPolicy> {
  const run = async (): Promise<ResolvedPolicy> => {
    // ... existing body verbatim (camera findUnique → policy findMany → merge → return) ...
  };

  // Defensive: legacy 1-arg test construction lacks cls.
  if (!this.cls) {
    return run();
  }

  // Mirrors prisma-tenancy.extension.ts gating byte-for-byte.
  const hasCtx =
    this.cls.isActive() &&
    (this.cls.get<string | undefined>('ORG_ID') ||
     this.cls.get<string | undefined>('IS_SUPERUSER'));

  if (hasCtx) {
    return run(); // HTTP path — caller already holds CLS context.
  }

  this.logger.debug(
    `No CLS context — running resolve with IS_SUPERUSER bypass for camera ${cameraId}`,
  );
  return this.cls.run(async () => {
    this.cls.set('IS_SUPERUSER', 'true');
    return run();
  });
}
```

The `cls.run(callback)` + `cls.set('IS_SUPERUSER','true')` shape matches the `onModuleInit` in-file precedent (lines 60-63) byte-for-byte — no novel pattern introduced.

`apps/api/tests/policies/policies-no-cls-bypass.test.ts` — 2 new regression tests:

- **Test A:** `service.resolve(camera.id)` invoked outside any cls.run scope → resolves successfully AND emits a debug log line containing `"No CLS context"` plus the cameraId.
- **Test B:** `service.resolve(camera.id)` invoked inside `cls.run(() => { cls.set('ORG_ID', org.id); ... })` → resolves successfully AND emits NO bypass log.

Both tests use a real `ClsService(new AsyncLocalStorage())` instance — the existing `makeTestClsService` Map stub does not implement `.run`/`.isActive`, so it cannot exercise this branch.

## Snapshot pipeline impact

The residual failure surviving 260426-0m4 (`PlaybackService.createSystemSession` → `policiesService.resolve` 404'ing under RLS) is now closed at the PoliciesService boundary:

1. Snapshot worker fires `PlaybackService.createSystemSession(cameraId)` outside any AuthGuard → no CLS context.
2. createSystemSession (already fixed in 260426-0m4) wraps its own camera lookup with `cls.run + IS_SUPERUSER='true'`.
3. Inside that scope it calls `policiesService.resolve(cameraId)` — but the inner call to resolve previously did NOT inherit a positive ORG_ID/IS_SUPERUSER signal cleanly, since createSystemSession's CLS scope ended before the resolve call (or the cls.run wrapper around the camera lookup was already exited).
4. With this fix: even if resolve is called outside the createSystemSession's bypass scope, it now self-detects the empty context and applies its own bypass.

UAT remains user-side (the user handles rebuild + restart of the live API) — see CLAUDE.md "Prisma schema change workflow" reminder; this fix has NO schema change so only an API rebuild + restart is needed.
