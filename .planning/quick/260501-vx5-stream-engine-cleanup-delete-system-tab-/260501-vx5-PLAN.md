<objective>
Two cleanup fixes shipped as a single quick task, three atomic commits.

**Issue A — System tab silent failure.** The /admin/stream-engine "System" tab writes to `SystemSettings` then calls `regenerateAndReloadSrsAtBoot()`/`regenerateAndReloadSrs()` which writes `/app/config/srs.conf` from inside the api container. After commit `d7f5b17` the canonical srs.conf moved to `deploy/srs.conf` (mounted RO into the srs container only — api container has no such file). The write either ENOENT-fails silently or scribbles into a non-functional path; the running SRS never picks up the change. Production DB confirms the row has `createdAt = updatedAt = 2026-04-29 06:17:11` and all default values — no user has ever successfully saved against it. Delete the entire dead path: UI tab, controller endpoints, service methods, DTO, the cluster `getConfig` endpoint that also reads `SystemSettings`, and the Prisma model + migration.

**Issue B — Live Logs tab broken.** `srs-log.gateway.ts` defaults to `spawn('docker', ['logs', '-f', ...])` from inside the api container, but the api image is `node:22-bookworm-slim` (no docker CLI, no socket mount). The gateway already has a file-mode branch when `SRS_LOG_PATH` env is set. Fix by configuring SRS for file logging, mounting a shared `srs_logs` named volume between srs (RW) and api (RO), and setting `SRS_LOG_PATH=/srs-logs/srs.log` on the api service.

Purpose: Restore "Save Settings" honesty (no UI for backend that doesn't work) and unbreak Live Logs in production. Net code reduction; zero new behaviour.
Output: Cleaned web page, slimmer settings module, dropped Prisma model + migration, file-logging SRS config, shared-volume compose wiring.
</objective>
