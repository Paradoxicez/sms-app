2. Add a handler immediately above the existing `handleListRowClick` callback:
```ts
const handleListDeleted = useCallback(
  (deletedId: string) => {
    if (deletedId !== id) return; // current recording was NOT the one deleted — list refresh already handled it
    // The currently-playing recording was just deleted. Navigate to the next available
    // recording on the same date if any; otherwise fall back to /app/recordings.
    const next = recordings.find((r) => r.id !== deletedId);
    if (next) {
      router.push(`/app/recordings/${next.id}`);
    } else {
      router.push('/app/recordings');
    }
  },
  [id, recordings, router],
);
```

NOTE: at the time this callback runs, `recordings` has ALREADY been refreshed (Task 1's `confirmDelete` awaits `refetch()` before calling `onDeleted`). So `recordings` here is the post-delete list — `find((r) => r.id !== deletedId)` is just defensive belt-and-braces in case the refetch hasn't propagated yet. Pick the FIRST element of the new list as "next", with that exclusion as a safety net.

3. Update the JSX at the bottom of the file:
```tsx
<RecordingsList
  recordings={recordings}
  loading={listLoading}
  currentRecordingId={id}
  selectedDate={selectedDate}
  onRowClick={handleListRowClick}
  onDeleted={handleListDeleted}
  refetch={refetchRecordings}
/>
```

4. Do NOT change anything else in `page.tsx` — no Timeline changes, no header changes, no error-state changes.

WHY: Lifting `refetch` to the consumer keeps `recordings-list.tsx` decoupled from the data layer (it just calls the function its parent supplied). The `onDeleted` callback is the seam that lets the page decide what to do when the user deletes the recording they're currently watching — without that, the page would render a 404 manifest until the next user action.
