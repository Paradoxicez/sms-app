# Quick 260419-h84: Fix tenancy RLS superuser bypass + StreamProcessor transition Summary

Positive-signal tenancy RLS (`app.is_superuser='true'`) replaces negative-signal bypass across 18 tables, AuthGuard enforces active-org or super-admin at the boundary, and StatusService now allows `reconnecting -> connecting` for retry re-enqueue.

## One-liner

Closed a multi-tenancy bypass where authenticated users without `activeOrganizationId` could read/write any org's rows, and unblocked the StreamProcessor retry loop that never reached FFmpeg when a camera was in `reconnecting`.

## What changed

### Gap 1 — RLS bypass (high severity)

**Before:** `superuser_bypass_*` policies matched when `app.current_org_id` was NULL/empty. Any session without an active org (fresh sign-up, unset state) bypassed `tenant_isolation_*` for all 18 tenant tables.

**After:** Bypass matches ONLY when `current_setting('app.is_superuser', true) = 'true'`. AuthGuard sets this flag in CLS when `session.user.role === 'admin'`, and the Prisma extension propagates it via `set_config('app.is_superuser', 'true', TRUE)`. Sessions without either signal see zero rows.

#### Migration filename + tables

`apps/api/src/prisma/migrations/rls_superuser_bypass_positive_signal/migration.sql` — drops + recreates superuser_bypass_* on:
Camera, Project, Site, StreamProfile, PlaybackSession, Policy, ApiKey, WebhookSubscription, OrgSettings, Recording, RecordingSegment, RecordingSchedule, Member, Invitation, UserPermissionOverride, AuditLog, Notification, NotificationPreference.

Verified via `SELECT polname, qual FROM pg_policies WHERE polname LIKE 'superuser_bypass_%'` — all 18 rows now show `(current_setting('app.is_superuser'::text, true) = 'true'::text)`.

#### AuthGuard diff (IS_SUPERUSER flag + 403 branch)

```typescript
// After session resolution
const orgId = session.session?.activeOrganizationId;
const isSuperAdmin = session.user?.role === 'admin';

if (!orgId && !isSuperAdmin) {
  throw new ForbiddenException(
    'No active organization. User must be a member of at least one organization.',
  );
}

if (orgId) this.cls.set('ORG_ID', orgId);
if (isSuperAdmin) this.cls.set('IS_SUPERUSER', 'true');
```

#### Prisma tenancy extension diff (dual set_config)

```typescript
const orgId = cls.get<string | undefined>('ORG_ID');
const isSuperuser = cls.get<string | undefined>('IS_SUPERUSER');

if (!orgId && !isSuperuser) return query(args);   // closed-by-default

const stmts: any[] = [];
if (orgId)        stmts.push(prisma.$executeRaw`SELECT set_config('app.current_org_id', ${orgId}, TRUE)`);
if (isSuperuser)  stmts.push(prisma.$executeRaw`SELECT set_config('app.is_superuser',   'true',    TRUE)`);
stmts.push(query(args));
const results = await prisma.$transaction(stmts);
return results[results.length - 1];
```

### Gap 2 — StreamProcessor transition (medium severity)

**Before:** `validTransitions.reconnecting = ['online', 'offline']`. When CameraHealthService or the user re-enqueued a stream job while the camera was in `reconnecting`, `StreamProcessor.process` called `transition(..., 'connecting')` → throw → BullMQ retry loop → no FFmpeg spawn.

**After:** `validTransitions.reconnecting = ['online', 'offline', 'connecting']`. One-line change; idempotency for same-target transitions was already handled at line 39-42 (`if (newStatus === currentStatus) return`).
