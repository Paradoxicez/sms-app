# Quick Task 260426-29p: Stream Profiles Page Default Indicator + Backend Auto-Default Summary

Adds an amber Star + tooltip next to the default Stream Profile in the admin table, and makes `StreamProfileService.create` silently auto-mark the first profile per org as `isDefault=true`, closing the "0 default profile in a populated org" invariant gap that the runtime fallback (260426-07r) used to absorb.

## Tasks Executed

### Task 1: Frontend Star indicator on Name cell + Vitest coverage

**Commit:** `9f2015e`

- Imported `Star` from `lucide-react` and Tooltip primitives (`TooltipProvider`, `Tooltip`, `TooltipTrigger`, `TooltipContent`) into `stream-profiles-columns.tsx`.
- Replaced the Name cell render with a flex row containing `<span>{name}</span>` plus a conditional `<TooltipProvider delay={200}>...<Star className="h-4 w-4 fill-amber-400 text-amber-400" aria-label="Default profile" />` block when `row.original.isDefault === true`.
- Pattern matches `cameras-columns.tsx` exactly (base-ui-react primitives — `delay` not `delayDuration`; `render` prop on `TooltipTrigger`, not `asChild`).
- Tooltip content: `Default profile — pre-selected when adding new cameras`.
- Created `__tests__/stream-profiles-columns.test.tsx` with 3 cases:
  - (a) Star renders with `aria-label="Default profile"` when `isDefault=true`.
  - (b) No element with that aria-label when `isDefault=false`.
  - (c) Profile name text always renders regardless of `isDefault`.

**Verification:** `pnpm --filter @sms-platform/web test -- stream-profiles-columns` → 3/3 passing.

### Task 2: Backend auto-mark first profile in org as isDefault + Vitest coverage

**Commit:** `3c69577`

- Modified `StreamProfileService.create(orgId, dto)`:
  - Added a `prisma.streamProfile.count({ where: { orgId } })` call before the existing mutual-exclusion block.
  - When `existingCount === 0` and `dto.isDefault` is falsy, emits `logger.log("auto-marked first profile \"<name>\" as isDefault=true for org <orgId>")` and uses an `effectiveDto = { ...dto, isDefault: true }` for the rest of the method.
  - Used a local `effectiveDto` variable instead of reassigning the `dto` parameter (cleaner TypeScript ergonomics).
- Created `tests/streams/stream-profile-create-auto-default.test.ts` with 4 cases:
  - (a) `dto.isDefault=false` on empty org → persisted with `isDefault=true`, logger emits auto-mark message.
  - (b) `dto.isDefault=false` on populated org (count=3) → persisted with `isDefault=false`, `updateMany` NOT called, no log.
  - (c) `dto.isDefault=true` on empty org → persisted with `isDefault=true`, `updateMany` fires (existing mutual-exclusion path matching 0 rows is fine), no log.
  - (d) `dto.isDefault=true` on populated org (count=2) → `updateMany` unsets prior defaults; persisted with `isDefault=true`, no log.
- Patched `tests/streams/profile-builder.test.ts` to add `count` to the shared `mockStreamProfile` (default `1` for the populated-org branch) and reset it in `beforeEach`.

**Verification:**

### Auto-fixed Issues

**1. [Rule 3 - Blocking] profile-builder.test.ts mocks missing `streamProfile.count`**
- **Found during:** Task 2 full-suite regression run.
- **Issue:** The legacy `profile-builder.test.ts` shared `mockStreamProfile` did not include a `count` mock. After Task 2's service change introduced `await this.prisma.streamProfile.count(...)` in `create()`, two tests crashed with `TypeError: this.prisma.streamProfile.count is not a function`.
- **Fix:** Added `count: vi.fn().mockResolvedValue(1)` to the shared mock and a matching `mockStreamProfile.count.mockResolvedValue(1)` reset in `beforeEach` (after `vi.clearAllMocks()`). Default `1` keeps these tests on the existing-profile (populated-org) branch, preserving their original mutual-exclusion assertion semantics.
- **Files modified:** `apps/api/tests/streams/profile-builder.test.ts`
- **Commit:** `3c69577` (folded into Task 2 commit per plan guidance).
